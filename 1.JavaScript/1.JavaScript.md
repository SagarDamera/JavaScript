# JavaScript Fundamentals

## 1. What are the primitive data types in JavaScript?

Primitive data types in JavaScript are values that are not objects and are stored directly as values.

JavaScript has **7 primitive data types**:

- `string`
- `number`
- `bigint`
- `boolean`
- `undefined`
- `null`
- `symbol`

#### Explanation

Primitive values are immutable. That means the actual value cannot be changed directly. When we perform operations on primitives, JavaScript creates a new value instead of modifying the existing one.

For example, strings look like they are being changed, but they are not mutated in place.

```js
let name = "React";
name.toUpperCase();

console.log(name); // "React"
```

`name.toUpperCase()` creates a new string, but the original string remains unchanged.

#### Real-World Usage

Understanding primitive types is important when working with state comparison, props comparison, dependency arrays, and memoization in React.

## 2. What is the difference between primitive values and reference values?

Primitive values are compared and copied by their actual value, while reference values are compared and copied by their memory reference.

#### Explanation

Primitive values include types like `string`, `number`, and `boolean`. When assigned to another variable, the actual value is copied.

Reference values include objects, arrays, and functions. When assigned to another variable, the reference to the same object is copied, not the object itself.

```js
let a = 10;
let b = a;

b = 20;

console.log(a); // 10
console.log(b); // 20
```

Here, `a` and `b` are independent because numbers are primitive values.

```js
const user1 = { name: "Alex" };
const user2 = user1;

user2.name = "John";

console.log(user1.name); // "John"
```

Here, both variables point to the same object reference.

#### Real-World Usage

In React, this is very important for state updates. Mutating an object directly may not trigger a re-render because React relies on reference changes to detect updates.

## 3. What is the difference between var, let, and const?

`var` is function-scoped, while `let` and `const` are block-scoped. `let` allows reassignment, `const` does not allow reassignment, and `var` should generally be avoided in modern JavaScript.

#### Explanation

`var` is scoped to the nearest function and is hoisted with an initial value of `undefined`.

`let` and `const` are scoped to the nearest block and are also hoisted, but they are not accessible before declaration because of the temporal dead zone.

```js
function example() {
  if (true) {
    var a = 1;
    let b = 2;
    const c = 3;
  }

  console.log(a); // 1
  // console.log(b); // ReferenceError
  // console.log(c); // ReferenceError
}
```

`const` means the variable binding cannot be reassigned, but it does not make objects immutable.

```js
let count = 1;
count = 2; // allowed

const name = "React";
// name = "Vue"; // not allowed
```

#### Real-World Usage

In modern React code, `const` is preferred by default, and `let` is used only when reassignment is needed. `var` is rarely used because function scoping can create bugs in large codebases.

## 4. What is temporal dead zone in JavaScript?

The temporal dead zone is the time between entering a block scope and the actual declaration of a `let` or `const` variable, during which accessing the variable causes a `ReferenceError`.

#### Explanation

`let` and `const` declarations are hoisted, but they are not initialized immediately. They stay in the temporal dead zone until JavaScript reaches the declaration line.

```js
console.log(name); // ReferenceError

let name = "React";
```

This is different from `var`, which is hoisted and initialized with `undefined`.

```js
console.log(name); // undefined

var name = "React";
```

The temporal dead zone helps catch bugs where variables are used before they are properly declared.

#### Real-World Usage

In real projects, temporal dead zone errors usually happen during refactoring, when code is moved above a `let` or `const` declaration accidentally.

## 5. Why can const objects still be mutated?

`const` prevents reassignment of the variable binding, but it does not make the object itself immutable.

#### Explanation

When we declare an object with `const`, the variable must always point to the same object reference. However, the properties inside that object can still be changed.

```js
const user = {
  name: "Alex",
};

user.name = "John"; // allowed

console.log(user); // { name: "John" }

// user = { name: "Sam" }; // not allowed
```

The reassignment is blocked, but mutation of the existing object is allowed.

To prevent mutation, we need approaches like `Object.freeze()` or immutable update patterns.

```js
const user = Object.freeze({
  name: "Alex",
});

// user.name = "John"; // ignored or throws in strict mode
```

#### Real-World Usage

In React, this is a common source of bugs. Even if state is declared using `const`, directly mutating an object or array can prevent React from detecting changes correctly. Instead, create a new object or array reference when updating state.

## 6. What is hoisting?

Hoisting is JavaScript's behavior of processing variable and function declarations before code execution, making some declarations available earlier in their scope.

#### Explanation

JavaScript does not physically move code to the top. Instead, during the creation phase of execution, it registers declarations in memory.

For `var`, the declaration is hoisted and initialized with `undefined`.

```js
console.log(count); // undefined

var count = 10;
```

For `let` and `const`, declarations are also hoisted, but they are not initialized until the actual declaration line. Accessing them before that causes a `ReferenceError` because of the temporal dead zone.

```js
console.log(count); // ReferenceError

let count = 10;
```

Function declarations are hoisted with their full function body, so they can be called before they appear in the code.

#### Real-World Usage

Hoisting matters when reading legacy JavaScript, debugging initialization issues, or refactoring code. In modern React codebases, using `const`, `let`, and clear declaration order reduces hoisting-related confusion.

#### Common Follow-Up Question

## 7. How are function declarations hoisted differently from function expressions?

Function declarations are hoisted with their full implementation, but function expressions follow the hoisting behavior of the variable they are assigned to.

#### Explanation

A function declaration can be called before it is defined because JavaScript stores the full function during the creation phase.

```js
sayHello(); // "Hello"

function sayHello() {
  console.log("Hello");
}
```

A function expression is different because the variable is hoisted, not the function value.

With `var`, the variable is initialized as `undefined`, so calling it before assignment causes a `TypeError`.

```js
sayHello(); // TypeError: sayHello is not a function

var sayHello = function () {
  console.log("Hello");
};
```

With `let` or `const`, calling it before declaration causes a `ReferenceError` due to the temporal dead zone.

```js
sayHello(); // ReferenceError

const sayHello = function () {
  console.log("Hello");
};
```

Arrow functions behave the same way as function expressions because they are assigned to variables.

#### Real-World Usage

In React, components are commonly written as function declarations or `const` arrow functions. Function declarations can be referenced earlier in the file, while `const` components must be declared before use.

## 8. What is the difference between undefined and null?

`undefined` means a variable has been declared but has not been assigned a value, while `null` is an intentional value that represents no value or empty value.

#### Explanation

`undefined` is usually assigned by JavaScript automatically.


`null` is usually assigned intentionally by the developer.


A common interview distinction is:

```js
typeof undefined; // "undefined"

```

`typeof null` returning `"object"` is a long-standing JavaScript behavior and is considered a historical bug.

#### Real-World Usage

In React, `undefined` often means data has not been provided yet, while `null` is commonly used to intentionally represent no selected item, no result, or no component rendering.

```jsx

```

## 9. What is the difference between undeclared, undefined, and null variables?

An undeclared variable has not been declared in the current accessible scope, an undefined variable has been declared but not assigned a value, and a null variable has been intentionally assigned no value.

#### Explanation

An undeclared variable does not exist in scope. Accessing it usually causes a `ReferenceError`.

```js
console.log(user); // ReferenceError
```

An undefined variable exists, but no value has been assigned.

```js
let user;

console.log(user); // undefined
```

A null variable exists and intentionally points to no value.

```js
let user = null;

console.log(user); // null
```

The key difference is intent:

- `undeclared`: variable does not exist
- `undefined`: variable exists but has no assigned value
- `null`: variable exists and was intentionally set to empty

#### Real-World Usage

This distinction helps in API handling and React rendering. For example, `undefined` may mean data is still loading, while `null` may mean the API returned no matching record.

## 10. What is type coercion in JavaScript?

Type coercion is JavaScript's automatic or explicit conversion of a value from one type to another.

#### Explanation

JavaScript performs coercion in many operations, especially with comparison operators, arithmetic operators, and string concatenation.

Implicit coercion happens automatically.

```js
console.log("5" + 1); // "51"
console.log("5" - 1); // 4
console.log(true + 1); // 2
```

In `"5" + 1`, JavaScript converts `1` to a string because `+` can mean string concatenation.

In `"5" - 1`, JavaScript converts `"5"` to a number because subtraction only works with numbers.

Explicit coercion is when we manually convert values.

```js
Number("5"); // 5
String(123); // "123"
Boolean(0); // false
```

In interviews, it is important to mention that coercion is not always bad, but uncontrolled coercion can create bugs. Using `===` instead of `==` avoids many unexpected conversions.

#### Real-World Usage

In React applications, type coercion issues often happen with form values because input values usually come as strings. For example, numeric inputs may need explicit conversion before validation, calculation, or API submission.

Implicit coercion happens automatically by JavaScript, while explicit coercion is when we intentionally convert a value from one type to another.

## 12. What is the difference between == and ===?

`==` compares values after performing type coercion, while `===` compares both value and type without coercion.

#### Explanation

`==` allows JavaScript to convert types before comparison.

```js
console.log(5 == "5"); // true
console.log(false == 0); // true
```

`===` does a strict comparison. Both the value and the type must match.

```js
console.log(5 === "5"); // false
console.log(false === 0); // false
```

In modern JavaScript, `===` is preferred because it avoids unexpected behavior caused by implicit coercion.

```js
const idFromApi = "10";
const selectedId = 10;

console.log(idFromApi == selectedId); // true
console.log(idFromApi === selectedId); // false
```

The second comparison is safer because it shows that the values are not actually the same type.

#### Real-World Usage

In React applications, strict equality is commonly used in conditional rendering, filtering lists, comparing IDs, and checking state values. It prevents bugs when API values and UI values have different types.

## 13. Why does [] == false return true?

`[] == false` returns `true` because JavaScript performs implicit type coercion when using `==`.

#### Explanation

The comparison happens in multiple conversion steps.

```js
console.log([] == false); // true
```

First, `false` is converted to a number:

```js
false -> 0
```

Then the empty array is converted to a primitive value. An empty array becomes an empty string:

```js
[] -> ""
```

Then the empty string is converted to a number:

```js
"" -> 0
```

So the final comparison becomes:

```js
0 == 0; // true
```

That is why `[] == false` returns `true`.

With strict equality, no coercion happens:

```js
console.log([] === false); // false
```

An array is an object, and `false` is a boolean, so they are not strictly equal.

#### Real-World Usage

This is a classic example of why strict equality is preferred in production code. In React rendering logic, relying on loose equality can create unexpected UI behavior.

## 14. Why does typeof null return "object"?

`typeof null` returns `"object"` because of a historical JavaScript implementation bug that was kept for backward compatibility.

In JavaScript, `null` is a primitive value that represents intentional absence of value.

Even though the result is `"object"`, `null` is not actually an object.

A safe way to check for `null` is direct comparison:

This behavior has existed since early JavaScript versions. Changing it now would break a large amount of existing code, so the language keeps it as-is.

## 15. What is the difference between Number.isNaN() and global isNaN()?

`Number.isNaN()` checks whether a value is actually the number value `NaN`, while global `isNaN()` first coerces the value to a number and then checks if the result is `NaN`.

#### Explanation

`Number.isNaN()` is stricter and more predictable.

```js
Number.isNaN(NaN); // true
Number.isNaN("hello"); // false
Number.isNaN(undefined); // false
```

Global `isNaN()` performs type coercion before checking.

```js
isNaN(NaN); // true
isNaN("hello"); // true
isNaN(undefined); // true
isNaN("123"); // false
```

For example:

```js
isNaN("hello");
```

JavaScript first tries:

```js
Number("hello"); // NaN
```

So the result becomes `true`.

But:

```js
Number.isNaN("hello"); // false
```

Because `"hello"` itself is not the numeric value `NaN`.

In modern JavaScript, `Number.isNaN()` is usually preferred because it avoids unexpected coercion.

#### Real-World Usage

In React forms, `Number.isNaN()` is useful after explicitly converting user input to a number. This keeps validation predictable and avoids bugs caused by automatic coercion.

# Scope, Closure, and Execution Context

## 16. What is lexical scope?

Lexical scope means a variable’s accessibility is decided by where it is written in the source code, not by where the function is called from.

#### Explanation

In JavaScript, functions remember the scope in which they were created. An inner function can access variables from its outer function because of lexical scope.

```js
function outer() {
  const message = "Hello";

  function inner() {
    console.log(message);
  }

  inner();
}

outer(); // "Hello"
```

Here, `inner` can access `message` because `inner` is physically written inside `outer`.

Lexical scope is the foundation for closures in JavaScript.

#### Real-World Usage

In React, lexical scope is used heavily in event handlers, hooks, callbacks, and closures. For example, a click handler can access props, state, and local variables from the component where it was defined.

#### Common Follow-Up Question

## 17. What is function scope?

Function scope means variables declared inside a function are accessible only within that function.

#### Explanation

Variables declared with `var` are function-scoped. They are not limited to blocks like `if`, `for`, or `while`.

```js
function example() {
  var name = "React";

  if (true) {
    var framework = "Next.js";
  }

  console.log(name); // "React"
  console.log(framework); // "Next.js"
}

example();
```

Here, `framework` is accessible outside the `if` block because `var` is scoped to the entire function.

However, variables declared inside a function cannot be accessed from outside that function.

```js
function example() {
  var name = "React";
}

console.log(name); // ReferenceError
```

#### Real-World Usage

Function scope is important when working with older JavaScript code that uses `var`. In modern React code, we mostly use `let` and `const`, but understanding function scope helps when debugging legacy code.

#### Common Follow-Up Question

## 18. What is block scope?

Block scope means variables are accessible only inside the block where they are declared.

#### Explanation

A block is usually anything inside `{}` such as an `if`, `for`, `while`, or function block.

Variables declared with `let` and `const` are block-scoped.

```js
if (true) {
  let count = 10;
  const name = "React";

  console.log(count); // 10
  console.log(name); // "React"
}

console.log(count); // ReferenceError
console.log(name); // ReferenceError
```

Unlike `var`, `let` and `const` do not leak outside the block.

This makes code safer and easier to reason about.

#### Real-World Usage

In React and modern JavaScript, block scope helps avoid accidental variable leaks, especially inside conditions, loops, and component logic.

## 19. What is the scope chain?

The scope chain is the mechanism JavaScript uses to resolve variables by looking in the current scope first, then moving outward through parent scopes until it reaches the global scope.

#### Explanation

When JavaScript sees a variable, it first checks the local scope. If it cannot find the variable there, it checks the outer scope, then the next outer scope, and so on.

```js
const globalValue = "global";

function outer() {
  const outerValue = "outer";

  function inner() {
    const innerValue = "inner";

    console.log(innerValue); // local scope
    console.log(outerValue); // outer function scope
    console.log(globalValue); // global scope
  }

  inner();
}

outer();
```

For `inner`, the scope chain is:

```txt
inner scope -> outer scope -> global scope
```

If the variable is not found anywhere in the chain, JavaScript throws a `ReferenceError`.

#### Real-World Usage

In React, scope chain behavior is common in component functions, nested handlers, custom hooks, and utility functions. It explains how callbacks access props, state, and variables from the component body.

#### Common Follow-Up Question

## 20. What is an execution context?

An execution context is the environment JavaScript creates to run code. It contains information like variables, functions, scope chain, and the value of `this`.

#### Explanation

Whenever JavaScript runs code, it does so inside an execution context.

There are mainly three types:

- Global execution context
- Function execution context
- Eval execution context, rarely used in real applications

When a function is called, JavaScript creates a new function execution context for that function.

```js
const appName = "React App";

function greet(user) {
  const message = `Hello ${user}`;
  return message;
}

greet("Alex");
```

At a high level:

- The global execution context stores `appName` and `greet`
- Calling `greet("Alex")` creates a new function execution context
- That function context stores `user`, `message`, scope information, and `this`

Once the function finishes, its execution context is removed from the call stack.

#### Real-World Usage

Execution context helps explain how JavaScript handles function calls, closures, `this`, recursion, and the call stack. In React, it is useful for understanding why each render creates a new function execution context for a component.

## 21. What happens during the creation phase of an execution context?

During the creation phase, JavaScript prepares the environment before executing the code. It sets up memory for variables and functions, creates the scope chain, and determines the value of `this`.

#### Explanation

For a function execution context, JavaScript mainly does these things:

- Creates the variable environment
- Creates the lexical environment
- Sets up the scope chain
- Determines `this`
- Hoists function declarations
- Hoists variables differently based on `var`, `let`, and `const`

Function declarations are stored with their full function body.

```js
sayHello(); // works

function sayHello() {
  console.log("Hello");
}
```

`var` declarations are initialized with `undefined`.

```js
console.log(count); // undefined

var count = 10;
```

`let` and `const` are hoisted but not initialized, so accessing them before declaration causes a `ReferenceError`.

```js
console.log(count); // ReferenceError

let count = 10;
```

#### Real-World Usage

This helps explain hoisting, temporal dead zone, and why some functions can be called before declaration. In React projects, it is useful when debugging initialization errors and refactoring JavaScript modules.

## 22. What happens during the execution phase of an execution context?

During the execution phase, JavaScript runs the code line by line, assigns actual values to variables, executes function calls, and evaluates expressions.

#### Explanation

After the creation phase prepares memory, the execution phase performs the actual work.

```js
var count = 10;

function double(value) {
  return value * 2;
}

const result = double(count);

console.log(result); // 20
```

At execution time:

- `count` gets assigned `10`
- `double(count)` is called
- A new function execution context is created for `double`
- The return value is assigned to `result`
- `console.log(result)` prints `20`

The important point is that declarations are prepared in the creation phase, but assignments and function calls happen during the execution phase.

#### Real-World Usage

This is useful when debugging order-of-execution issues, especially with module initialization, function calls, and variables used before assignment.

## 23. What is the call stack?

The call stack is the data structure JavaScript uses to keep track of function calls. It follows a Last-In, First-Out order.

#### Explanation

When a function is called, JavaScript pushes a new execution context onto the call stack. When the function finishes, that execution context is popped off the stack.

```js
function first() {
  second();
}

function second() {
  third();
}

function third() {
  console.log("Done");
}

first();
```

The stack flow looks like this:

```txt
global()
global() -> first()
global() -> first() -> second()
global() -> first() -> second() -> third()
```

After `third` finishes, functions are removed from the stack in reverse order.

If too many function calls happen without returning, such as infinite recursion, JavaScript throws a stack overflow error.

```js
function repeat() {
  repeat();
}

repeat(); // RangeError: Maximum call stack size exceeded
```

#### Real-World Usage

The call stack helps debug stack traces, recursive functions, synchronous errors, and the order in which JavaScript functions execute. In React, stack traces often show the component or function path that led to an error.

## 24. What is a closure?

A closure is created when a function remembers variables from its lexical scope even after the outer function has finished executing.

#### Explanation

In simple terms, a closure allows an inner function to access variables from an outer function.

```js
function createCounter() {
  let count = 0;

  return function increment() {
    count += 1;
    return count;
  };
}

const counter = createCounter();

console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3
```

Here, `createCounter` finishes execution, but the returned `increment` function still remembers the `count` variable.

That remembered scope is the closure.

#### Real-World Usage

Closures are used in event handlers, callbacks, private state, custom hooks, memoized functions, and async logic. In React, every render creates closures around the props and state from that render.

## 25. Why are closures useful?

Closures are useful because they allow functions to preserve state, create private variables, and access values from their original scope.

#### Explanation

Closures help us keep data available without exposing it globally.

```js
function createUserSession() {
  let token = null;

  return {
    login(newToken) {
      token = newToken;
    },
    getToken() {
      return token;
    },
    logout() {
      token = null;
    },
  };
}

const session = createUserSession();

session.login("abc123");

console.log(session.getToken()); // "abc123"
```

Here, `token` is not directly accessible from outside. It can only be accessed through the returned methods.

This is useful for encapsulation and controlled access.

#### Real-World Usage

Closures are heavily used in React for event handlers, hooks, debouncing, throttling, custom hooks, and async callbacks. For example, a click handler can access the state and props from the render in which it was created.

## 27. What memory issues can closures create?

Closures can create memory issues when they keep references to large objects or unused data longer than necessary, preventing garbage collection.

#### Explanation

A closure keeps its outer variables alive as long as the inner function is still reachable.

```js
function createHandler() {
  const largeData = new Array(1000000).fill("data");

  return function handleClick() {
    console.log(largeData.length);
  };
}

const handler = createHandler();
```

Here, `largeData` cannot be garbage collected because `handler` still references it through closure.

This becomes a problem when closures are stored in long-lived places like:

- Global variables
- Timers
- Event listeners
- Subscriptions
- Cached callbacks
- Uncleaned React effects

A common fix is to remove unused listeners, clear timers, unsubscribe from subscriptions, and avoid capturing unnecessary large objects.

#### Real-World Usage

In React, closure-related memory issues commonly happen when effects create event listeners, intervals, or subscriptions without cleanup. Always return a cleanup function from `useEffect` when registering long-lived side effects.

## 28. How would you fix a closure bug inside a loop?

I would fix a closure bug inside a loop by using block-scoped variables with `let`, so each iteration gets its own separate binding.

#### Explanation

A classic bug happens when using `var` inside a loop because `var` is function-scoped. All callbacks share the same variable reference.

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}

// 3
// 3
// 3
```

By the time the callbacks run, the loop has finished and `i` is `3`.

Using `let` fixes it because `let` creates a new binding for each loop iteration.

```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}

// 0
// 1
// 2
```

Before `let`, this was commonly fixed using an IIFE.

```js
for (var i = 0; i < 3; i++) {
  ((index) => {
    setTimeout(() => {
      console.log(index);
    }, 1000);
  })(i);
}
```

#### Real-World Usage

This matters in event handlers, timers, async callbacks, and loops that create functions. In modern JavaScript and React code, `let` usually solves this cleanly.

#### Common Follow-Up Question

## 29. What is the difference between lexical scope and dynamic scope?

Lexical scope resolves variables based on where functions are written in the code, while dynamic scope resolves variables based on where functions are called from.

#### Explanation

JavaScript uses lexical scope, not dynamic scope.

In lexical scope, the function carries the scope from where it was defined.

```js
const name = "global";

function printName() {
  console.log(name);
}

function run() {
  const name = "local";
  printName();
}

run(); // "global"
```

`printName` logs `"global"` because it was defined in the global scope. It does not use the caller's local `name`.

If JavaScript had dynamic scope, this example would log `"local"` because `printName` was called from inside `run`.

But JavaScript does not work that way.

#### Real-World Usage

This is important when reasoning about callbacks, closures, and React event handlers. A callback uses the variables from where it was created, not from where it is eventually executed.

#### Common Follow-Up Question

How does lexical scope help create closures?

## 30. Can closures access updated values or only the value at creation time?

Closures capture variables by reference to their lexical environment, not just a frozen copy of the value. However, in React, each render creates a new lexical environment, so closures may hold values from a specific render.

#### Explanation

In plain JavaScript, if the closed-over variable is updated, the closure can see the updated value.

```js
function example() {
  let count = 0;

  const increment = () => {
    count += 1;
  };

  const log = () => {
    console.log(count);
  };

  increment();
  increment();
  log(); // 2
}

example();
```

Here, `log` sees the updated `count` because both functions close over the same variable binding.

But React has an important difference. Each render creates a new function execution context and new bindings for state values.

```jsx
function Counter() {
  const [count, setCount] = React.useState(0);

  React.useEffect(() => {
    const id = setInterval(() => {
      console.log(count);
    }, 1000);

    return () => clearInterval(id);
  }, []);

  return <button onClick={() => setCount(count + 1)}>Increment</button>;
}
```

In this example, the interval logs the `count` from the initial render because the effect runs only once. This is a stale closure.

A common fix is to include dependencies or use a functional state update depending on the requirement.

```jsx
function Counter() {
  const [count, setCount] = React.useState(0);

  return (
    <button onClick={() => setCount((prev) => prev + 1)}>Increment</button>
  );
}
```

#### Real-World Usage

This is one of the most important closure concepts in React. It affects `useEffect`, async callbacks, timers, subscriptions, event handlers, and memoized callbacks.

## 31. What are first-class functions?

First-class functions mean functions in JavaScript are treated like values. They can be assigned to variables, passed as arguments, returned from other functions, and stored in objects or arrays.

```js
const greet = function () {

function execute(fn) {
```

## 32. What are higher-order functions?

A higher-order function is a function that either accepts another function as an argument, returns a function, or both.

#### Explanation

Because JavaScript has first-class functions, we can build higher-order functions.

```js
function withLogging(fn) {
```

Here, `withLogging` is a higher-order function because it accepts a function and returns a new function.

## 33. What is a callback function?

```js
function fetchData(onSuccess) {
```

```js
button.addEventListener("click", callback);
```

## 34. What is the difference between function declaration and function expression?

A function declaration is defined using the `function` keyword as a standalone statement, while a function expression creates a function as part of an expression and usually assigns it to a variable.

#### Explanation

Function declarations are hoisted with their full implementation, so they can be called before they appear in the code.

```js
sayHello(); // "Hello"

function sayHello() {
  console.log("Hello");
}
```

Function expressions are not hoisted the same way. The variable is hoisted based on whether it is declared with `var`, `let`, or `const`, but the function value is assigned only during execution.

```js
sayHello(); // ReferenceError

const sayHello = function () {
  console.log("Hello");
};
```

Function expressions are often useful when passing functions around as values or creating inline behavior.

```js
const add = function (a, b) {
  return a + b;
};
```

#### Real-World Usage

In React, both styles are valid. Function declarations are useful when readability and hoisting are helpful. Function expressions, especially arrow functions assigned to `const`, are common for components, handlers, and hooks-based code.

## 35. What is the difference between regular functions and arrow functions?

Regular functions have their own `this`, `arguments`, and can be used as constructors. Arrow functions do not have their own `this` or `arguments`, and they cannot be used as constructors.

#### Explanation

The biggest practical difference is how `this` works.

Regular functions get `this` based on how they are called.

```js
const user = {
  name: "Alex",
  regularFunction: function () {
    console.log(this.name);
  },
};

user.regularFunction(); // "Alex"
```

Arrow functions capture `this` from the surrounding lexical scope.

```js
const user = {
  name: "Alex",
  arrowFunction: () => {
    console.log(this.name);
  },
};

user.arrowFunction(); // usually undefined
```

Arrow functions are great for callbacks because they preserve the surrounding `this`.

```js
function Timer() {
  this.seconds = 0;

  setInterval(() => {
    this.seconds += 1;
  }, 1000);
}
```

Regular functions are better when a function needs its own `this`, such as object methods, constructors, or prototype methods.

#### Real-World Usage

In modern React functional components, arrow functions are commonly used for event handlers, callbacks, and utility functions. In class components, arrow functions were often used to avoid manually binding `this`.

## 36. How does this behave inside an arrow function?

Inside an arrow function, `this` is lexically inherited from the surrounding scope. Arrow functions do not create their own `this`.

#### Explanation

The value of `this` inside an arrow function is decided by where the arrow function is defined, not by how it is called.

```js
const user = {
  name: "Alex",

  regularMethod() {
    const arrowFn = () => {
      console.log(this.name);
    };

    arrowFn();
  },
};

user.regularMethod(); // "Alex"
```

Here, `arrowFn` does not have its own `this`. It uses `this` from `regularMethod`, where `this` points to `user`.

But using an arrow function directly as an object method is usually incorrect if you need object `this`.

```js
const user = {
  name: "Alex",

  arrowMethod: () => {
    console.log(this.name);
  },
};

user.arrowMethod(); // usually undefined
```

The arrow function does not bind `this` to `user`.

#### Real-World Usage

In React class components, arrow functions were commonly used to avoid manual binding. In modern functional components, `this` is rarely used, but arrow functions are still common for callbacks, event handlers, and hooks.

## 37. How does this behave inside a regular function?

Inside a regular function, `this` is determined by how the function is called.

#### Explanation

A regular function does not capture `this` lexically. Its `this` value depends on the call site.

When called as an object method, `this` points to the object before the dot.

```js
const user = {
  name: "Alex",

  greet() {
    console.log(this.name);
  },
};

user.greet(); // "Alex"
```

When called as a standalone function, `this` is usually `undefined` in strict mode.

```js
"use strict";

function greet() {
  console.log(this);
}

greet(); // undefined
```

When used with `call`, `apply`, or `bind`, we can explicitly control `this`.

```js
function greet() {
  console.log(this.name);
}

greet.call({ name: "React" }); // "React"
```

#### Real-World Usage

Understanding regular function `this` is useful when working with object methods, legacy class components, event handlers, third-party libraries, and older JavaScript codebases.

## 38. What are call, apply, and bind?

`call`, `apply`, and `bind` are function methods used to explicitly control the value of `this`.

#### Explanation

`call` invokes the function immediately and passes arguments one by one.

```js
function greet(message) {
  console.log(`${message}, ${this.name}`);
}

greet.call({ name: "Alex" }, "Hello"); // "Hello, Alex"
```

`apply` also invokes the function immediately, but arguments are passed as an array.

```js
greet.apply({ name: "Alex" }, ["Hi"]); // "Hi, Alex"
```

`bind` does not invoke the function immediately. It returns a new function with `this` permanently bound.

```js
const greetAlex = greet.bind({ name: "Alex" });

greetAlex("Welcome"); // "Welcome, Alex"
```

Simple difference:

```txt
call  -> calls immediately, arguments individually
apply -> calls immediately, arguments as array
bind  -> returns a new bound function
```

#### Real-World Usage

In older React class components, `bind` was commonly used to bind event handlers. In modern React functional components, these are less common, but they are still important for JavaScript interviews and library-level code.

## 39. What is partial application using bind?

Partial application using `bind` means creating a new function by pre-filling some arguments of an existing function.

#### Explanation

`bind` can bind both `this` and initial arguments.

```js
function multiply(a, b) {
  return a * b;
}

const double = multiply.bind(null, 2);

console.log(double(5)); // 10
console.log(double(8)); // 16
```

Here, `2` is pre-filled as the first argument. Later, when we call `double(5)`, it becomes:

```js
multiply(2, 5);
```

The first argument to `bind` is the `this` value. Since `multiply` does not use `this`, we pass `null`.

This is useful when we want to create specialized functions from more generic functions.

#### Real-World Usage

Partial application is useful for event handlers, reusable utilities, analytics tracking, and pre-configured callbacks. In React, we often use inline arrow functions instead, but understanding `bind` helps with JavaScript fundamentals and older React code.

## 40. What happens when you use new with a function?

When `new` is used with a function, JavaScript creates a new object, links it to the function’s prototype, binds `this` to that new object, and returns the object by default.

#### Explanation

Constructor functions are regular functions intended to be called with `new`.

```js
function User(name) {
  this.name = name;
}

const user = new User("Alex");

console.log(user.name); // "Alex"
```

When this runs, JavaScript roughly does:

```txt
1. Creates a new empty object
2. Links the object to User.prototype
3. Calls User with this pointing to the new object
4. Returns the new object unless the constructor returns another object explicitly
```

That is why methods added to the prototype are available on instances.

```js
function User(name) {
  this.name = name;
}

User.prototype.sayHello = function () {
  return `Hello, ${this.name}`;
};

const user = new User("Alex");

console.log(user.sayHello()); // "Hello, Alex"
```

Arrow functions cannot be used with `new` because they do not have their own `this` and do not have a `prototype` property.

#### Real-World Usage

Modern React functional components do not use `new`, but understanding it is important for JavaScript fundamentals, classes, prototypes, object creation, and older React class components.

## 41. Can arrow functions be used as constructors?

No, arrow functions cannot be used as constructors. They do not have their own `this`, they do not have a `prototype` property, and calling them with `new` throws a `TypeError`.

#### Explanation

Constructor functions need their own `this` so JavaScript can create a new object and bind `this` to that object.

Regular functions can be used as constructors:

```js
function User(name) {

```

Arrow functions cannot be used this way:

```js
const User = (name) => {
  this.name = name;
};

const user = new User("Alex"); // TypeError: User is not a constructor
```

Arrow functions are designed for lexical `this`, not object construction.

In modern React functional components, we rarely write constructor functions directly. But this concept is important when working with JavaScript classes, prototypes, older libraries, and interview questions around object creation.

## 42. What is the arguments object?

#### Explanation

```js
function sum() {
```

```js
function example() {

  const args = Array.from(arguments);

  return args.reduce((total, value) => total + value, 0);
}

console.log(example(1, 2, 3)); // 6
```

In modern JavaScript, rest parameters are usually preferred over `arguments`.

#### Real-World Usage

You may still see `arguments` in older JavaScript utilities, libraries, or legacy React code. In modern code, rest parameters are cleaner and safer.

## 43. Why do arrow functions not have their own arguments object?

Arrow functions do not have their own `arguments` object because they do not create their own function binding for `this`, `arguments`, `super`, or `new.target`.

#### Explanation

An arrow function uses the `arguments` object from its surrounding regular function scope, if one exists.

```js
function outer() {
  const arrow = () => {
```

The `"inner"` argument passed to `arrow` is not captured by an arrow-specific `arguments` object because arrow functions do not create one.

For arrow functions, use rest parameters instead.

```js
const sum = (...numbers) => {
```

## 44. What are default parameters?

Default parameters allow function parameters to have fallback values when an argument is `undefined` or not provided.

#### Explanation

Default parameters make function APIs cleaner and reduce manual checks inside the function body.

```js
function createUser(name = "Guest", role = "viewer") {
  return {
    name,
    role,
  };
}

console.log(createUser());
// { name: "Guest", role: "viewer" }

console.log(createUser("Alex", "admin"));
// { name: "Alex", role: "admin" }
```

A key point is that default parameters apply only when the argument is `undefined`.

```js
function greet(name = "Guest") {
  return `Hello, ${name}`;
}

console.log(greet(undefined)); // "Hello, Guest"
console.log(greet(null)); // "Hello, null"
```

`null` is treated as an intentional value, so the default is not used.

#### Real-World Usage

In React, default parameters are useful in utility functions, custom hooks, and helper functions. For component props, default values are commonly handled during destructuring.

```jsx
function Button({ variant = "primary", disabled = false }) {
  return <button disabled={disabled}>{variant}</button>;
}
```

## 45. What is the difference between rest parameters and the arguments object?

Rest parameters collect function arguments into a real array, while the `arguments` object is array-like and only available in regular functions.

#### Explanation

Rest parameters are modern, explicit, and work with both regular functions and arrow functions.

```js
function sum(...numbers) {
  return numbers.reduce((total, value) => total + value, 0);
}

console.log(sum(1, 2, 3)); // 6
```

Here, `numbers` is a real array, so array methods work directly.

The `arguments` object is only available inside regular functions and is not a real array.

```js
function sumOldWay() {
  const numbers = Array.from(arguments);

  return numbers.reduce((total, value) => total + value, 0);
}

console.log(sumOldWay(1, 2, 3)); // 6
```

Rest parameters are also more readable because they clearly show that the function accepts a variable number of arguments.

```js
function log(level, ...messages) {
  console.log(level, messages);
}

log("info", "User logged in", "Dashboard loaded");
```

#### Real-World Usage

In modern React and JavaScript, rest parameters are preferred for reusable utilities, event wrappers, logging helpers, and flexible APIs. They are cleaner than `arguments` and work naturally with arrow functions.

## 46. How are objects stored and compared in JavaScript?

Objects in JavaScript are stored and compared by reference, not by their actual content.

#### Explanation

When we assign an object to a variable, the variable holds a reference to the object in memory. If we assign that variable to another variable, both variables point to the same object.

```js
const user1 = { name: "Alex" };
const user2 = user1;

user2.name = "John";

console.log(user1.name); // "John"
```

Both `user1` and `user2` refer to the same object.

When comparing objects, JavaScript compares the references, not the values inside the objects.

```js
const a = { id: 1 };
const b = { id: 1 };

console.log(a === b); // false
```

Even though both objects have the same content, they are different objects in memory.

```js
const c = a;

console.log(a === c); // true
```

Here, `a` and `c` point to the same object reference.

#### Real-World Usage

In React, object references are very important for state updates, memoization, dependency arrays, and rendering optimization. If we mutate the same object reference, React may not detect the change correctly.

## 47. What is object destructuring?

```js
const user = {
```

Default values can also be used.

```js
const user = {
  name: "Alex",
};

const { role = "viewer" } = user;

console.log(role); // "viewer"
```

#### Real-World Usage

Object destructuring is heavily used in React for props, state, hooks, and API responses.

```jsx
function UserCard({ name, role = "viewer" }) {
```

## 48. What is array destructuring?

## 49. What is the difference between shallow copy and deep copy?

A shallow copy copies only the top-level properties, while nested objects are still shared by reference. A deep copy copies all levels, including nested objects, so the new object is fully independent.

#### Explanation

With a shallow copy, primitive top-level values are copied, but nested objects still point to the same reference.

```js
const user = {
  name: "Alex",
  address: {
    city: "New York",
  },
};

const copy = { ...user };

copy.name = "John";
copy.address.city = "Boston";

console.log(user.name); // "Alex"
console.log(user.address.city); // "Boston"
```

`name` changed only in the copy because it is a primitive top-level value. But `address.city` changed in both because `address` is still shared.

A deep copy creates a new copy of nested objects too.

```js
const deepCopy = structuredClone(user);

deepCopy.address.city = "Chicago";

console.log(user.address.city); // "Boston"
console.log(deepCopy.address.city); // "Chicago"
```

#### Real-World Usage

In React, shallow copying is often enough when updating one level of state, but nested state requires copying each changed level. For complex nested updates, teams often normalize state or use libraries like Immer.

## 50. What are different ways to clone an object?

Objects can be cloned using the spread operator, `Object.assign()`, `structuredClone()`, JSON serialization, or custom/library-based deep cloning depending on the use case.

#### Explanation

For shallow cloning, the spread operator is commonly used.

```js
const user = {
  name: "Alex",
  role: "admin",
};

const copy = { ...user };
```

`Object.assign()` also creates a shallow copy.

```js
const copy = Object.assign({}, user);
```

For deep cloning, `structuredClone()` is the modern built-in option.

```js
const user = {
  name: "Alex",
  address: {
    city: "New York",
  },
};

const copy = structuredClone(user);
```

JSON serialization can also deep clone simple data, but it has limitations.

```js
const copy = JSON.parse(JSON.stringify(user));
```

However, JSON cloning does not handle functions, `undefined`, `Date`, `Map`, `Set`, circular references, or special object types correctly.

For complex cloning requirements, a utility like `lodash.cloneDeep()` is often used.

#### Real-World Usage

In React, we usually avoid deep cloning entire objects unless required because it can be expensive. For state updates, it is better to copy only the parts that changed and keep the rest of the references stable.

## 51. What are the limitations of JSON.parse(JSON.stringify(obj)) for cloning?

`JSON.parse(JSON.stringify(obj))` can deep clone only simple JSON-safe data. It loses or breaks many JavaScript-specific values and object types.

#### Explanation

This approach converts the object to a JSON string and then parses it back into a new object.

It works for simple objects:

```js
const user = {
  name: "Alex",
  address: {
    city: "New York",
  },
};

const copy = JSON.parse(JSON.stringify(user));
```

But it has important limitations:

- Removes properties with `undefined`
- Removes functions
- Converts `Date` objects into strings
- Does not preserve `Map`, `Set`, `RegExp`, `Error`, or class instances correctly
- Fails on circular references
- Loses prototype information
- Does not preserve special values like `Infinity`, `-Infinity`, and `NaN` correctly

```js
const obj = {
  name: "Alex",
  age: undefined,
  greet() {

```

`age` and `greet` are removed, `Date` becomes a string, and `NaN` becomes `null`.

## 52. What is structuredClone()?

`structuredClone()` is a modern built-in JavaScript API used to create a deep clone of many JavaScript values more safely than JSON cloning.

#### Explanation

`structuredClone()` creates a deep copy using the structured clone algorithm.

```js
const user = {
  name: "Alex",
  address: {
    city: "New York",
  },
  preferences: new Map([["theme", "dark"]]),
};

const copy = structuredClone(user);

copy.address.city = "Boston";

console.log(user.address.city); // "New York"
console.log(copy.address.city); // "Boston"
```

Unlike JSON cloning, `structuredClone()` supports many complex values such as:

- Nested objects
- Arrays
- Dates
- Maps
- Sets
- ArrayBuffers
- Circular references

```js
const obj = {
  date: new Date(),
  roles: new Set(["admin", "editor"]),
};

const copy = structuredClone(obj);

console.log(copy.date instanceof Date); // true
console.log(copy.roles instanceof Set); // true
```

However, it still cannot clone functions, DOM nodes, or some custom object behavior.

#### Real-World Usage

In React, `structuredClone()` can be useful for cloning complex plain state during data transformation. But for normal state updates, we should still prefer copying only the changed parts instead of deep cloning everything.

## 53. What is a prototype in JavaScript?

A prototype is an object from which other objects can inherit properties and methods.

#### Explanation

Every JavaScript object has an internal link to another object called its prototype. If a property or method is not found on the object itself, JavaScript looks for it on the prototype.

```js
const user = {
  name: "Alex",
};

console.log(user.toString()); // inherited from Object.prototype
```

Here, `toString` is not directly defined on `user`, but it is available through the prototype chain.

Constructor functions use the `prototype` property to share methods across instances.

```js
function User(name) {
  this.name = name;
}

User.prototype.sayHello = function () {
  return `Hello, ${this.name}`;
};

const user = new User("Alex");

console.log(user.sayHello()); // "Hello, Alex"
```

The `sayHello` method is not copied into every object. It is shared through `User.prototype`.

#### Real-World Usage

Most modern React code does not manually modify prototypes, but understanding prototypes is important for JavaScript classes, inheritance, built-in methods, object behavior, and interview questions.

## 54. What is prototypal inheritance?

Prototypal inheritance is JavaScript’s inheritance model where objects can inherit properties and methods from other objects through prototypes.

#### Explanation

Instead of classical inheritance where classes are the main building blocks, JavaScript objects can directly inherit from other objects.

```js
const animal = {
  speak() {
    return "Animal sound";
  },
};

const dog = Object.create(animal);

dog.name = "Buddy";

console.log(dog.name); // "Buddy"
console.log(dog.speak()); // "Animal sound"
```

Here, `dog` does not have its own `speak` method. JavaScript finds `speak` on `animal`, which is the prototype of `dog`.

ES6 classes are mostly cleaner syntax over JavaScript’s prototype-based inheritance.

```js
class User {
  constructor(name) {
    this.name = name;
  }

  sayHello() {
    return `Hello, ${this.name}`;
  }
}
```

Internally, methods like `sayHello` are placed on `User.prototype`.

#### Real-World Usage

Prototypal inheritance helps explain how built-in JavaScript methods work, how classes work internally, and how objects share behavior efficiently without duplicating methods.

## 55. What is the prototype chain?

The prototype chain is the lookup path JavaScript follows when trying to access a property or method on an object.

#### Explanation

When we access a property, JavaScript first checks the object itself. If it does not find the property, it checks the object’s prototype. If it still does not find it, it continues moving up the prototype chain until it reaches `null`.

```js
const animal = {
  speak() {
    return "Animal sound";
  },
};

const dog = Object.create(animal);

dog.name = "Buddy";

console.log(dog.name); // found on dog
console.log(dog.speak()); // found on animal prototype
```

The lookup flow is:

```txt
dog -> animal -> Object.prototype -> null
```

If the property is not found anywhere in the chain, the result is `undefined`.

```js
console.log(dog.age); // undefined
```

The prototype chain is also why normal objects can access methods like `toString`.

```js
const user = { name: "Alex" };

console.log(user.toString()); // inherited from Object.prototype
```

#### Real-World Usage

Understanding the prototype chain is useful for debugging unexpected inherited properties, class behavior, method lookup, and performance issues caused by deep inheritance chains.

## 56. What is the difference between `__proto__` and `prototype`?

`__proto__` is the actual prototype link of an object, while `prototype` is a property that exists on constructor functions and is used when creating objects with `new`.

#### Explanation

Every object has an internal prototype link. Historically, we access that link using `__proto__`.

```js
const user = {
  name: "Alex",
};

console.log(user.__proto__ === Object.prototype); // true
```

`prototype` is different. It exists on functions that can be used as constructors.

```js
function User(name) {
  this.name = name;
}

const user = new User("Alex");

console.log(user.__proto__ === User.prototype); // true
```

When we call `new User("Alex")`, JavaScript creates a new object and links that object’s internal prototype to `User.prototype`.

A cleaner modern way to inspect prototypes is:

```js
Object.getPrototypeOf(user);
```

So the simple distinction is:

```txt
__proto__  -> prototype link of an object
prototype  -> property on constructor functions used for instances
```

#### Real-World Usage

In day-to-day React development, we rarely use `__proto__` or manually work with prototypes. But this is important for understanding classes, inheritance, object creation, and JavaScript internals.

## 57. How does JavaScript look up a property on an object?

JavaScript first checks whether the property exists directly on the object. If it does not, JavaScript looks up the prototype chain until it finds the property or reaches `null`.

#### Explanation

This lookup process is called prototype chain lookup.

```js
const animal = {
  speak() {
    return "Animal sound";
  },
};

const dog = Object.create(animal);

dog.name = "Buddy";

console.log(dog.name); // found directly on dog
console.log(dog.speak()); // found on animal prototype
```

For `dog.name`, JavaScript finds the property directly on `dog`.

For `dog.speak()`, JavaScript does not find `speak` on `dog`, so it checks `dog`’s prototype, which is `animal`.

The lookup flow is:

```txt
dog -> animal -> Object.prototype -> null
```

If JavaScript cannot find the property anywhere in the chain, it returns `undefined`.

```js
console.log(dog.age); // undefined
```

#### Real-World Usage

This helps explain how built-in methods like `toString`, `map`, and `filter` are available even when they are not directly defined on the object or array instance.

## 58. What is the difference between own properties and inherited properties?

Own properties are properties that exist directly on the object. Inherited properties come from the object’s prototype chain.

#### Explanation

An object can have its own properties and also access properties from its prototype.

```js
const animal = {
  type: "animal",
  speak() {
    return "Animal sound";
  },
};

const dog = Object.create(animal);

dog.name = "Buddy";

console.log(dog.name); // own property
console.log(dog.type); // inherited property
console.log(dog.speak()); // inherited method
```

Here, `name` belongs directly to `dog`, while `type` and `speak` are inherited from `animal`.

We can check own properties using `Object.hasOwn()` or `hasOwnProperty()`.

```js
console.log(Object.hasOwn(dog, "name")); // true
console.log(Object.hasOwn(dog, "type")); // false
```

The `in` operator checks both own and inherited properties.

```js
console.log("name" in dog); // true
console.log("type" in dog); // true
```

#### Real-World Usage

This matters when iterating over objects, validating data, checking config objects, or avoiding bugs caused by inherited properties appearing in lookup checks.

## 59. What is the difference between `Object.keys()`, `Object.values()`, and `Object.entries()`?

`Object.keys()` returns an array of property names, `Object.values()` returns an array of property values, and `Object.entries()` returns an array of key-value pairs.

#### Explanation

All three methods work on an object’s own enumerable properties.

```js
const user = {

// ["id", "name", "role"]

console.log(Object.values(user));
// [1, "Alex", "admin"]

console.log(Object.entries(user));
```

```js
Object.entries(user).forEach(([key, value]) => {
  console.log(`${key}: ${value}`);

```

These methods do not include inherited properties.

```js
const parent = { inherited: true };
const child = Object.create(parent);

child.own = true;

console.log(Object.keys(child)); // ["own"]
```

## 60. What is the difference between `in` operator and `hasOwnProperty()`?

The `in` operator checks whether a property exists on the object or anywhere in its prototype chain, while `hasOwnProperty()` checks only whether the property exists directly on the object.

#### Explanation

```js
const parent = {
  inheritedValue: true,
};

const child = Object.create(parent);

child.ownValue = true;

console.log("ownValue" in child); // true
console.log("inheritedValue" in child); // true

console.log(child.hasOwnProperty("ownValue")); // true
console.log(child.hasOwnProperty("inheritedValue")); // false
```

The `in` operator returns `true` for both own and inherited properties.

`hasOwnProperty()` returns `true` only for direct properties.

A safer modern option is `Object.hasOwn()` because some objects may not inherit from `Object.prototype`, or they may override `hasOwnProperty`.

```js
console.log(Object.hasOwn(child, "ownValue")); // true
console.log(Object.hasOwn(child, "inheritedValue")); // false
```

#### Real-World Usage

In production JavaScript, use `Object.hasOwn()` or `hasOwnProperty()` when validating object data. Use `in` when inherited properties should also count, such as checking feature support or property availability across an object’s prototype chain.

## 61. What are getters and setters?

Getters and setters are special methods that allow us to control how object properties are read and updated.

#### Explanation

A getter runs when a property is accessed. A setter runs when a property is assigned a value.

```js
const user = {
  firstName: "Alex",
  lastName: "Smith",

  get fullName() {
    return `${this.firstName} ${this.lastName}`;
  },

  set fullName(value) {
    const [firstName, lastName] = value.split(" ");
    this.firstName = firstName;
    this.lastName = lastName;
  },
};

console.log(user.fullName); // "Alex Smith"

user.fullName = "John Doe";

console.log(user.firstName); // "John"
console.log(user.lastName); // "Doe"
```

From the outside, `fullName` looks like a normal property, but internally it runs logic.

Getters are useful for computed values. Setters are useful when we want validation, transformation, or controlled updates.

#### Real-World Usage

In frontend applications, getters and setters are useful in model objects, SDKs, stores, and libraries where property access needs extra logic. In React components, we usually prefer explicit functions and state updates instead of relying heavily on getters and setters.

## 62. What is the difference between class syntax and prototype-based inheritance?

## 63. What does `super` do in a JavaScript class?

`super` is used to call the parent class constructor or parent class methods from a child class.

#### Explanation

In a subclass constructor, we must call `super()` before using `this`.

```js
class Animal {
  constructor(name) {
    this.name = name;
  }

  speak() {
    return `${this.name} makes a sound`;
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name);
    this.breed = breed;
  }

  speak() {
    return `${super.speak()} as a ${this.breed}`;
  }
}

const dog = new Dog("Buddy", "Labrador");

console.log(dog.speak());
// "Buddy makes a sound as a Labrador"
```

Here:

- `super(name)` calls the parent `Animal` constructor
- `super.speak()` calls the parent class method

Without `super()` in a derived class constructor, JavaScript does not know how to initialize `this`.

#### Real-World Usage

`super` is mostly seen in class-based code, framework internals, SDKs, and older React class components. In modern React functional components, we usually do not use `super`.

## 64. What are static methods in classes?

```js
class MathUtils {
```

We do not call static methods on an instance.

```js
const utils = new MathUtils();
```

Static methods are useful for utility behavior related to the class but not dependent on instance-specific data.

```js
class User {

```

Here, `fromApi` is a factory method that creates a `User` instance.

#### Real-World Usage

Static methods are useful for factory methods, validation helpers, utility methods, and parsing logic. In frontend code, they are commonly seen in service classes, SDK clients, and model classes.

## 65. What are private class fields?

Private class fields are class properties that can only be accessed from inside the class. They are declared using the `#` syntax.

#### Explanation

Private fields provide real language-level privacy.

```js
class Counter {
  #count = 0;

  increment() {
    this.#count += 1;
  }

  getValue() {
    return this.#count;
  }
}

const counter = new Counter();

counter.increment();

console.log(counter.getValue()); // 1

// console.log(counter.#count); // SyntaxError
```

The `#count` field cannot be accessed or modified directly from outside the class.

This is different from the older convention of using underscores.

```js
class User {
  constructor() {
    this._token = "abc";
  }
}
```

`_token` is only a naming convention. It is still publicly accessible. `#token` is actually private.

#### Real-World Usage

Private fields are useful when building reusable JavaScript classes, SDKs, services, and internal libraries where implementation details should not be exposed. In React functional components, local variables, closures, and hooks usually handle private component state instead.

## 66. What is the difference between `map`, `filter`, and `reduce`?

`map` transforms each item and returns a new array of the same length. `filter` selects items and returns a new array with matching items. `reduce` combines array items into a single accumulated result.

#### Explanation

Use `map` when you want to convert each item into something else.

```js
const numbers = [1, 2, 3];

const doubled = numbers.map((num) => num * 2);

console.log(doubled); // [2, 4, 6]
```

Use `filter` when you want only items that match a condition.

```js
const numbers = [1, 2, 3, 4];

const evenNumbers = numbers.filter((num) => num % 2 === 0);

console.log(evenNumbers); // [2, 4]
```

Use `reduce` when you want to accumulate values into one result.

```js
const numbers = [1, 2, 3];

const total = numbers.reduce((sum, num) => sum + num, 0);

console.log(total); // 6
```

Simple rule:

```txt
map    -> transform
filter -> select
reduce -> accumulate
```

#### Real-World Usage

In React, `map` is commonly used to render lists, `filter` is used for search or visibility logic, and `reduce` is useful for grouping, totals, lookup maps, and transforming API responses.

## 67. What is the difference between `forEach` and `map`?

`forEach` is used for side effects and returns `undefined`, while `map` is used to transform an array and returns a new array.

## 68. Why does `forEach` not work well with `async/await`?

`forEach` does not work well with `async/await` because it does not wait for promises returned from its callback, and the outer function does not pause until all async operations finish.

#### Explanation

`forEach` executes callbacks synchronously and ignores the returned promises.

```js
async function processUsers(users) {
  users.forEach(async (user) => {
    await saveUser(user);
  });

  console.log("Done");
}
```

In this example, `"Done"` may log before `saveUser` finishes for all users.

For sequential async processing, use `for...of`.

```js
async function processUsers(users) {
  for (const user of users) {
    await saveUser(user);
  }

  console.log("Done");
}
```

For parallel async processing, use `Promise.all`.

```js
async function processUsers(users) {
  await Promise.all(users.map((user) => saveUser(user)));

  console.log("Done");
}
```

The right choice depends on whether the operations must run one by one or can run in parallel.

#### Real-World Usage

In frontend apps, this matters when making multiple API calls, uploading files, validating form rows, or processing bulk actions. Use `for...of` for ordered execution and `Promise.all` for parallel execution.

## 69. What is the difference between `find` and `filter`?

`find` returns the first matching item, while `filter` returns all matching items in a new array.

```js
const users = [
```

```js
const user = users.find((user) => user.id === 99);
```

```js
const activeUsers = [

```

`find` stops as soon as it finds the first match. `filter` checks the whole array.

In React, `find` is useful for selecting one item by ID, such as selected user or selected product. `filter` is useful for rendering lists based on search, category, status, or permissions.

## 70. What is the difference between `some` and `every`?

`some` checks whether at least one item matches a condition, while `every` checks whether all items match a condition.

#### Explanation

```js
const users = [
```

```js
const allActive = users.every((user) => user.active);

```

Both methods return booleans and short-circuit:

```txt
some  -> stops when it finds true
every -> stops when it finds false
```

In React applications, `some` is useful for checks like “does this user have any required permission?” and `every` is useful for validations like “are all form sections complete?”

## 71. What is the difference between `slice` and `splice`?

`slice` returns a shallow copy of part of an array without changing the original array. `splice` changes the original array by adding, removing, or replacing elements.

#### Explanation

`slice` is immutable. It takes a start index and an optional end index. The end index is not included.

```js
const numbers = [1, 2, 3, 4, 5];

const result = numbers.slice(1, 4);

console.log(result); // [2, 3, 4]
console.log(numbers); // [1, 2, 3, 4, 5]
```

`splice` is mutable. It starts at an index and removes a given number of items. It can also insert new items.

```js
const numbers = [1, 2, 3, 4, 5];

const removed = numbers.splice(1, 2);

console.log(removed); // [2, 3]
console.log(numbers); // [1, 4, 5]
```

`splice` can also replace items.

```js
const numbers = [1, 2, 3];

numbers.splice(1, 1, 20);

console.log(numbers); // [1, 20, 3]
```

Simple rule:

```txt
slice  -> copies part of an array
splice -> modifies the original array
```

#### Real-World Usage

In React, `slice` is safe for deriving new arrays from state. `splice` should be avoided directly on state because it mutates the original array and can cause rendering bugs.

## 72. What is the difference between `push`, `pop`, `shift`, and `unshift`?

`push` adds an item to the end, `pop` removes an item from the end, `shift` removes an item from the beginning, and `unshift` adds an item to the beginning.

#### Explanation

All four methods mutate the original array.

```js
const numbers = [1, 2, 3];

console.log(numbers); // [1, 2, 3]

console.log(numbers); // [2, 3]

console.log(numbers); // [0, 2, 3]
```

```js
const numbers = [1, 2, 3];

console.log(numbers.push(4)); // 4, new length
console.log(numbers.pop()); // 4, removed item
console.log(numbers.shift()); // 1, removed item
console.log(numbers.unshift(0)); // 3, new length
```

Simple rule:

```txt
push    -> add at end
pop     -> remove from end
shift   -> remove from start
unshift -> add at start
```

#### Real-World Usage

In React state updates, avoid using these methods directly on state arrays. Instead, create a new array using spread, `filter`, `slice`, or other immutable patterns.

## 73. What is the difference between mutable and immutable array methods?

Mutable array methods change the original array. Immutable array methods return a new array or value without changing the original array.

#### Explanation

Mutable methods modify the same array reference.

```js
const numbers = [1, 2, 3];

numbers.push(4);

console.log(numbers); // [1, 2, 3, 4]
```

Common mutable methods include:

```txt
push, pop, shift, unshift, splice, sort, reverse, fill
```

Immutable methods do not change the original array.

```js
const numbers = [1, 2, 3];

const updated = [...numbers, 4];

console.log(updated); // [1, 2, 3, 4]
console.log(numbers); // [1, 2, 3]
```

Common immutable methods or patterns include:

```txt
map, filter, reduce, slice, concat, spread, toSorted, toReversed, toSpliced
```

A common gotcha is `sort`.

```js
const numbers = [3, 1, 2];

const sorted = numbers.sort();

console.log(sorted); // [1, 2, 3]
console.log(numbers); // [1, 2, 3]
```

`sort` mutates the original array. To avoid mutation:

```js
const numbers = [3, 1, 2];

const sorted = [...numbers].sort((a, b) => a - b);

console.log(sorted); // [1, 2, 3]
console.log(numbers); // [3, 1, 2]
```

#### Real-World Usage

In React, immutable updates are important because React compares references to decide whether state changed. Mutating an existing array can prevent expected re-renders or create hard-to-debug UI issues.

## 74. What are sparse arrays?

Sparse arrays are arrays that have empty slots, meaning some indexes do not actually contain values.

#### Explanation

A sparse array is different from an array containing `undefined`.

```js
const sparse = [1, , 3];

console.log(sparse.length); // 3
console.log(sparse[1]); // undefined
```

Even though `sparse[1]` returns `undefined`, the index does not actually exist.

```js
console.log(1 in sparse); // false
```

Compare that with an explicit `undefined` value:

```js
const dense = [1, undefined, 3];

console.log(dense.length); // 3
console.log(dense[1]); // undefined
console.log(1 in dense); // true
```

Sparse arrays can also be created using `Array(length)`.

```js
const items = Array(3);

console.log(items); // [empty × 3]
console.log(items.length); // 3
```

#### Real-World Usage

Sparse arrays can create confusing behavior when rendering lists, transforming data, or working with array iteration methods. In React, it is better to use dense arrays with real values instead of relying on empty slots.

## 75. What happens when you use `map` on a sparse array?

When `map` is used on a sparse array, it skips empty slots and preserves them in the returned array.

#### Explanation

`map` only runs the callback for indexes that actually exist.

```js
const sparse = [1, , 3];

const result = sparse.map((item) => item * 2);

console.log(result); // [2, empty, 6]
```

The callback does not run for the empty slot.

```js
const sparse = [1, , 3];

sparse.map((item, index) => {
  console.log(index, item);
});

// 0 1
// 2 3
```

The empty slot at index `1` is skipped.

This is different from an explicit `undefined` value.

```js
const dense = [1, undefined, 3];

dense.map((item, index) => {
  console.log(index, item);
});

// 0 1
// 1 undefined
// 2 3
```

If you want to create a dense array from a length, use `Array.from`.

```js
const items = Array.from({ length: 3 }, (_, index) => index);

console.log(items); // [0, 1, 2]
```

#### Real-World Usage

In React, sparse arrays can cause unexpected rendering behavior because some indexes are skipped. When generating lists, prefer `Array.from`, real API data, or explicitly filled arrays.

## 76. What is the difference between `Array.from()` and spread syntax?

`Array.from()` creates an array from array-like or iterable values and can also map values during creation. Spread syntax expands iterable values into a new array, but it does not work with all array-like objects.

#### Explanation

Spread syntax works with iterables such as arrays, strings, Sets, and Maps.

```js
const set = new Set([1, 2, 3]);

const result = [...set];

console.log(result); // [1, 2, 3]
```

`Array.from()` also works with iterables.

```js
const set = new Set([1, 2, 3]);

const result = Array.from(set);

console.log(result); // [1, 2, 3]
```

The important difference is that `Array.from()` can also convert array-like objects, such as objects with `length` and indexed properties.

```js
const arrayLike = {
  0: "a",
  1: "b",
  length: 2,
};

console.log(Array.from(arrayLike)); // ["a", "b"]
```

Spread would not work here unless the object is iterable.

```js
// [...arrayLike]; // TypeError: arrayLike is not iterable
```

`Array.from()` can also apply a mapping function during creation.

```js
const numbers = Array.from({ length: 3 }, (_, index) => index + 1);

console.log(numbers); // [1, 2, 3]
```

#### Real-World Usage

In React, `Array.from()` is useful for generating fixed-length UI lists, skeleton loaders, pagination numbers, or converting array-like values. Spread is commonly used for copying arrays, merging arrays, and converting Sets into arrays.

## 77. How would you remove duplicates from an array?

The most common way to remove duplicates from an array is to use a `Set`, because a `Set` stores only unique values.

#### Explanation

For primitive values, `Set` is clean and efficient.

```js
const numbers = [1, 2, 2, 3, 3, 4];

const uniqueNumbers = [...new Set(numbers)];

console.log(uniqueNumbers); // [1, 2, 3, 4]
```

For strings, it works the same way.

```js
const names = ["Alex", "John", "Alex"];

const uniqueNames = [...new Set(names)];

console.log(uniqueNames); // ["Alex", "John"]
```

For objects, `Set` compares by reference, not by object content.

```js
const users = [{ id: 1 }, { id: 1 }];

console.log([...new Set(users)]); // [{ id: 1 }, { id: 1 }]
```

To remove duplicates from an array of objects, deduplicate by a specific key.

```js
const users = [
  { id: 1, name: "Alex" },
  { id: 2, name: "John" },
  { id: 1, name: "Alex" },

const uniqueUsers = Array.from(
  new Map(users.map((user) => [user.id, user])).values(),
);

console.log(uniqueUsers);
// [{ id: 1, name: "Alex" }, { id: 2, name: "John" }]
```

## 78. How would you group an array of objects by a property?

I would usually use `reduce` to group an array of objects by a property into an object or a `Map`.

```js
const users = [

const groupedByRole = users.reduce((groups, user) => {

console.log(groupedByRole);

//   ],
//   user: [
//     { name: "John", role: "user" }
//   ]
// }
```

If keys can be non-string values or we need better key handling, `Map` is a good option.

```js
const grouped = new Map();

for (const user of users) {
  const key = user.role;

    grouped.set(key, []);
  }

  grouped.get(key).push(user);
}
```

## 79. How would you flatten a nested array?

I would use `flat()` for simple flattening, and pass a depth depending on how deeply nested the array is.

#### Explanation

For one-level nested arrays, use `flat()`.

```js
const values = [1, [2, 3], [4, 5]];

const flattened = values.flat();

console.log(flattened); // [1, 2, 3, 4, 5]
```

For multiple levels, pass a depth.

```js
const values = [1, [2, [3, [4]]]];

console.log(values.flat(2)); // [1, 2, 3, [4]]
```

To flatten all levels, use `Infinity`.

```js
const values = [1, [2, [3, [4]]]];

console.log(values.flat(Infinity)); // [1, 2, 3, 4]
```

For older environments, a recursive approach can be used.

```js
function flatten(array) {
  return array.reduce((result, item) => {
    return result.concat(Array.isArray(item) ? flatten(item) : item);
  }, []);
}

console.log(flatten([1, [2, [3]]])); // [1, 2, 3]
```

## 80. How would you sort an array of numbers correctly?

To sort numbers correctly, provide a numeric comparator to `sort`, such as `(a, b) => a - b` for ascending order.

#### Explanation

By default, JavaScript `sort()` converts values to strings and sorts them lexicographically.

```js
const numbers = [10, 2, 30, 4];

numbers.sort();

console.log(numbers); // [10, 2, 30, 4]
```

This is not numeric sorting. To sort correctly, use a comparator.

```js
const numbers = [10, 2, 30, 4];

numbers.sort((a, b) => a - b);

console.log(numbers); // [2, 4, 10, 30]
```

For descending order:

```js
const numbers = [10, 2, 30, 4];

numbers.sort((a, b) => b - a);

console.log(numbers); // [30, 10, 4, 2]
```

One important point is that `sort()` mutates the original array. To avoid mutation, copy first.

```js
const numbers = [10, 2, 30, 4];

const sorted = [...numbers].sort((a, b) => a - b);

console.log(sorted); // [2, 4, 10, 30]
console.log(numbers); // [10, 2, 30, 4]
```

#### Real-World Usage

In React, avoid sorting state arrays directly because `sort()` mutates the original array. Always create a new array before sorting to keep state updates immutable and predictable.

## 81. What is asynchronous JavaScript?

Asynchronous JavaScript allows code to start a long-running operation, such as an API call or timer, without blocking the rest of the program from running.

#### Explanation

JavaScript is single-threaded for executing code, but the browser or runtime can handle async operations outside the main call stack.

Examples of asynchronous operations include:

```txt
setTimeout
fetch
Promise
async/await
event listeners
file operations in Node.js
```

Example:

```js
console.log("Start");

setTimeout(() => {
  console.log("Timer finished");
}, 1000);

console.log("End");
```

Output:

```txt
Start
End
Timer finished
```

`setTimeout` does not block the next line. JavaScript continues running synchronous code, and the callback runs later.

#### Real-World Usage

In React applications, asynchronous JavaScript is used for API calls, debounced search, form submissions, lazy loading, analytics events, and background data refreshes.

## 82. What is the event loop?

The event loop is the mechanism that coordinates the call stack, task queues, and asynchronous callbacks in JavaScript.

#### Explanation

JavaScript executes synchronous code on the call stack. When async work completes, its callback is placed into a queue. The event loop checks whether the call stack is empty, then moves queued callbacks to the stack for execution.

Simple flow:

```txt
1. Run synchronous code on the call stack
2. Async operations are handled by the runtime
3. Completed callbacks go into queues
4. Event loop waits for the call stack to be empty
5. Event loop pushes queued callbacks back onto the stack
```

Example:

```js
console.log("A");

setTimeout(() => {
  console.log("B");
}, 0);

console.log("C");
```

Output:

```txt
A
C
B
```

Even with `0ms`, `setTimeout` runs after synchronous code because its callback must wait for the call stack to clear.

#### Real-World Usage

The event loop is important for understanding UI responsiveness, async rendering behavior, timers, promises, API calls, and why heavy synchronous work can freeze the browser.

## 83. What is the difference between call stack, callback queue, and microtask queue?

The call stack runs currently executing code. The callback queue holds macrotask callbacks like `setTimeout`. The microtask queue holds higher-priority callbacks like `Promise.then`, and it is processed before the callback queue.

#### Explanation

The **call stack** contains active execution contexts.

```js
function first() {
  second();
}

function second() {
  console.log("Done");
}

first();
```

Here, function calls are pushed and popped from the call stack.

The **callback queue**, also called the macrotask queue, stores callbacks from APIs like:

```txt
setTimeout
setInterval
DOM events
message events
```

The **microtask queue** stores callbacks from:

```txt
Promise.then
Promise.catch
Promise.finally
queueMicrotask
MutationObserver
```

After synchronous code finishes, JavaScript clears the microtask queue before processing the next macrotask.

```js
console.log("Start");

setTimeout(() => console.log("Timeout"), 0);

Promise.resolve().then(() => console.log("Promise"));

console.log("End");
```

Output:

```txt
Start
End
Promise
Timeout
```

#### Real-World Usage

This is important when debugging async ordering issues in React apps, especially with promises, timers, effects, state updates, and user interactions.

## 84. What is the difference between macro tasks and microtasks?

Macrotasks are larger scheduled tasks like timers and browser events. Microtasks are smaller high-priority tasks like promise callbacks that run immediately after the current synchronous code finishes and before the next macrotask.

#### Explanation

Common macrotasks include:

```txt
setTimeout
setInterval
setImmediate in Node.js
I/O callbacks
UI events
```

Common microtasks include:

```txt
Promise.then
Promise.catch
Promise.finally
queueMicrotask
MutationObserver
```

Execution order is generally:

```txt
1. Run current synchronous code
2. Clear all microtasks
3. Run one macrotask
4. Clear microtasks again
5. Continue the cycle
```

Example:

```js
setTimeout(() => {
  console.log("macrotask");
}, 0);

Promise.resolve().then(() => {
  console.log("microtask");
});

console.log("sync");
```

Output:

```txt
sync
microtask
macrotask
```

A key detail is that the microtask queue is drained fully before the next macrotask runs. If microtasks keep adding more microtasks, they can delay timers and rendering.

#### Real-World Usage

In React and frontend performance work, understanding microtasks and macrotasks helps explain timing issues, promise behavior, UI updates, rendering delays, and why long-running JavaScript blocks user interactions.

## 85. What is the execution order of `setTimeout`, `Promise.then`, and synchronous code?

Synchronous code runs first, `Promise.then` runs next as a microtask, and `setTimeout` runs after that as a macrotask.

#### Explanation

Example:

```js
console.log("1");

setTimeout(() => {
  console.log("2");
}, 0);

Promise.resolve().then(() => {
  console.log("3");
});

console.log("4");
```

Output:

```txt
1
4
3
2
```

Step-by-step:

```txt
1. console.log("1") runs immediately
2. setTimeout callback is scheduled as a macrotask
3. Promise.then callback is scheduled as a microtask
4. console.log("4") runs immediately
5. Microtask queue runs, so "3" is logged
6. Macrotask queue runs, so "2" is logged
```

Even though `setTimeout` has `0ms`, it does not run immediately. It waits until the call stack is empty and microtasks are completed.

#### Real-World Usage

This is useful when debugging async bugs in React apps, especially around API responses, timers, loading states, state updates, and code that depends on execution order.

## 86. What is a Promise?

A `Promise` is a JavaScript object that represents the eventual result of an asynchronous operation, either success or failure.

#### Explanation

```js
const promise = fetch("/api/users");

    console.error(error);
  });
```

A Promise helps avoid deeply nested callbacks and gives a structured way to handle async results.

At a high level:

```txt
Promise -> represents a future value
.then() -> handles success
.catch() -> handles failure
.finally() -> runs cleanup logic
```

#### Real-World Usage

In React applications, Promises are used for API calls, lazy loading, authentication flows, file uploads, and background data fetching.

## 87. What are the possible states of a Promise?

A Promise has three possible states: `pending`, `fulfilled`, and `rejected`.

#### Explanation

When a Promise is first created, it starts in the `pending` state.

If the async operation succeeds, the Promise becomes `fulfilled`.

If the async operation fails, the Promise becomes `rejected`.

```js
const promise = new Promise((resolve, reject) => {
  const success = true;

  if (success) {
    resolve("Data loaded");
  } else {
    reject("Something went wrong");
  }
});
```

Promise states:

```txt
pending   -> initial state
fulfilled -> completed successfully
rejected  -> completed with failure
```

Once a Promise is fulfilled or rejected, it is settled and cannot change state again.

```js
const promise = new Promise((resolve, reject) => {
  resolve("Success");
  reject("Error"); // ignored
});
```

The first settled result wins.

#### Real-World Usage

In React, these states usually map to UI states like loading, success, and error. Most data-fetching components manage these states either manually or through libraries like React Query.

## 88. What is the difference between `Promise.resolve()` and `new Promise()`?

`Promise.resolve()` creates an already resolved Promise from a value, while `new Promise()` is used when we need to manually control when to resolve or reject an async operation.

#### Explanation

Use `Promise.resolve()` when you already have a value and want to wrap it in a Promise.

```js
const promise = Promise.resolve("Success");

promise.then((value) => {
  console.log(value); // "Success"
});
```

Use `new Promise()` when you need to create a custom async operation.

```js
const promise = new Promise((resolve, reject) => {
  setTimeout(() => {
    resolve("Data loaded");
  }, 1000);
});
```

`Promise.resolve()` is simpler and safer when no custom async control is needed.

Another important behavior is that `Promise.resolve()` can adopt another Promise’s state.

```js
const original = fetch("/api/users");

const wrapped = Promise.resolve(original);

console.log(wrapped === original); // true
```

#### Real-World Usage

In real projects, `Promise.resolve()` is useful for normalizing sync and async results into a Promise. `new Promise()` is useful when wrapping callback-based APIs, timers, or custom async logic.

## 89. What is promise chaining?

Promise chaining is the process of returning values or Promises from `.then()` callbacks so async operations can run in a sequence.

#### Explanation

Each `.then()` returns a new Promise. The value returned from one `.then()` becomes the input to the next `.then()`.

```js
fetch("/api/users")

  .then((activeUsers) => {
    console.log(activeUsers);
  })
  .catch((error) => {
    console.error(error);
  });
```

If a `.then()` returns a Promise, the next `.then()` waits for it to resolve.

```js
getUser()
  .then((user) => getOrders(user.id))
  .then((orders) => {
    console.log(orders);
  });
```

This avoids callback nesting and keeps async flows readable.

A key interview point is: always return the Promise if the next step depends on it.

```js
getUser()
  .then((user) => {
    return getOrders(user.id);
  })
  .then((orders) => {
    console.log(orders);
  });
```

#### Real-World Usage

## 90. What is the difference between `.then()` and `async/await`?

`.then()` and `async/await` both work with Promises, but `async/await` lets us write asynchronous code in a more synchronous-looking style.

#### Explanation

Using `.then()`:

```js
function loadUsers() {
  return fetch("/api/users")
    .then((response) => response.json())
    .then((users) => {
      return users;
    })
    .catch((error) => {
      console.error(error);
    });
}
```

Using `async/await`:

```js
async function loadUsers() {
  try {
    const response = await fetch("/api/users");
    const users = await response.json();

    return users;
  } catch (error) {
    console.error(error);
  }
}
```

Both versions are Promise-based. An `async` function always returns a Promise.

```js
async function getValue() {
  return 10;
}

console.log(getValue()); // Promise
```

The main difference is readability and error handling style:

```txt
.then()      -> callback-based Promise chaining
async/await -> synchronous-looking Promise code with try/catch
```

`async/await` is often easier to read for sequential async logic. `.then()` can still be useful for simple chains or when composing Promise flows directly.

#### Real-World Usage

In React applications, `async/await` is commonly used inside event handlers, service functions, and data-fetching utilities. Promise chaining is still useful in reusable async helpers and functional composition.

## 91. How does error handling work with promises?

Promise errors are handled using `.catch()` in promise chains or `try/catch` when using `async/await`.

#### Explanation

When a Promise is rejected, the rejection travels down the promise chain until it finds a `.catch()` handler.

```js
fetch("/api/users")
  .then((response) => response.json())
  .then((users) => {
    console.log(users);
  })
  .catch((error) => {
    console.error("Failed to load users:", error);
  });
```

A `.catch()` handles:

- Explicit `reject()` calls
- Errors thrown inside `.then()`
- Failed async operations

```js
Promise.resolve()
  .then(() => {
    throw new Error("Something failed");
  })
  .catch((error) => {
    console.error(error.message); // "Something failed"
  });
```

With `async/await`, Promise rejections are handled using `try/catch`.

```js
async function loadUsers() {
  try {
    const response = await fetch("/api/users");
    const users = await response.json();

    return users;
  } catch (error) {
    console.error("Failed to load users:", error);
  }
}
```

One important point: if you do not handle a rejected Promise, it can result in an unhandled promise rejection.

#### Real-World Usage

In React apps, promise error handling is commonly used for API calls, form submissions, authentication, file uploads, and background data fetching. Good error handling helps show proper error UI instead of silently failing.

## 92. What is the difference between `Promise.all`, `Promise.allSettled`, `Promise.race`, and `Promise.any`?

`Promise.all` waits for all promises to succeed, `Promise.allSettled` waits for all promises to finish, `Promise.race` settles as soon as the first promise settles, and `Promise.any` resolves as soon as the first promise succeeds.

#### Explanation

`Promise.all` is used when all operations must succeed.

```js
const [user, orders] = await Promise.all([fetchUser(), fetchOrders()]);
```

If any promise rejects, `Promise.all` rejects immediately.

`Promise.allSettled` waits for every promise, whether fulfilled or rejected.

```js
const results = await Promise.allSettled([fetchUser(), fetchOrders()]);
```

It returns result objects with `status`.

```js
[
  { status: "fulfilled", value: user },
  { status: "rejected", reason: error },
];
```

`Promise.race` returns the first settled result, either fulfilled or rejected.

```js
const result = await Promise.race([fetchData(), timeoutPromise()]);
```

`Promise.any` returns the first successfully fulfilled promise. It ignores rejections unless all promises reject.

```js
const result = await Promise.any([fetchFromServerA(), fetchFromServerB()]);
```

Simple comparison:

```txt
Promise.all        -> all must succeed
Promise.allSettled -> wait for all results
Promise.race       -> first settled wins
Promise.any        -> first successful wins
```

#### Real-World Usage

In frontend apps, `Promise.all` is useful for loading independent page data together. `Promise.allSettled` is useful when partial failure is acceptable. `Promise.race` is useful for timeout patterns. `Promise.any` is useful for fallback sources or fastest successful response.

## 93. What happens if one promise fails inside `Promise.all()`?

If one promise fails inside `Promise.all()`, the entire `Promise.all()` rejects immediately with that error.

#### Explanation

`Promise.all()` is fail-fast. It resolves only when all promises fulfill. If any promise rejects, the returned promise rejects.

```js
async function loadData() {
  try {
    const result = await Promise.all([
      fetchUser(),
      fetchOrders(),
      fetchSettings(),
    ]);

    console.log(result);
  } catch (error) {
    console.error("One request failed:", error);
  }
}
```

If `fetchOrders()` rejects, the `catch` block runs, and we do not get the successful results from the other promises through `Promise.all()`.

Important point: `Promise.all()` rejects immediately from the caller’s perspective, but the other already-started async operations are not automatically cancelled.

```js
Promise.all([
  Promise.resolve("A"),
  Promise.reject("B failed"),
  Promise.resolve("C"),
]).catch((error) => {
  console.log(error); // "B failed"
});
```

If we need all results, including failures, we should use `Promise.allSettled()`.

```js
const results = await Promise.allSettled([
  fetchUser(),
  fetchOrders(),
  fetchSettings(),
]);
```

#### Real-World Usage

In React, use `Promise.all()` when the screen cannot render unless all required data is available. Use `Promise.allSettled()` when you can show partial data with individual error states.

## 94. How do you run async operations sequentially?

To run async operations sequentially, use `async/await` with a `for...of` loop or await each operation one after another.

#### Explanation

Sequential execution means the next async operation starts only after the previous one finishes.

```js
async function processUsers(users) {
  for (const user of users) {
    await saveUser(user);
  }

  console.log("All users processed");
}
```

This is different from starting all operations at once.

Sequential execution is useful when order matters or when each step depends on the previous result.

```js
async function checkout() {
  const cart = await createCart();
  const payment = await processPayment(cart.id);
  const order = await createOrder(payment.id);

  return order;
}
```

Avoid using `forEach` for sequential async work because it does not wait for each async callback.

```js
// Avoid this for sequential async operations
users.forEach(async (user) => {
  await saveUser(user);
});
```

#### Real-World Usage

Sequential async operations are useful for dependent API calls, ordered workflows, rate-limited requests, transaction-like flows, and multi-step form submissions.

## 95. How do you run async operations in parallel?

To run async operations in parallel, start them together and wait for them using `Promise.all()`.

#### Explanation

Parallel execution means multiple async operations are started at the same time instead of waiting for each one to finish before starting the next.

```js
async function loadPageData() {
  const [user, orders, settings] = await Promise.all([
    fetchUser(),
    fetchOrders(),
    fetchSettings(),
  ]);

  return {
    user,
    orders,
    settings,
  };
}
```

This is faster when the operations are independent.

For an array of items, use `map` to create promises and then await them together.

```js
async function saveUsers(users) {
  await Promise.all(users.map((user) => saveUser(user)));

  console.log("All users saved");
}
```

If partial failure is acceptable, use `Promise.allSettled()` instead.

```js
const results = await Promise.allSettled(users.map((user) => saveUser(user)));
```

A key interview point is: parallel execution improves performance only when operations do not depend on each other.

#### Real-World Usage

In React apps, parallel async operations are useful when loading independent page data, fetching dropdown options, preloading resources, or submitting independent background requests.

## 96. What is cancellation in async JavaScript?

Cancellation in async JavaScript means stopping or ignoring an asynchronous operation when its result is no longer needed.

#### Explanation

JavaScript Promises themselves cannot be truly cancelled once they start. However, we can cancel certain underlying operations, like `fetch`, using APIs such as `AbortController`.

For example, if a user types quickly in a search box, older API requests may no longer be useful. We should cancel the previous request or ignore its result.

There are two common strategies:

```txt
1. Cancel the actual request using AbortController
2. Ignore outdated responses using request IDs or flags
```

Cancellation helps avoid:

- Unnecessary network work
- Updating UI with stale data
- Memory leaks after component unmount
- Race conditions between multiple async calls

#### Real-World Usage

In React, cancellation is commonly used in `useEffect` when fetching data based on changing props, route params, filters, or search input. When the component unmounts or input changes, the previous request should be cancelled or ignored.

## 97. What is `AbortController`?

`AbortController` is a browser API that allows us to cancel abortable async operations, most commonly `fetch` requests.

#### Explanation

`AbortController` creates a `signal`. That signal is passed to an async operation like `fetch`. When we call `abort()`, the request is cancelled and the Promise rejects with an abort-related error.

```jsx
function UserProfile({ userId }) {
  const [user, setUser] = React.useState(null);

  React.useEffect(() => {
    const controller = new AbortController();

    async function loadUser() {
      try {
        const response = await fetch(`/api/users/${userId}`, {
          signal: controller.signal,
        });

        const data = await response.json();
        setUser(data);
      } catch (error) {
        if (error.name !== "AbortError") {
          console.error("Failed to load user:", error);
        }
      }
    }

    loadUser();

    return () => {
      controller.abort();
    };
  }, [userId]);

  return <div>{user?.name}</div>;
}
```

When `userId` changes or the component unmounts, the cleanup function aborts the previous request.

This prevents old requests from completing and updating state after they are no longer relevant.

#### Real-World Usage

`AbortController` is useful for search autocomplete, route-based data fetching, filter changes, modal data loading, and any API call that may become stale before completion.

## 98. How would you avoid race conditions in async code?

I would avoid async race conditions by ensuring only the latest request is allowed to update state, and by cancelling or ignoring older requests.

#### Explanation

A race condition happens when multiple async operations run at the same time and complete in an unexpected order.

Example problem:

```txt
1. User searches "re"
2. API request A starts
3. User quickly searches "react"
4. API request B starts
5. Request B finishes first and shows correct data
6. Request A finishes later and overwrites UI with stale data
```

Common fixes include:

```txt
1. Abort older requests using AbortController
2. Track request IDs and update state only for the latest request
3. Debounce user input to reduce unnecessary calls
4. Use a data-fetching library that handles stale requests
```

A simple request ID approach:

```js
let latestRequestId = 0;

async function searchUsers(query) {
  const requestId = ++latestRequestId;

  const response = await fetch(`/api/users?q=${query}`);
  const data = await response.json();

  if (requestId === latestRequestId) {
    return data;
  }

  return null;
}
```

Only the latest request is allowed to return usable data.

#### Real-World Usage

In React, this is common in autocomplete, filters, pagination, route changes, tab switching, and dashboards where users can trigger multiple API calls quickly.

## 99. How would you implement retry logic for a failed API call?

I would implement retry logic by wrapping the API call in a function that retries only for safe, transient failures, usually with a retry limit and backoff delay.

#### Explanation

Not every failure should be retried. Retrying makes sense for temporary issues like network errors, timeouts, or `5xx` server errors. It usually should not retry validation errors or most `4xx` responses.

```js
async function fetchWithRetry(url, options = {}, retries = 3) {
  for (let attempt = 1; attempt <= retries; attempt++) {
    try {
      const response = await fetch(url, options);

      if (!response.ok) {
        throw new Error(`Request failed with status ${response.status}`);
      }

      return response.json();
    } catch (error) {
      if (attempt === retries) {
        throw error;
      }

      await new Promise((resolve) => setTimeout(resolve, attempt * 500));
    }
  }
}
```

This retries with a simple increasing delay:

```txt
Attempt 1 -> fail
Wait 500ms
Attempt 2 -> fail
Wait 1000ms
Attempt 3 -> final try
```

In production, I would also consider exponential backoff, jitter, request cancellation, logging, and whether the request is idempotent.

#### Real-World Usage

Retry logic is useful for unstable networks, background refreshes, analytics, non-critical API calls, and resilient data loading. In React apps, libraries like React Query often provide retry behavior out of the box.

## 100. In production, users report stale data after multiple quick API calls. How would you debug and fix it?

I would suspect an async race condition first. I would verify which requests are being fired, which response finishes last, and whether older responses are overwriting newer UI state.

#### Explanation

The debugging approach would be:

```txt
1. Reproduce the issue with quick user actions
2. Check the Network tab for request order and response timing
3. Add request IDs or correlation IDs to logs
4. Confirm whether an older response updates state after a newer response
5. Check React effects, dependency arrays, and cleanup logic
6. Verify caching behavior from browser, service worker, or data-fetching library
```

The likely fix is to make sure only the latest request can update the UI.

One approach is to cancel old requests with `AbortController`.

```js
const controller = new AbortController();

fetch(url, { signal: controller.signal });

controller.abort();
```

Another approach is to track the latest request and ignore stale responses.

```js
let latestRequestId = 0;

async function loadData(query) {
  const requestId = ++latestRequestId;

  const response = await fetch(`/api/search?q=${query}`);
  const data = await response.json();

  if (requestId !== latestRequestId) {
    return;
  }

  updateUI(data);
}
```

In a React app, I would usually fix this inside the data-fetching layer or hook, not separately in every component.

I would also consider:

```txt
- Debouncing fast user input
- Cancelling requests on dependency change
- Using React Query/SWR stale-time and cache controls correctly
- Showing loading states tied to the latest request
- Avoiding state updates after unmount
```

#### Real-World Usage

This issue is common in autocomplete, search filters, dashboards, pagination, route changes, and tabbed UIs. A senior-level fix is not only patching one component, but creating a reusable request strategy that prevents stale updates across the application.
