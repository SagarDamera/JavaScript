# Topic: useCallback

- `useCallback` memoizes a function reference between renders.

- Use it when passing callbacks to memoized child components.

- It helps prevent unnecessary child re-renders caused by changing function references.

- It returns the same function unless one of its dependencies changes.

- Dependencies must be correct; missing dependencies can cause stale data bugs.

- Do not use `useCallback` for every function; it adds complexity without always improving performance.

- `useCallback(() => fn, deps)` is similar to `useMemo(() => fn, deps)`.

- Common mistake: using `useCallback` while the child component is not memoized with `React.memo`.

- Interview trap: `useCallback` does not stop the parent from re-rendering; it only stabilizes function identity.

- Real-world use: optimized lists, table row actions, memoized children, and event handlers passed deep into components.

## One-Line Final Recall

`useCallback` keeps a function reference stable between renders to help avoid unnecessary re-renders in memoized children.

---

### 1. What problem does `useCallback` solve in React?

`useCallback` helps keep the same function reference between renders.  
It is mainly useful when passing callbacks to memoized child components or using functions as dependencies.

```jsx
const handleClick = useCallback(() => {
  console.log("Clicked");
}, []);
```

Tip: `useCallback` memoizes function identity, not the function result.

---

### 2. What does `useCallback` return?

`useCallback` returns a memoized version of the function you pass to it.  
React returns the same function reference unless one of its dependencies changes.

```jsx
const handleSave = useCallback(() => {
  saveUser(user);
}, [user]);
```

Tip: `useCallback` returns a function.

---

### 3. How is `useCallback` different from defining a function directly inside a component?

A normal function inside a component is recreated on every render.  
`useCallback` keeps the same function reference when dependencies do not change.

```jsx
// New function every render
const handleClick = () => {
  setCount(count + 1);
};

// Same function until count changes
const handleClickMemo = useCallback(() => {
  setCount(count + 1);
}, [count]);
```

Tip: `useCallback` is useful when function reference stability matters.

---

### 4. When should you use `useCallback` in a real React application?

Use `useCallback` when passing a function to a `React.memo` child or when a function is used inside another hook dependency array.  
It helps avoid unnecessary child renders or repeated effect executions.

```jsx
const handleSelect = useCallback((id) => {
  setSelectedId(id);
}, []);

return <UserList onSelect={handleSelect} />;
```

Tip: Use `useCallback` for real reference-stability needs, not for every function.

---

### 5. How does the dependency array control when `useCallback` creates a new function?

React compares the dependencies between renders.  
If any dependency changes, React creates and returns a new function reference.

```jsx
const handleSearch = useCallback(() => {
  fetchResults(searchText);
}, [searchText]);
```

Tip: The callback updates only when one of its dependencies changes.

---

### 6. What happens if the dependency array is empty in `useCallback`?

If the dependency array is empty, React creates the function only during the initial render.  
After that, React keeps returning the same function reference for that component instance.

```jsx
const handleClick = useCallback(() => {
  console.log("Clicked");
}, []);
```

Tip: Empty dependency array means the callback does not update with changing props or state.

---

### 7. What happens if you omit the dependency array in `useCallback`?

If you omit the dependency array, React creates a new function on every render.  
In that case, `useCallback` gives no memoization benefit.

```jsx
const handleClick = useCallback(() => {
  console.log("Clicked");
});
```

Tip: Without dependencies, `useCallback` behaves like a normal inline function.

---

### 8. Can `useCallback` prevent a component from re-rendering?

No, `useCallback` cannot prevent the current component from re-rendering.  
It only keeps a function reference stable between renders.

```jsx
const handleSave = useCallback(() => {
  saveData();
}, []);
```

Tip: `useCallback` helps with function identity, not render blocking.

---

### 9. How does `useCallback` help when passing functions to `React.memo` child components?

`React.memo` skips re-rendering when props are the same by reference.  
`useCallback` keeps the function prop reference stable, so the child does not re-render unnecessarily.

```jsx
const handleSelect = useCallback((id) => {
  setSelectedId(id);
}, []);

return <MemoizedUserList onSelect={handleSelect} />;
```

Tip: `useCallback` is most useful when combined with memoized child components.

---

### 10. What is the difference between `useCallback` and `useMemo`?

`useCallback` memoizes a function reference.  
`useMemo` memoizes the result of a calculation.

```jsx
const handleClick = useCallback(() => {
  setCount(count + 1);
}, [count]);

const doubled = useMemo(() => {
  return count * 2;
}, [count]);
```

Tip: Use `useCallback` for functions and `useMemo` for computed values.

---

### 11. Why can incorrect dependencies in `useCallback` cause stale closures?

`useCallback` captures values from the render where the function was created.  
If a changing value is missing from the dependency array, the callback may keep using an old value.

```jsx
const handleSubmit = useCallback(() => {
  console.log(searchText);
}, []); // missing searchText
```

Tip: Include every changing value used inside the callback.

---

### 12. How can adding unnecessary dependencies to `useCallback` reduce its benefit?

`useCallback` creates a new function whenever any dependency changes.  
If you add dependencies that are not used inside the function, the callback changes more often than needed.

```jsx
const handleClick = useCallback(() => {
  setCount(count + 1);
}, [count, theme]); // theme is unnecessary
```

Tip: Extra dependencies can break function reference stability.

---

### 13. Why is `useCallback` often unnecessary for simple inline event handlers?

Simple inline handlers are cheap to recreate during render.  
Using `useCallback` everywhere can add complexity without improving performance.

```jsx
<button onClick={() => setOpen(true)}>Open</button>
```

Tip: Do not use `useCallback` unless function identity matters.

---

### 14. What are the performance costs of using `useCallback` itself?

React must store the function and compare dependencies on every render.  
For simple handlers, this overhead can be more than recreating the function normally.

```jsx
const handleClick = useCallback(() => {
  setCount(count + 1);
}, [count]);
```

Tip: `useCallback` is an optimization, not a default requirement.

---

### 15. How does `useCallback` behave during the initial render?

During the initial render, React stores the function you pass to `useCallback`.  
On later renders, React reuses it if dependencies have not changed.

```jsx
const handleSave = useCallback(() => {
  saveUser(user);
}, [user]);
```

Tip: `useCallback` always creates/stores the first function during initial render.

---

### 16. How does `useCallback` behave when a parent component re-renders but dependencies do not change?

When the component re-renders, React checks the dependencies.  
If dependencies are unchanged, React returns the same function reference from the previous render.

```jsx
const handleClick = useCallback(() => {
  console.log("Clicked");
}, []);
```

Tip: Re-render does not always mean a new callback is created.

---

### 17. What happens if a dependency is an object or array created inline during render?

An inline object or array gets a new reference on every render.  
React sees it as changed, so `useCallback` creates a new function every time.

```jsx
const options = { role: "admin" };

const handleFilter = useCallback(() => {
  filterUsers(options);
}, [options]);
```

Tip: Inline objects/arrays can break callback memoization.

---

### 18. How can `useCallback` be useful with `useEffect` dependencies?

If a function is used inside `useEffect`, it may need to be included in the dependency array.  
`useCallback` keeps that function stable so the effect does not run unnecessarily.

```jsx
const fetchData = useCallback(() => {
  return fetch(`/api/users/${userId}`);
}, [userId]);

useEffect(() => {
  fetchData();
}, [fetchData]);
```

Tip: `useCallback` helps keep effect dependencies predictable.

---

### 19. How can `useCallback` help avoid repeatedly subscribing or unsubscribing inside `useEffect`?

If an effect depends on a function that changes every render, the effect cleanup and setup may run repeatedly.  
`useCallback` keeps the function reference stable until its real dependencies change.

```jsx
const handleMessage = useCallback((message) => {
  console.log(message);
}, []);

useEffect(() => {
  socket.subscribe(handleMessage);

  return () => {
    socket.unsubscribe(handleMessage);
  };
}, [handleMessage]);
```

Tip: Stable callbacks help avoid unnecessary effect cleanup and re-subscription.

---

### 20. What is the relationship between `useCallback` and function identity?

In JavaScript, every new function has a new identity/reference.  
`useCallback` preserves the same function identity between renders when dependencies do not change.

```jsx
const handleSave = useCallback(() => {
  saveData();
}, []);
```

Tip: `useCallback` is mainly about stable function identity.

---

### 21. Why does wrapping every function in `useCallback` usually make code worse?

Wrapping every function in `useCallback` adds extra dependency management and makes code harder to read.  
It can also add overhead without improving performance if function identity does not matter.

```jsx
// Often unnecessary
const handleClick = useCallback(() => {
  setOpen(true);
}, []);

// Simple and fine
const handleClick = () => {
  setOpen(true);
};
```

Tip: Use `useCallback` only when stable function reference is actually useful.

---

### 22. How can updater functions help reduce dependencies inside `useCallback`?

Updater functions let you update state based on the previous value without reading that value from the closure.  
This can remove the state value from the dependency array.

```jsx
const handleAdd = useCallback(() => {
  setCount((prevCount) => prevCount + 1);
}, []);
```

Tip: Prefer updater functions when the new state depends on the previous state.

---

### 23. Can `useCallback` guarantee that a function will never be recreated?

No, `useCallback` is a performance optimization, not a permanent guarantee.  
React may return the cached function when dependencies are unchanged, but your logic should not depend on it for correctness.

```jsx
const handleSave = useCallback(() => {
  saveUser(user);
}, [user]);
```

Tip: Use `useCallback` for optimization, not application correctness.

---

### 24. How does React internally associate a `useCallback` call with a component Fiber across renders?

React stores hook data in a linked list on the componentâ€™s Fiber.  
On each render, React matches hooks by call order, so the same `useCallback` position gets the previous function and dependencies.

```jsx
const handleSave = useCallback(() => saveUser(user), [user]);
const handleCancel = useCallback(() => closeModal(), []);
```

Tip: React tracks hooks by call order, not by function or variable name.

---

### 25. Why canâ€™t `useCallback` be called conditionally inside a component?

Hooks must run in the same order on every render.  
Calling `useCallback` conditionally changes the hook order and breaks Reactâ€™s ability to match hook state correctly.

```jsx
// Wrong
if (isEditMode) {
  const handleSave = useCallback(() => saveUser(user), [user]);
}

// Correct
const handleSave = useCallback(() => {
  if (isEditMode) {
    saveUser(user);
  }
}, [isEditMode, user]);
```

Tip: Always call `useCallback` at the top level of the component.
