# Topic: useReducer

- `useReducer` is used to manage complex state logic in functional components.

- It works with a reducer function that takes current state and action, then returns new state.

- Use it when state has multiple related values or many update cases.

- It is better than `useState` when updates depend on action types or complex rules.

- Reducer must be pure; do not call APIs or mutate state inside it.

- Always return a new state object/array instead of modifying existing state directly.

- Actions usually have a `type` and optional `payload`.

- Common mistake: using `useReducer` for simple state like a boolean toggle or input value.

- Interview trap: `useReducer` is local component state, not automatically global state.

- Real-world use: forms, multi-step flows, cart logic, filters, and complex UI state.

## One-Line Final Recall

`useReducer` manages complex local state using actions and a pure reducer function.

---

### 1. What problem does `useReducer` solve in React?

`useReducer` helps manage complex state logic in a predictable way.  
It is useful when state has multiple fields, many update paths, or transitions based on action types.

```jsx
const [state, dispatch] = useReducer(reducer, initialState);
```

Tip: `useReducer` is best when state updates are more complex than simple `setState`.

---

### 2. What does `useReducer` return?

`useReducer` returns an array with two values: the current state and a `dispatch` function.  
The `dispatch` function is used to send actions to the reducer.

```jsx
const [state, dispatch] = useReducer(reducer, {
  count: 0,
});
```

Tip: `useReducer` returns `[state, dispatch]`.

---

### 3. How is `useReducer` different from `useState`?

`useState` updates state directly using a setter function.  
`useReducer` updates state by dispatching actions to a reducer function.

```jsx
// useState
const [count, setCount] = useState(0);

// useReducer
const [state, dispatch] = useReducer(reducer, { count: 0 });
```

Tip: Use `useState` for simple state and `useReducer` for structured state logic.

---

### 4. When should you use `useReducer` instead of `useState`?

Use `useReducer` when state updates depend on action types or when multiple state values change together.  
It also helps when state logic is complex enough to move outside the component body.

```jsx
dispatch({ type: "increment" });
dispatch({ type: "reset" });
```

Tip: Choose `useReducer` when the update logic needs clear rules.

---

### 5. What is the role of the reducer function in `useReducer`?

The reducer receives the current state and an action, then returns the next state.  
It centralizes state update logic in one predictable function.

```jsx
function reducer(state, action) {
  switch (action.type) {
    case "increment":
      return { count: state.count + 1 };
    default:
      return state;
  }
}
```

Tip: A reducer describes how state changes for each action.

---

### 6. What is the role of the dispatch function in `useReducer`?

`dispatch` is used to send an action to the reducer.  
It tells React what happened, and React uses the reducer to calculate the next state.

```jsx
dispatch({ type: "increment" });
```

Tip: `dispatch` triggers state updates through the reducer.

---

### 7. What is an action in `useReducer`?

An action is a value that describes what happened.  
Usually, it is an object with a `type` field and optional extra data called payload.

```jsx
dispatch({
  type: "addTodo",
  payload: "Learn useReducer",
});
```

Tip: Actions describe events, not directly how state changes.

---

### 8. Why should a reducer function be pure?

A reducer should always return the same next state for the same state and action.  
It should not perform side effects like API calls, timers, logging, or mutations.

```jsx
function reducer(state, action) {
  if (action.type === "increment") {
    return { count: state.count + 1 };
  }

  return state;
}
```

Tip: Pure reducers make state updates predictable and easier to test.

---

### 9. What happens when you call `dispatch` in a component?

React schedules an update for that component.  
During the next render, React calls the reducer with the current state and dispatched action.

```jsx
<button onClick={() => dispatch({ type: "increment" })}>Increment</button>
```

Tip: `dispatch` does not immediately change state in the same render.

---

### 10. How does React decide whether to re-render after a reducer returns new state?

React compares the previous state and the new state using `Object.is`.  
If the state is different, React re-renders the component.

```jsx
function reducer(state, action) {
  if (action.type === "increment") {
    return { count: state.count + 1 };
  }

  return state;
}
```

Tip: Returning a new state reference usually causes a re-render.

---

### 11. What happens if the reducer returns the same state object?

If the reducer returns the same state object, React usually skips re-rendering because the state reference did not change.  
React compares old and new state using `Object.is`.

```jsx
function reducer(state, action) {
  if (action.type === "unknown") {
    return state;
  }
}
```

Tip: Same state reference means React treats state as unchanged.

---

### 12. Why should you avoid mutating state inside a reducer?

Mutating state changes the existing object instead of creating a new one.  
React may not detect the change if the same object reference is returned.

```jsx
// Wrong
state.count += 1;
return state;

// Correct
return {
  ...state,
  count: state.count + 1,
};
```

Tip: Always return a new object when state changes.

---

### 13. How do you handle multiple action types inside a reducer?

Use a `switch` statement or conditional logic based on `action.type`.  
Each action type should clearly return the next state for that specific update.

```jsx
function reducer(state, action) {
  switch (action.type) {
    case "increment":
      return { count: state.count + 1 };

    case "decrement":
      return { count: state.count - 1 };

    default:
      return state;
  }
}
```

Tip: Action types make complex state updates easier to organize.

---

### 14. What is the purpose of the initial state argument in `useReducer`?

The initial state argument provides the starting state for the reducer.  
React uses it when the component renders for the first time.

```jsx
const initialState = {
  count: 0,
};

const [state, dispatch] = useReducer(reducer, initialState);
```

Tip: Initial state is used only to set up the reducer state initially.

---

### 15. What is lazy initialization in `useReducer`?

Lazy initialization lets you calculate the initial state using a separate initializer function.  
React calls this function only during the initial render.

```jsx
function init(initialCount) {
  return {
    count: initialCount,
  };
}

const [state, dispatch] = useReducer(reducer, 0, init);
```

Tip: Lazy initialization is useful when creating initial state is expensive.

---

### 16. When is lazy initialization useful in `useReducer`?

Lazy initialization is useful when the initial state is expensive to calculate.  
It is also useful when initial state needs to be derived from props only once during the first render.

```jsx
function init(userData) {
  return {
    name: userData.name,
    permissions: calculatePermissions(userData),
  };
}

const [state, dispatch] = useReducer(reducer, user, init);
```

Tip: Lazy initialization avoids recalculating initial state on every render.

---

### 17. Can `useReducer` replace Redux?

`useReducer` can replace Redux for small or medium local state logic.  
But Redux provides extra features like global store, middleware, devtools, selectors, and better large-scale state patterns.

```jsx
const [state, dispatch] = useReducer(reducer, initialState);
```

Tip: `useReducer` manages component state; Redux manages app-level state.

---

### 18. How can `useReducer` be combined with `useContext`?

`useReducer` can manage state, and `useContext` can share that state and dispatch function across components.  
This pattern is useful for simple global state without adding a separate library.

```jsx
const AppContext = createContext(null);

function AppProvider({ children }) {
  const [state, dispatch] = useReducer(reducer, initialState);

  return (
    <AppContext.Provider value={{ state, dispatch }}>
      {children}
    </AppContext.Provider>
  );
}
```

Tip: `useReducer + useContext` is a common lightweight state management pattern.

---

### 19. Why is `dispatch` considered stable across renders?

React guarantees that the `dispatch` function identity stays the same between renders.  
That means React does not create a new `dispatch` function every time the component renders.

```jsx
const [state, dispatch] = useReducer(reducer, initialState);
```

Tip: `dispatch` has a stable reference.

---

### 20. Do you need to include `dispatch` in dependency arrays?

Usually no, because React guarantees `dispatch` has a stable identity.  
Including it does not hurt, but it is not necessary.

```jsx
useEffect(() => {
  dispatch({ type: "load" });
}, []); // dispatch is safe to omit
```

Tip: `dispatch` can be safely omitted from dependency arrays.

---

### 21. How can stale closures happen when using `useReducer`?

Stale closures usually happen in event handlers, effects, or callbacks that capture old state values.  
The reducer itself receives the latest state from React, but surrounding functions can still close over outdated values.

```jsx
const handleLog = useCallback(() => {
  console.log(state.count);
}, []); // missing state.count
```

Tip: `dispatch` is stable, but values used around it can still become stale.

---

### 22. Why should side effects not be placed inside a reducer?

Reducers should be pure and only calculate the next state.  
Side effects like API calls, timers, navigation, or localStorage updates should happen outside the reducer.

```jsx
// Wrong
function reducer(state, action) {
  fetch("/api/save");
  return state;
}
```

Tip: Reducers should return state, not perform side effects.

---

### 23. How should async logic be handled when using `useReducer`?

Handle async logic outside the reducer, usually inside `useEffect` or an event handler.  
Dispatch actions for loading, success, and error states.

```jsx
useEffect(() => {
  dispatch({ type: "loading" });

  fetchUsers()
    .then((data) => dispatch({ type: "success", payload: data }))
    .catch((error) => dispatch({ type: "error", payload: error.message }));
}, []);
```

Tip: Async work happens outside; state transitions happen inside the reducer.

---

### 24. How does React internally associate a `useReducer` call with a component Fiber across renders?

React stores hook data in a linked list on the component’s Fiber.  
On every render, React matches hooks by call order, so the same `useReducer` position gets the previous state queue.

```jsx
const [countState, countDispatch] = useReducer(countReducer, countInitial);
const [formState, formDispatch] = useReducer(formReducer, formInitial);
```

Tip: React tracks hooks by order, not by variable name.

---

### 25. Why can’t `useReducer` be called conditionally inside a component?

Hooks must run in the same order on every render.  
Calling `useReducer` conditionally changes the hook order and breaks React’s ability to match previous hook state.

```jsx
// Wrong
if (isFormVisible) {
  const [state, dispatch] = useReducer(reducer, initialState);
}

// Correct
const [state, dispatch] = useReducer(reducer, initialState);
```

Tip: Always call `useReducer` at the top level of the component.
