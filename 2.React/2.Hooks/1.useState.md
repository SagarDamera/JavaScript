# Topic: useState

- `useState` is used to add local state to functional components.

- It returns two values: current state and a setter function.

- Calling the setter updates state and triggers a re-render.

- Never update state directly; always use the setter.

- Use functional update when new state depends on previous state.

- State updates may be batched, so do not expect immediate updated value after setter call.

- Initial state is used only on the first render.

- Use lazy initialization when initial state calculation is expensive.

- Common mistake: mutating arrays or objects instead of creating a new copy.

- Real-world use: form inputs, toggles, counters, selected items, modal open/close, and loading flags.

## One-Line Final Recall

`useState` lets functional components store local data and re-render when that data changes.

---

### 1. Why does `useState` persist state across renders even though the component function runs again?

`useState` persists state because React stores the state outside the component function, inside React's internal data structure for that component. When the component re-renders, the function runs again, but React gives back the previously stored state value instead of recreating it.

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

Even though `Counter()` runs again after every update, `count` does not reset to `0`.

---

### 2. What problem does `useState` solve compared to using a normal local variable inside a component?

A normal local variable is recreated every time the component function runs, so it cannot preserve values between renders. `useState` solves this by letting React remember the value and trigger a re-render when the value changes.

```jsx
function Counter() {
  let count = 0;

  return <button onClick={() => count++}>{count}</button>;
}
```

Here, updating `count` does not update the UI because React is not tracking this variable.

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

Here, React tracks `count` and re-renders the component when it changes.

---

### 3. When is the initial value passed to `useState` used, and what happens to it after the first render?

The initial value passed to `useState` is used only during the initial mount. After the first render, React ignores that initial value and returns the latest stored state from the previous render.

```jsx
const [count, setCount] = useState(0);
```

`0` is used only on the first render. After `setCount(5)`, the next render returns `5`, not `0`.

---

### 4. What is lazy initialization in `useState`, and when would you use it?

Lazy initialization means passing a function to `useState` so React runs it only during the initial render. It is useful when the initial state is expensive to compute, such as reading from local storage or processing large data.

```jsx
const [user, setUser] = useState(() => {
  return JSON.parse(localStorage.getItem("user"));
});
```

Without lazy initialization, the expensive calculation could run on every render before React ignores the initial value.

---

### 5. What happens internally when you call a `useState` setter, and how does React queue that update?

When you call a `useState` setter, React does not immediately change the current state variable. It creates an update, adds it to that hook's update queue, schedules a re-render, and processes the queued updates during the next render.

```jsx
setCount(count + 1);
console.log(count);
```

The `console.log` still shows the old value because the current render's state is fixed. The updated value becomes available in the next render.

---

### 6. Why does `console.log(state)` immediately after calling `setState` often show the old value?

`setState` does not update the current render's state variable immediately. It queues an update and React applies it during the next render, so the current function still sees the old state snapshot.

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
    console.log(count); // old value from current render
  }

  return <button onClick={handleClick}>{count}</button>;
}
```

---

### 7. What does it mean that state is a “snapshot” in React?

Each render receives a fixed version of state. Calling `setState` schedules a new render, but it does not change the state value inside the currently running render.

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);

    // This render still sees the same count value
    console.log(count);
  }

  return <button onClick={handleClick}>{count}</button>;
}
```

Think of state as frozen for one render.

---

### 8. What is the difference between `setCount(count + 1)` and `setCount(prev => prev + 1)`?

`setCount(count + 1)` uses the `count` value from the current render. `setCount(prev => prev + 1)` uses the latest queued state value, so it is safer when the next state depends on the previous state.

```jsx
setCount(count + 1); // uses current render snapshot
setCount((prev) => prev + 1); // uses latest queued state
```

Use functional updates when the new state depends on the old state.

---

### 9. What will happen if you call `setCount(count + 1)` three times in the same event handler?

All three calls use the same `count` value from the current render. So if `count` is `0`, each call becomes `setCount(1)`, and the final result is usually `1`, not `3`.

```jsx
function handleClick() {
  setCount(count + 1);
  setCount(count + 1);
  setCount(count + 1);
}
```

React queues three updates, but they all set the same value.

---

### 10. What will happen if you call `setCount(prev => prev + 1)` three times in the same event handler?

Each functional update receives the latest queued value. So if `count` starts at `0`, React applies the updates one by one and the final result becomes `3`.

```jsx
function handleClick() {
  setCount((prev) => prev + 1);
  setCount((prev) => prev + 1);
  setCount((prev) => prev + 1);
}
```

This is the correct approach for multiple updates based on previous state.

---

### 11. How does React batch multiple `useState` updates?

React batching means multiple state updates are grouped together and processed in a single render. This avoids unnecessary re-renders and improves performance.

```jsx
function handleClick() {
  setCount((prev) => prev + 1);
  setName("John");
  setIsActive(true);
}
```

React usually processes these updates together and performs one re-render instead of three separate renders.

---

### 12. How did automatic batching change in React 18?

Before React 18, batching mostly happened inside React event handlers. In React 18, automatic batching also works in promises, `setTimeout`, native event handlers, and async callbacks.

```jsx
setTimeout(() => {
  setCount((prev) => prev + 1);
  setFlag(true);
}, 1000);
```

In React 18, these updates are batched into one render. In older React versions, they could cause separate renders.

---

### 13. Why can mutating an object or array stored in state cause bugs?

React compares state by reference. If you mutate the existing object or array and pass the same reference back, React may think nothing changed and skip the update.

```jsx
const [user, setUser] = useState({ name: "John", age: 25 });

function updateAge() {
  user.age = 26;
  setUser(user); // same object reference, can cause bug
}
```

The object value changed internally, but the reference stayed the same.

---

### 14. Why should object and array state updates be immutable?

Immutable updates create a new reference, which helps React detect that state changed. It also makes state updates predictable, easier to debug, and safer for rendering.

```jsx
setUser((prev) => ({
  ...prev,
  age: 26,
}));
```

For arrays:

```jsx
setItems((prev) => [...prev, newItem]);
```

Instead of changing the old object or array, create a new one.

---

### 15. How does React decide whether a `useState` update should cause a re-render?

React compares the new state value with the previous state value using `Object.is`. If both values are the same, React can skip re-rendering that component.

```jsx
const [count, setCount] = useState(0);

setCount(0); // same value, React may skip re-render
```

For objects and arrays, React checks reference equality, not deep equality.

```jsx
setUser({ name: "John" }); // new reference, causes re-render
```

---

### 16. What role does `Object.is` play in `useState` updates?

React uses `Object.is` to compare the previous state and the next state. If both values are considered the same, React can skip re-rendering the component.

```jsx
const [count, setCount] = useState(0);

setCount(0); // Object.is(0, 0) is true, so React can skip re-render
```

For objects and arrays, `Object.is` compares references, not deep values.

```jsx
const user = { name: "John" };

setUser(user);
setUser(user); // same reference, so React can skip re-render
```

---

### 17. What is a stale closure problem with `useState`? (CHECK AGAIN)

A stale closure happens when a function remembers an old state value from a previous render. This commonly happens with timers, async callbacks, subscriptions, or event listeners.

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  function handleAlert() {
    setTimeout(() => {
      alert(count); // old count value from the render when this function was created
    }, 3000);
  }

  return <button onClick={handleAlert}>Show Count</button>;
}
```

The callback closes over the `count` value from that specific render.

---

### 18. Why do async callbacks like `setTimeout`, promises, or event listeners sometimes read old state?

Async callbacks do not automatically get the latest state. They capture the state value from the render in which the callback was created.

```jsx
function handleClick() {
  setTimeout(() => {
    console.log(count); // may log old count
  }, 1000);
}
```

React creates a new state value on the next render, but the old callback still points to the old render's variables snapshot.

---

### 19. How can functional updates help avoid stale state issues?

Functional updates do not depend on the state value captured by the current closure. React passes the latest queued state value into the updater function.

```jsx
setCount((prev) => prev + 1);
```

This is safer than:

```jsx
setCount(count + 1);
```

Use functional updates when the next state depends on the previous state, especially inside async callbacks or repeated updates.

---

### 20. When is `useState` state preserved, and when is it reset during conditional rendering?

React preserves state when the same component stays in the same position in the UI tree. State resets when React removes the component and later mounts it again.

```jsx
function App({ show }) {
  return <div>{show && <Counter />}</div>;
}
```

If `show` changes from `true` to `false`, `Counter` unmounts and its state is destroyed. When `show` becomes `true` again, `Counter` mounts with fresh state.

---

### 21. How can changing a component `key` reset `useState` state?

React uses `key` to decide whether a component is the same instance or a new one. When the `key` changes, React unmounts the old component and mounts a new component, so all `useState` values reset.

```jsx
function App({ userId }) {
  return <ProfileForm key={userId} userId={userId} />;
}
```

When `userId` changes, `ProfileForm` gets recreated with fresh state.

---

### 22. Why is storing derived state with `useState` often a bad idea?

Derived state is state that can be calculated from props or other state. Storing it separately can cause bugs because the derived value may become out of sync with the real source of truth.

```jsx
const [firstName, setFirstName] = useState("John");
const [lastName, setLastName] = useState("Doe");

// Avoid unnecessary derived state
const [fullName, setFullName] = useState(firstName + " " + lastName);
```

Better:

```jsx
const fullName = firstName + " " + lastName;
```

If a value can be calculated during render or re-render, usually do not store it in `useState`.

---

### 23. When would you choose `useReducer` over multiple `useState` calls?

Choose `useReducer` when state updates are complex, related, or depend on multiple fields changing together. Multiple `useState` calls are fine for simple independent values, but `useReducer` is better when update logic becomes difficult to manage.

```jsx
const [form, setForm] = useState({
  name: "",
  email: "",
  touched: false,
  error: null,
});
```

If many updates depend on form rules, validation, or transitions, `useReducer` can make the logic more predictable.

---

### 24. Where is `useState` state stored internally in React, and how is it connected to the component Fiber?

`useState` state is stored inside the component's Fiber, not inside the component function. React keeps hook data on the Fiber so it can remember state between renders.

```jsx
function Counter() {
  const [count, setCount] = useState(0);
}
```

Internally, React links this `useState` call to hook data stored on the current Fiber for `Counter`.

---

### 25. How does React use hook call order to match `useState` values across renders, and why does this make conditional hooks invalid?

React matches hooks by the order they are called during render. If a hook is called conditionally, the order can change between renders, so React may connect the wrong state value to the wrong hook.

```jsx
function Example({ enabled }) {
  if (enabled) {
    const [count, setCount] = useState(0); // invalid
  }

  const [name, setName] = useState("");
}
```

If `enabled` changes, the hook order changes. That is why hooks must always be called at the top level of the component.
