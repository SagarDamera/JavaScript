# Topic: useRef

- `useRef` stores a mutable value that persists across renders.

- Updating `ref.current` does not trigger a re-render.

- Common use: access DOM elements directly, like focusing an input.

- Useful for storing timer IDs, previous values, scroll positions, or external instance references.

- Use `useRef` when you need to remember a value without updating the UI.

- Do not use `useRef` for data that should be shown on the screen; use state for that.

- `useRef` returns the same object on every render.

- Common mistake: expecting UI to update after changing `ref.current`.

- Interview trap: `useRef` is not only for DOM access; it can store any mutable value.

- Real-world use: input focus, video controls, debounce timers, preventing duplicate API calls, and tracking previous state.

## One-Line Final Recall

`useRef` keeps mutable values across renders without causing re-renders.

---

### 1. What is the core mental model of `useRef` in React?

`useRef` is a persistent mutable box that React keeps across renders.  
Changing the value inside `.current` does not cause React to re-render the component.

```jsx
const countRef = useRef(0);

countRef.current += 1;
```

---

### 2. Why does React provide `useRef` when we already have `useState`?

`useState` is for values that affect the UI and should trigger re-render.  
`useRef` is for values that need to persist across renders but should not trigger re-render.

```jsx
const [count, setCount] = useState(0); // UI state
const timerIdRef = useRef(null); // non-UI value
```

---

### 3. What are the most common practical use cases for `useRef`?

Common use cases are accessing DOM elements, storing timer IDs, keeping previous values, and avoiding stale closures.  
It is useful when a value must survive renders but does not directly control UI output.

```jsx
const inputRef = useRef(null);

function focusInput() {
  inputRef.current.focus();
}

return <input ref={inputRef} />;
```

---

### 4. What value does `useRef` return, and why is the returned object stable across renders?

`useRef` returns an object with a `.current` property.  
React creates this ref object once and returns the same object on every render.

```jsx
const ref = useRef("initial");

console.log(ref.current);
```

---

### 5. Why does updating `ref.current` not trigger a component re-render?

React does not track mutations to `ref.current` like state updates.  
Since no state setter is called, React does not schedule a new render.

```jsx
const countRef = useRef(0);

function handleClick() {
  countRef.current += 1; // no re-render
}
```

---

### 6. How is `useRef` useful for accessing DOM elements directly?

`useRef` can hold a reference to a real DOM element after React renders it.  
This is useful for actions like focusing an input, scrolling, measuring size, or controlling media elements.

```jsx
const inputRef = useRef(null);

function handleFocus() {
  inputRef.current.focus();
}

return <input ref={inputRef} />;
```

---

### 7. When should you prefer `useRef` over `useState`?

Prefer `useRef` when the value needs to persist across renders but changing it should not update the UI.  
Examples include timer IDs, DOM nodes, previous values, or mutable flags.

```jsx
const timerRef = useRef(null);

timerRef.current = setTimeout(() => {
  console.log("Done");
}, 1000);
```

---

### 8. When should you avoid using `useRef` even though it seems convenient?

Avoid `useRef` when the value affects what the user sees on the screen.  
Using refs for UI state can make the UI outdated because ref updates do not trigger re-render.

```jsx
const countRef = useRef(0);

// Bad if count should appear in UI
countRef.current += 1;
```

---

### 9. How does `useRef` persist values across renders?

React stores the ref object internally for that component instance.  
On every render, React returns the same ref object, so `.current` keeps its latest value.

```jsx
const renderCountRef = useRef(0);

renderCountRef.current += 1;
```

---

### 10. What happens to a `useRef` value when a component unmounts and remounts?

When a component unmounts, its ref object is discarded with that component instance.  
On remount, React creates a new ref object initialized with the original initial value.

```jsx
const valueRef = useRef("initial");

// After unmount and remount,
// valueRef.current becomes "initial" again
```

---

### 11. Can you use `useRef` to store previous props or previous state values? How?

Yes, `useRef` can store the previous value because it persists across renders.  
Usually, you update the ref inside `useEffect` after the render completes.

```jsx
const prevCountRef = useRef();

useEffect(() => {
  prevCountRef.current = count;
}, [count]);
```

---

### 12. How can `useRef` help avoid stale closure problems?

A ref can always hold the latest value, even when an old callback or timer closes over old state.  
The callback reads from `ref.current` instead of using the stale value captured during render.

```jsx
const latestCountRef = useRef(count);

useEffect(() => {
  latestCountRef.current = count;
}, [count]);

useEffect(() => {
  const id = setInterval(() => {
    console.log(latestCountRef.current);
  }, 1000);

  return () => clearInterval(id);
}, []);
```

---

### 13. What is the difference between storing a value in a local variable and storing it in `useRef`?

A local variable is recreated on every render, so it does not persist reliably.  
A ref is stored by React and keeps the same `.current` value across renders.

```jsx
let localValue = 0; // resets every render

const refValue = useRef(0); // persists across renders
```

---

### 14. What is the difference between `useRef` and `createRef` in function components?

`useRef` returns the same ref object across renders.  
`createRef` creates a new ref object every render, so it is mainly used in class components.

```jsx
const stableRef = useRef(null); // good in function components

const newRefEachRender = createRef(); // not ideal in function components
```

---

### 15. How is `useRef` different from `useMemo`?

`useMemo` caches a computed value and may recompute it when dependencies change.  
`useRef` stores a mutable value that can be changed without dependencies or re-rendering.

```jsx
const memoValue = useMemo(() => expensiveCalculation(data), [data]);

const refValue = useRef(null);
refValue.current = someMutableValue;
```

---

### 16. How is `useRef` different from `useCallback`?

`useCallback` memoizes a function reference between renders based on dependencies.  
`useRef` stores a mutable value in `.current` and changing it does not trigger re-render.

```jsx
const handleClick = useCallback(() => {
  console.log(count);
}, [count]);

const latestCountRef = useRef(count);
latestCountRef.current = count;
```

---

### 17. Why is reading or writing `ref.current` during render usually discouraged?

Render should be pure and predictable.  
Reading or mutating refs during render can create inconsistent behavior because React may render, pause, discard, or re-render components.

```jsx
// Avoid changing refs during render
ref.current = props.value;
```

---

### 18. What are safe cases where reading or initializing a ref during render is acceptable?

It is usually safe when initialization is predictable and only happens once.  
For example, lazy-creating an object when `ref.current` is `null`.

```jsx
const playerRef = useRef(null);

if (playerRef.current === null) {
  playerRef.current = new VideoPlayer();
}
```

---

### 19. How can `useRef` be used to store timer IDs or interval IDs?

A timer ID can be stored in a ref so it survives renders without causing re-renders.  
This allows cleanup code or event handlers to access and clear the same timer.

```jsx
const intervalRef = useRef(null);

function startTimer() {
  intervalRef.current = setInterval(() => {
    console.log("Running");
  }, 1000);
}
```

---

### 20. Why is cleanup still required when using refs for timers, subscriptions, or external resources?

`useRef` only stores the resource reference; it does not automatically clean it up.  
Timers, subscriptions, and external resources must be cleaned manually to avoid memory leaks or duplicate side effects.

```jsx
const intervalRef = useRef(null);

useEffect(() => {
  intervalRef.current = setInterval(() => {
    console.log("Running");
  }, 1000);

  return () => clearInterval(intervalRef.current);
}, []);
```

---

### 21. How does `useRef` behave with dependency arrays in `useEffect`?

The ref object returned by `useRef` is stable, so adding the ref object itself to dependencies usually does not cause the effect to rerun.  
Changing `ref.current` does not notify React, so it does not trigger dependency-based effect execution.

```jsx
const valueRef = useRef(0);

useEffect(() => {
  console.log(valueRef.current);
}, []); // does not rerun when valueRef.current changes
```

---

### 22. Should `ref.current` usually be added to a `useEffect` dependency array? Why or why not?

Usually no, because React does not track mutations to `ref.current`.  
Updating `ref.current` will not trigger a render, so React cannot reliably compare it as a dependency.

```jsx
useEffect(() => {
  console.log(inputRef.current);
}, []); // usually better than [inputRef.current]
```

---

### 23. How can mutating `ref.current` lead to bugs in UI consistency?

If a ref value controls something visible in the UI, changing it will not re-render the component.  
This can make the displayed UI different from the actual value stored in the ref.

```jsx
const countRef = useRef(0);

function increment() {
  countRef.current += 1; // UI will not update
}

return <p>{countRef.current}</p>;
```

---

### 24. How does React internally store `useRef` on the component Fiber?

React stores hooks as a linked list on the component Fiber.  
For `useRef`, React stores the same ref object in the hook’s memoized state and returns it on later renders.

```jsx
const ref = useRef(null);

// Internally similar idea:
// hook.memoizedState = { current: null }
```

---

### 25. Why must `useRef` follow the Rules of Hooks and not be called conditionally?

React matches hooks by call order, not by variable name.  
If `useRef` is called conditionally, the hook order can change between renders and React may read the wrong hook state.

```jsx
// ❌ Wrong
if (isEnabled) {
  const valueRef = useRef(null);
}

// ✅ Correct
const valueRef = useRef(null);
```
