# Topic: useContext

- `useContext` lets a component read values from React Context directly.

- Use it to avoid prop drilling for shared data across many components.

- Common use cases: theme, logged-in user, language, auth state, and app settings.

- Create context with `createContext`, provide value using `Context.Provider`, and consume with `useContext`.

- Context updates cause all consuming components to re-render when the provider value changes.

- Avoid putting frequently changing values in a large global context if many components consume it.

- Memoize provider values when needed to avoid unnecessary re-renders.

- Common mistake: using context for all state instead of keeping state close to where it is used.

- Interview trap: Context is not a full state management library; it mainly solves data sharing.

- Real-world use: sharing auth info, theme mode, feature flags, and user preferences across the app.

## One-Line Final Recall

`useContext` lets components access shared values without passing props through every level.

---

### 1. What problem does `useContext` solve in React?

`useContext` helps share data across components without passing props through every level.  
It is useful for global or shared values like theme, logged-in user, language, or permissions.

```jsx
const theme = useContext(ThemeContext);
```

Tip: `useContext` solves prop drilling for shared data.

---

### 2. What does `useContext` return?

`useContext` returns the current value from the nearest matching Context Provider above the component.  
If no Provider exists, it returns the default value passed to `createContext`.

```jsx
const user = useContext(UserContext);
```

Tip: `useContext` reads the current context value.

---

### 3. How do you create and consume a context using `useContext`?

First create a context using `createContext`, then provide a value using `Provider`.  
Any child component can read that value using `useContext`.

```jsx
const ThemeContext = createContext("light");

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Profile />
    </ThemeContext.Provider>
  );
}

function Profile() {
  const theme = useContext(ThemeContext);
  return <div>{theme}</div>;
}
```

Tip: Create context once, provide it above, and consume it below.

---

### 4. What happens when a component calls `useContext` without a matching Provider above it?

React uses the default value given to `createContext`.  
This is useful for fallback values, but it can hide bugs if the Provider was expected.

```jsx
const AuthContext = createContext(null);

function Profile() {
  const user = useContext(AuthContext);
  return <div>{user?.name}</div>;
}
```

Tip: Without a Provider, `useContext` returns the context default value.

---

### 5. How does the `value` prop of a Context Provider affect consuming components?

When the Provider `value` changes, all components consuming that context can re-render.  
React compares the old and new value by reference using `Object.is`.

```jsx
<AuthContext.Provider value={user}>
  <Profile />
</AuthContext.Provider>
```

Tip: Changing Provider value updates context consumers.

---

### 6. When should you use `useContext` in a real React application?

Use `useContext` when many components need the same shared value.  
Common examples are theme, authenticated user, locale, feature flags, and app-level settings.

```jsx
const theme = useContext(ThemeContext);
const user = useContext(AuthContext);
```

Tip: Use context for shared app-level data, not every local state.

---

### 7. When should you avoid using `useContext`?

Avoid `useContext` for state that belongs only to one component or a small component group.  
Also avoid one large context with frequently changing values, because it can cause many consumers to re-render.

```jsx
// Better as local state, not global context
const [isOpen, setIsOpen] = useState(false);
```

Tip: Keep local state local unless many components truly need it.

---

### 8. Can `useContext` replace all state management libraries?

No, `useContext` only provides a way to pass values through the component tree.  
It does not provide advanced features like selectors, middleware, caching, devtools, or normalized global stores.

```jsx
const cart = useContext(CartContext);
```

Tip: Context is not always a full replacement for Redux, Zustand, or React Query.

---

### 9. How does `useContext` help avoid prop drilling?

Without context, props must be passed through every intermediate component.  
With `useContext`, deeply nested components can read the shared value directly from the nearest Provider.

```jsx
function Header() {
  const user = useContext(AuthContext);
  return <div>{user.name}</div>;
}
```

Tip: Context avoids passing props through components that do not use them.

---

### 10. Does `useContext` cause a component to re-render when the Provider value changes?

Yes, a component using `useContext` re-renders when the matching Provider value changes.  
This happens even if the component is deeply nested.

```jsx
<AuthContext.Provider value={user}>
  <Profile />
</AuthContext.Provider>
```

Tip: Context consumers update when their Provider value changes.

---

### 11. Why can passing a new object or function as Provider value cause unnecessary re-renders?

A new object or function gets a new reference on every render.  
React treats the Provider `value` as changed, so context consumers may re-render even when the actual data is the same.

```jsx
<AuthContext.Provider value={{ user, logout }}>
  <Profile />
</AuthContext.Provider>
```

Tip: Provider value reference matters.

---

### 12. How can `useMemo` help optimize a Context Provider value?

`useMemo` keeps the Provider value object stable between renders.  
The value is recreated only when its real dependencies change.

```jsx
const value = useMemo(() => {
  return { user, logout };
}, [user, logout]);

return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
```

Tip: Use `useMemo` when passing objects or arrays as context values.

---

### 13. How can `useCallback` help when passing functions through context?

`useCallback` keeps function references stable between renders.  
This helps prevent the Provider value from changing unnecessarily when the function logic has not changed.

```jsx
const logout = useCallback(() => {
  clearSession();
  setUser(null);
}, []);

const value = useMemo(() => {
  return { user, logout };
}, [user, logout]);
```

Tip: Use `useCallback` for stable functions inside context values.

---

### 14. What is the difference between `useContext` and passing props directly?

Props are explicit and best for parent-to-child data flow.  
`useContext` is better when many deeply nested components need the same shared value.

```jsx
// Props
<Profile user={user} />;

// Context
const user = useContext(AuthContext);
```

Tip: Use props for simple data flow and context for shared cross-tree data.

---

### 15. What is the difference between `useContext` and Redux or Zustand?

`useContext` only reads values from a Provider.  
Redux and Zustand provide external state management features like selectors, subscriptions, devtools, and better control over updates.

```jsx
const user = useContext(AuthContext);
```

Tip: Context shares data; state libraries manage complex global state.

---

### 16. How can splitting context improve performance?

Splitting context means separating unrelated values into different Providers.  
This helps components re-render only when the specific context they use changes.

```jsx
const UserContext = createContext(null);
const ThemeContext = createContext("light");

const user = useContext(UserContext);
const theme = useContext(ThemeContext);
```

Tip: Smaller focused contexts reduce unnecessary consumer updates.

---

### 17. Why is putting frequently changing state in one large context often a bad idea?

When one value inside a large context changes, all consumers of that context may re-render.  
This can hurt performance if the context contains many unrelated or frequently changing values.

```jsx
<AppContext.Provider value={{ user, theme, searchText, mousePosition }}>
  {children}
</AppContext.Provider>
```

Tip: Avoid putting fast-changing and unrelated state in one big context.

---

### 18. Can a component consume multiple contexts using `useContext`?

Yes, a component can call `useContext` multiple times.  
Each call reads from a different Context Provider.

```jsx
const user = useContext(UserContext);
const theme = useContext(ThemeContext);

return <div className={theme}>{user.name}</div>;
```

Tip: Multiple `useContext` calls are normal and valid.

---

### 19. How does nested Provider behavior work in React Context?

If Providers are nested, React uses the nearest matching Provider above the consuming component.  
Inner Providers override the value from outer Providers for their subtree.

```jsx
<ThemeContext.Provider value="light">
  <ThemeContext.Provider value="dark">
    <Profile />
  </ThemeContext.Provider>
</ThemeContext.Provider>
```

Tip: Context value is resolved from the closest matching Provider.

---

### 20. How does `useContext` behave during server-side rendering?

During server-side rendering, `useContext` reads the Provider value available in the rendered tree.  
It does not use browser APIs, so it works normally as long as the Provider is included.

```jsx
function Header() {
  const theme = useContext(ThemeContext);
  return <div>{theme}</div>;
}
```

Tip: `useContext` works in SSR because it is part of Reactâ€™s render process.

---

### 21. Can `useContext` be used outside a React component or custom hook?

No, `useContext` can only be called inside a React function component or a custom hook.  
Calling it outside React render breaks the Rules of Hooks.

```jsx
// Wrong
const user = useContext(UserContext);

// Correct
function Profile() {
  const user = useContext(UserContext);
  return <div>{user.name}</div>;
}
```

Tip: Hooks must be called only inside React components or custom hooks.

---

### 22. Why does context update bypass `React.memo` in consuming components?

`React.memo` only compares props passed from the parent.  
If a component consumes context, React still re-renders it when the context value changes, even if props are unchanged.

```jsx
const Profile = React.memo(function Profile() {
  const user = useContext(UserContext);
  return <div>{user.name}</div>;
});
```

Tip: `React.memo` cannot block updates from context changes.

---

### 23. How does React determine which components should update when context value changes?

React tracks which components read a specific context during render.  
When the Provider value changes, React schedules updates for the consumers of that context.

```jsx
function Header() {
  const theme = useContext(ThemeContext);
  return <div className={theme}>Header</div>;
}
```

Tip: Components that read the changed context are the ones React updates.

---

### 24. How does React internally associate a `useContext` call with a component Fiber across renders?

When a component calls `useContext`, React records a context dependency on that componentâ€™s Fiber.  
Later, when the Provider value changes, React uses that dependency to know which Fibers consumed the context.

```jsx
function Profile() {
  const user = useContext(UserContext);
  return <div>{user.name}</div>;
}
```

Tip: React tracks context dependencies on the component Fiber during render.

---

### 25. Why canâ€™t `useContext` be called conditionally inside a component?

Hooks should be called in the same order on every render.  
Calling `useContext` conditionally can break hook ordering and make the component behavior unpredictable.

```jsx
// Wrong
if (isLoggedIn) {
  const user = useContext(UserContext);
}

// Correct
const user = useContext(UserContext);

if (!isLoggedIn) {
  return null;
}
```

Tip: Always call `useContext` at the top level of the component.
