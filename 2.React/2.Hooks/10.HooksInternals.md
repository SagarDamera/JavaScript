# Topic: Hooks Internals

- React stores hook state internally in the order hooks are called during render.

- Hooks work by call order, not by hook variable names.

- Each component instance has its own hook state list.

- On every render, React walks through the hook list in the same order.

- This is why hooks cannot be called inside conditions, loops, or nested functions.

- `useState` keeps state between renders and returns the current value from React’s internal storage.

- `useEffect` stores effect logic and runs it after render when dependencies change.

- Dependency arrays help React decide whether to reuse previous hook result or run logic again.

- Common mistake: changing hook order between renders, which breaks React’s internal tracking.

- Interview trap: hooks are not magic variables; React connects them to a component using render order.

## One-Line Final Recall

Hooks work because React stores hook data by call order and reuses that order on every render.

---

### 1. How does React internally store hook state for a function component?

React stores hook state on the component’s Fiber object.  
Each hook call creates a hook record that contains data like state, queue, memoized value, or dependencies.

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  const value = useMemo(() => count * 2, [count]);
}
```

Tip: Hook state lives on the Fiber, not inside the function itself.

---

### 2. What is the relationship between hooks and the component Fiber?

A Fiber represents a React component instance in React’s internal tree.  
Hooks are stored on that Fiber so React can preserve state and hook data between renders.

```jsx
function Profile() {
  const [user, setUser] = useState(null);
  const theme = useContext(ThemeContext);
}
```

Tip: Fiber is where React connects a component render to its hook state.

---

### 3. Why does React rely on hook call order instead of hook names?

Variable names do not exist reliably at runtime after compilation and minification.  
React uses the order of hook calls because it is predictable when hooks are called the same way every render.

```jsx
const [count, setCount] = useState(0);
const [name, setName] = useState("");
```

Tip: React matches hooks by position, not by variable name.

---

### 4. What is the hook linked list inside a Fiber?

React stores hooks as a linked list on the component Fiber.  
Each hook node points to the next hook, allowing React to walk through them in order during render.

```jsx
function Example() {
  const [count, setCount] = useState(0); // hook 1
  const ref = useRef(null); // hook 2
  const value = useMemo(() => count * 2, [count]); // hook 3
}
```

Tip: The hook linked list is why hook order must stay stable.

---

### 5. What happens during the mount phase of a hook?

During mount, React creates a new hook node and stores its initial data.  
For example, `useState` stores initial state, and `useEffect` stores the effect and dependencies.

```jsx
function Counter() {
  const [count, setCount] = useState(0);
}
```

Tip: Mount phase creates hook records for the first render.

---

### 6. What happens during the update phase of a hook?

During update, React reuses the existing hook node from the previous render.  
It reads previous hook data, applies pending updates if needed, and returns the latest state or memoized value.

```jsx
function Counter() {
  const [count, setCount] = useState(0);
}
```

Tip: Update phase reuses existing hook records instead of creating them from scratch.

---

### 7. How does React know whether a hook is mounting or updating?

React checks whether the current Fiber already has previous hook state.  
If there is no previous hook list, it is mounting; if hook data already exists, it is updating.

```jsx
function Profile() {
  const [name, setName] = useState("");
}
```

Tip: First render creates hooks; later renders reuse and update them.

---

### 8. How does React match a `useState` call to its previous state across renders?

React matches `useState` by the order in which hooks are called.  
The first `useState` gets the first previous hook state, the second gets the second, and so on.

```jsx
const [count, setCount] = useState(0); // first hook
const [name, setName] = useState(""); // second hook
```

Tip: React matches hook state by position, not by variable name.

---

### 9. How does React store pending state updates for hooks?

React stores pending updates in an update queue attached to the hook.  
When a setter or dispatch is called, React adds an update to that queue and processes it during render.

```jsx
setCount((prevCount) => prevCount + 1);
setCount((prevCount) => prevCount + 1);
```

Tip: Hook updates are queued first, then processed during rendering.

---

### 10. What is an update queue in React hooks?

An update queue is React’s internal list of pending updates for a hook.  
For `useState` and `useReducer`, it stores updates until React processes them to calculate the next state.

```jsx
const [count, setCount] = useState(0);

setCount(1);
setCount(2);
```

Tip: The update queue lets React collect and process multiple state updates predictably.

---

### 11. How does calling a state setter schedule work in React?

When you call a state setter, React creates an update and adds it to the hook’s update queue.  
Then React schedules that component’s Fiber to render again with the new state.

```jsx
const [count, setCount] = useState(0);

function handleClick() {
  setCount(count + 1);
}
```

Tip: Calling a setter schedules a render; it does not instantly change state in the current render.

---

### 12. How does React decide the priority of a hook state update?

React assigns priority based on where the update came from.  
User interactions like clicks usually get higher priority than background updates or transition updates.

```jsx
setInputValue(text); // urgent update

startTransition(() => {
  setSearchResults(results); // lower-priority update
});
```

Tip: React prioritizes updates so the UI can stay responsive.

---

### 13. What are lanes in React, and how are they related to hook updates?

Lanes are React’s internal priority system for scheduling updates.  
When a hook update is created, React assigns it to a lane so it knows when and how urgently to process it.

```jsx
setCount((c) => c + 1);
```

Tip: Lanes help React organize urgent, normal, and deferred updates.

---

### 14. How does React batch multiple hook state updates?

React groups multiple state updates together before rendering.  
Instead of rendering after every setter call, React processes them in one render when possible.

```jsx
function handleClick() {
  setCount((c) => c + 1);
  setName("Meghana");
  setOpen(true);
}
```

Tip: Batching reduces unnecessary renders and improves performance.

---

### 15. What happens internally when multiple `setState` calls are made in the same event?

React adds all updates to the hook update queue and processes them together during the next render.  
Updater functions are applied in order, so each one receives the latest computed state.

```jsx
function handleClick() {
  setCount((c) => c + 1);
  setCount((c) => c + 1);
  setCount((c) => c + 1);
}

// count increases by 3
```

Tip: Use updater functions when the next state depends on the previous state.

---

### 16. How does React compare dependency arrays for `useEffect`, `useMemo`, and `useCallback`?

React compares each dependency with the previous render’s dependency using `Object.is`.  
If any dependency is different, the effect runs again or the memoized value/function is recreated.

```jsx
useEffect(() => {
  console.log(userId);
}, [userId]);

const value = useMemo(() => calculate(userId), [userId]);

const handler = useCallback(() => save(userId), [userId]);
```

Tip: Dependency comparison is shallow and reference-based.

---

### 17. How are passive effects from `useEffect` stored and executed internally?

React stores `useEffect` callbacks and cleanup functions in the Fiber’s effect list.  
After the DOM is committed and painted, React runs passive effects asynchronously.

```jsx
useEffect(() => {
  const id = setInterval(() => {
    console.log("tick");
  }, 1000);

  return () => clearInterval(id);
}, []);
```

Tip: `useEffect` runs after commit, usually after the browser paints.

---

### 18. How is `useLayoutEffect` scheduled differently from `useEffect` internally?

`useLayoutEffect` runs synchronously after DOM mutations but before the browser paints.  
`useEffect` runs later as a passive effect after paint.

```jsx
useLayoutEffect(() => {
  const height = ref.current.offsetHeight;
  setHeight(height);
}, []);

useEffect(() => {
  console.log("Runs after paint");
}, []);
```

Tip: Use `useLayoutEffect` only when you must read or update layout before paint.

---

### 19. Why can changing hook order corrupt React’s internal hook state matching?

React matches hooks by their position in the hook linked list.  
If hook order changes, React may connect a hook call to the wrong previous hook state.

```jsx
// Wrong
if (enabled) {
  useEffect(() => {}, []);
}

const [count, setCount] = useState(0);
```

Tip: Stable hook order keeps React’s internal hook mapping correct.

---

### 20. What happens internally when a component renders fewer hooks than the previous render?

React reaches the end of the current hook calls before finishing the previous hook list.  
This means some previous hooks were skipped, so React throws a hook order error.

```jsx
function Profile({ user }) {
  const [count, setCount] = useState(0);

  if (!user) {
    return null;
  }

  useEffect(() => {
    console.log("User exists");
  }, []);
}
```

Tip: Rendering fewer hooks usually happens because of early returns or conditional hooks.

---

### 21. What happens internally when a component renders more hooks than the previous render?

React finds a new hook call where no previous hook record exists at that position.  
This means the hook order changed, so React throws an error like “Rendered more hooks than during the previous render.”

```jsx
function Profile({ show }) {
  const [count, setCount] = useState(0);

  if (show) {
    useEffect(() => {
      console.log("extra hook");
    }, []);
  }

  return <div>{count}</div>;
}
```

Tip: Rendering more hooks usually happens because a hook was added inside a condition.

---

### 22. How does React preserve hook state between renders but reset it on unmount?

React stores hook state on the component Fiber and reuses it while the same component instance stays mounted.  
When the component unmounts, that Fiber is removed, so its hook state is discarded.

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  return <button>{count}</button>;
}
```

Tip: Hook state survives re-renders, but not unmount/remount.

---

### 23. How does React Strict Mode affect hook initialization and effects in development?

In development, Strict Mode may intentionally render components twice and re-run effects to detect unsafe side effects.  
This helps catch impure render logic, missing cleanup, and code that depends on effects running only once.

```jsx
useEffect(() => {
  console.log("effect setup");

  return () => {
    console.log("effect cleanup");
  };
}, []);
```

Tip: Strict Mode double behavior happens only in development.

---

### 24. How do custom hooks work internally if they are just JavaScript functions?

Custom hooks are normal functions, but when called during render, their internal hooks become part of the caller component’s hook list.  
React does not create a separate Fiber for the custom hook itself.

```jsx
function useCounter() {
  const [count, setCount] = useState(0);
  return { count, setCount };
}

function App() {
  const counter = useCounter();
}
```

Tip: Custom hook state belongs to the component that calls the custom hook.

---

### 25. Why do Hooks Rules exist from a React internals perspective?

Hooks Rules exist because React tracks hooks by call order on the Fiber.  
Stable hook order lets React correctly match each hook call with its previous state, queue, effect, or memoized value.

```jsx
const [count, setCount] = useState(0);
const value = useMemo(() => count * 2, [count]);
useEffect(() => {}, [value]);
```

Tip: Hooks Rules protect React’s internal hook matching system.
