# Topic: Server State vs Client State

- Client state is data controlled by the frontend, like modal open state, selected tab, form input, or theme.

- Server state is data fetched from backend APIs, like users, products, orders, or dashboard data.

- Client state is usually managed with `useState`, `useReducer`, Context, Redux, or Zustand.

- Server state needs fetching, caching, loading, error handling, refetching, and synchronization.

- Tools like React Query, SWR, or RTK Query are better for server state than manually using `useEffect`.

- Do not store server data in global client state unless there is a strong reason.

- Server state can become stale, so cache invalidation and refetching are important.

- Client state is usually immediately available, while server state is asynchronous.

- Common mistake: treating API data like normal local state and manually managing too much logic.

- Interview trap: Redux manages client/global state well, but server state often needs dedicated data-fetching tools.

## One-Line Final Recall

Client state belongs to the frontend UI, while server state comes from APIs and needs caching, syncing, and refetching.
