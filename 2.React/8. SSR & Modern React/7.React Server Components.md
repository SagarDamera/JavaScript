# Topic: React Server Components

- React Server Components render on the server and send lightweight UI output to the client.

- They reduce client-side JavaScript because server components are not bundled for the browser.

- Best use case: fetching data, reading databases, accessing backend services, and rendering static or data-heavy UI.

- Server Components cannot use browser-only APIs like `window`, `document`, or `localStorage`.

- Server Components cannot use client hooks like `useState`, `useEffect`, or event handlers.

- Use `"use client"` when a component needs interactivity, state, effects, or browser APIs.

- Server Components can pass props to Client Components, but props must be serializable.

- Keep heavy data logic on the server and interactive UI in Client Components.

- Common mistake: marking too many components as `"use client"`, which increases bundle size.

- Interview trap: Server Components are not the same as SSR; they are a component model for splitting server and client work.

## One-Line Final Recall

React Server Components keep data-heavy rendering on the server while sending only necessary interactive code to the client.
