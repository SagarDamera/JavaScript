# Topic: Commit Phase

- Commit phase is when React applies the finished changes to the real DOM.

- It happens after the render phase has calculated what needs to change.

- Commit phase is not interruptible; once started, React completes it.

- DOM mutations, ref updates, and layout effects happen during this phase.

- `useLayoutEffect` runs synchronously after DOM updates but before browser paint.

- `useEffect` usually runs after the browser paints.

- Side effects should happen in effects, not during render.

- Commit phase is where users actually see UI changes on the screen.

- Common mistake: thinking render phase updates the DOM directly.

- Interview trap: render phase can be paused or restarted, but commit phase cannot.

## One-Line Final Recall

Commit phase is React’s final step where calculated UI changes are applied to the real DOM.

## 1. What is the commit phase in React?

The commit phase is the step where React applies the prepared changes to the actual DOM and makes the final UI update visible to the user.

### Explanation

During the render phase, React calculates what the UI should look like. During the commit phase, React takes that completed work and updates the real browser DOM.

This is also when React handles things like refs, layout effects, and passive effects scheduling. Unlike the render phase, the commit phase cannot be paused once it starts because React is making real changes to the screen.

### Real-World Usage

This matters when debugging UI updates. A component may render many times, but the user only sees the result after React commits changes.

## 2. How is the commit phase different from the render phase?

The render phase calculates the next UI, while the commit phase applies that calculated result to the DOM.

### Explanation

The render phase is pure calculation. React calls components, reconciles elements, builds the work-in-progress Fiber tree, and determines what changed.

The commit phase is where React performs side-effectful work. It updates DOM nodes, removes old nodes, inserts new nodes, applies refs, and runs effects at the right time.

In simple terms: render decides what should happen; commit makes it happen.

### Real-World Usage

This distinction is important when analyzing performance. Render cost is CPU calculation cost, while commit cost is actual DOM mutation and browser update cost.

## 3. What does React actually update during the commit phase?

During the commit phase, React updates the host environment, which is usually the browser DOM in React DOM applications.

### Explanation

React may perform operations such as:

- Insert new DOM nodes
- Update existing DOM attributes, properties, text, and event handlers
- Remove DOM nodes
- Attach or detach refs
- Run layout effects
- Schedule passive effects like `useEffect`

React does not call component functions during the commit phase. Component functions are called during render. Commit is about applying the already-calculated changes.

### Real-World Usage

This is useful when debugging visual issues, focus behavior, layout measurements, animations, and refs because those are connected to the committed DOM.

## 4. Why does React separate rendering work from committing work?

React separates rendering from committing so it can prepare UI updates safely before changing the actual screen.

### Explanation

Rendering can be paused, restarted, or abandoned because it is only calculation work. React can use this flexibility to prioritize urgent updates and avoid committing outdated UI.

Committing is different. Once React starts applying DOM changes, it must finish consistently. Separating the two phases lets React optimize rendering while keeping the visible UI stable.

This separation is also what enables modern React features like Concurrent Rendering.

### Real-World Usage

In large applications, React can start rendering a heavy update, pause it for a more urgent user interaction, and only commit once it has a final consistent result.

## 5. Why does the UI become visible only after React commits changes?

The UI becomes visible only after commit because the render phase only creates an internal description of the next UI. The browser screen changes only when React updates the real DOM during commit.

### Explanation

JSX and React elements are not the actual DOM. During render, React calculates a new tree and determines the required changes. Until those changes are committed, the browser has nothing new to display.

Once React commits, the DOM is updated. Then the browser can paint the updated UI to the screen.

This explains why logs inside component bodies can appear before the user sees anything change. The component rendered, but the visible UI appears only after React commits the final DOM updates.

## 7. Why is the commit phase not interruptible?

The commit phase is not interruptible because React is making real changes to the DOM, and stopping halfway could leave the UI in an inconsistent state.

### Explanation

Render work can be interrupted because it is only internal calculation. Commit work is different because React is mutating the actual screen.

For example, if React inserted some DOM nodes but did not remove outdated ones, or updated refs without finishing DOM changes, the user could see a broken or inconsistent UI.

So once React starts committing a completed render result, it finishes that commit synchronously.

### Real-World Usage

## 8. When do refs get attached or updated during React’s lifecycle?

Refs are attached or updated during the commit phase, after React has created or updated the DOM nodes.

### Explanation

During render, React only determines what refs should point to. The actual ref assignment happens during commit because the real DOM node must exist first.

For object refs, React updates `ref.current`. For callback refs, React calls the callback with the DOM node. When a node is removed, React clears the ref during commit as well.

This means refs are reliable after the DOM has been committed.

### Example

```jsx
function SearchBox() {
  const inputRef = React.useRef(null);

  React.useLayoutEffect(() => {
    inputRef.current.focus();
  }, []);

  return <input ref={inputRef} />;
}
```

The ref is available inside `useLayoutEffect` because React has already committed the DOM node.

### Real-World Usage

Refs are commonly used for focus management, measuring DOM nodes, integrating with third-party libraries, scrolling, and imperative browser APIs.

## 9. When do layout effects run relative to DOM mutations?

Layout effects run after React has applied DOM mutations but before the browser paints the updated screen.

### Explanation

`useLayoutEffect` runs during the commit phase after the DOM is updated. This makes it suitable for reading layout, measuring elements, or synchronously adjusting the DOM before the user sees the final frame.

Because layout effects block painting, they should be used carefully. Heavy work inside `useLayoutEffect` can delay the browser from showing the updated UI.

For non-layout work like API calls, subscriptions, or logging, `useEffect` is usually preferred because it runs after paint.

### Example

```jsx
function Tooltip({ text }) {
  const ref = React.useRef(null);
  const [height, setHeight] = React.useState(0);

  React.useLayoutEffect(() => {
    setHeight(ref.current.getBoundingClientRect().height);
  }, []);

  return (
    <div ref={ref}>
      {text} - Height: {height}
    </div>
  );
}
```

`useLayoutEffect` can measure the DOM after mutation but before the browser paints.

### Real-World Usage

This is useful for tooltips, popovers, scroll positioning, animations, and layout correction where the user should not see an intermediate incorrect position.

## 13. What is the difference between mutation effects, layout effects, and passive effects?

Mutation effects update the DOM, layout effects run synchronously after DOM updates but before paint, and passive effects run asynchronously after paint.

### Explanation

Mutation effects are React’s actual DOM operations, such as inserting, updating, or deleting nodes.

Layout effects are created with `useLayoutEffect`. They run after DOM mutations, before the browser paints. They are useful for reading layout and making synchronous visual adjustments.

Passive effects are created with `useEffect`. They usually run after the browser has painted. They are better for non-visual side effects like data fetching, subscriptions, logging, and syncing with external systems.

The main difference is timing and whether they block paint.

### Example

```jsx
function Example() {
  React.useLayoutEffect(() => {
    // Runs after DOM mutation, before paint
  }, []);

  React.useEffect(() => {
    // Runs after paint
  }, []);

  return <div>Hello</div>;
}
```

### Real-World Usage

Use `useLayoutEffect` only when the work must happen before the user sees the screen. Use `useEffect` for most side effects to avoid blocking paint.

## 14. Why can expensive work inside useLayoutEffect block the browser from painting?

Expensive work inside `useLayoutEffect` blocks painting because `useLayoutEffect` runs synchronously after DOM mutations but before the browser paints the updated screen.

### Explanation

React commits the DOM changes, then runs layout effects before the browser gets a chance to display the updated frame. If a layout effect performs heavy calculations, synchronous state updates, large DOM reads/writes, or expensive loops, the browser must wait.

This can cause delayed visual updates, janky interactions, and poor perceived performance.

For this reason, `useLayoutEffect` should be small and focused. If the work does not need to happen before paint, it should usually move to `useEffect`.

### Real-World Usage

This matters in tooltips, modals, virtualized lists, drag-and-drop, and animation code. Layout effects should only do urgent layout-related work, not general business logic.

## 15. Scenario: A component reads layout measurements after render. Why might useLayoutEffect be more appropriate than useEffect?

`useLayoutEffect` may be more appropriate because it runs after the DOM is updated but before the browser paints, allowing the component to measure layout and adjust synchronously before the user sees the result.

### Explanation

If a component needs to measure size, position, scroll location, or bounding boxes, the DOM must already exist. That means the measurement cannot happen during render.

Using `useEffect` may be too late for layout-sensitive work because the browser may already paint once. This can cause flicker, incorrect positioning, or visible layout jumps.

Using `useLayoutEffect` allows React to:

- commit the DOM
- run the layout measurement
- apply any required synchronous correction
- then let the browser paint the final result

### Example

```jsx
function Popover({ anchorRef }) {
  const popoverRef = React.useRef(null);
  const [top, setTop] = React.useState(0);

  React.useLayoutEffect(() => {
    const anchorRect = anchorRef.current.getBoundingClientRect();
    setTop(anchorRect.bottom + 8);
  }, [anchorRef]);

  return (
    <div ref={popoverRef} style={{ position: "absolute", top }}>
      Popover content
    </div>
  );
}
```

The popover position is calculated after the DOM exists but before the browser paints the final screen.

### Real-World Usage

This is common for popovers, dropdowns, tooltips, sticky headers, scroll restoration, and layout-aware animations where visible flicker would hurt user experience.

Heavy commit phase work can cause jank because commit runs synchronously on the main thread and cannot be interrupted once it starts.

## 23. How should teams decide between useEffect, useLayoutEffect, and callback refs for DOM-related logic?

Teams should choose based on timing: use callback refs for getting access to a DOM node, `useLayoutEffect` for synchronous layout reads or corrections before paint, and `useEffect` for non-blocking side effects after paint.

### Explanation

`useEffect` should be the default for most side effects because it does not block the browser from painting. It is appropriate for subscriptions, logging, data fetching, non-visual DOM integration, and syncing with external systems.

`useLayoutEffect` should be used when the work must happen after DOM mutation but before paint. It is appropriate for measuring layout, correcting position, restoring scroll, or preventing flicker.

Callback refs are useful when logic needs to run exactly when a DOM node is attached or detached. They are often useful for integrating with imperative APIs, managing dynamic lists of nodes, or avoiding an extra effect just to capture a node reference.

### Example

```jsx
function AutoFocusInput() {
  const inputRef = React.useCallback((node) => {
    if (node) {
      node.focus();
    }
  }, []);

  return <input ref={inputRef} />;
}
```

This uses a callback ref to act when the DOM node is attached.

### Real-World Usage

A strong team guideline is: use `useEffect` by default, use `useLayoutEffect` only when visual correctness before paint requires it, and use callback refs when the logic is directly tied to node attachment.

## 30. Scenario: A concurrent render is interrupted several times, but only one final UI update appears. What exactly gets committed?

Only the final completed render result gets committed.

So even if React evaluated components multiple times internally, the user sees only the final committed result.

### Real-World Usage

This explains why logs inside render can appear multiple times while the screen updates once. It also reinforces why side effects should not run during render: interrupted renders may never be committed.
