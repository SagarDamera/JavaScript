# Topic: Diffing Algorithm

- Diffing is React’s process of comparing the previous UI tree with the new UI tree.

- React uses diffing to find the minimum changes needed for the real DOM.

- React assumes elements of different types create different trees.

- If the element type is the same, React updates changed props and checks children.

- If the element type changes, React removes the old component tree and mounts a new one.

- Keys are critical when diffing lists because they help React match items correctly.

- Stable keys preserve component state during list updates.

- Bad keys like array indexes can cause incorrect UI behavior when items reorder.

- Common mistake: thinking React deeply compares everything perfectly; it uses efficient assumptions.

- Interview trap: diffing finds what changed, while reconciliation applies React’s update strategy.

## One-Line Final Recall

React’s diffing algorithm compares old and new UI trees using efficient rules to update only what changed.

### 1. What is the diffing algorithm in React?

The diffing algorithm in React is the process React uses to compare the previous React element tree with the new React element tree and determine what changed.

It helps React figure out which parts of the UI need to be updated, inserted, removed, or reused.

#### Explanation

When state or props change, React calls components again and creates a new React element tree.

React then compares that new tree with the previous tree. This comparison is called diffing.

The result of diffing is not an immediate DOM update. It is a set of changes React prepares during reconciliation and later applies during the commit phase.

#### Real-World Usage

Diffing is used in every React application whenever UI changes due to user input, API responses, route changes, filters, sorting, or real-time updates.

### 2. Why does React need a diffing algorithm?

React needs a diffing algorithm so it can update the UI efficiently when state or props change.

Without diffing, React would have to either replace the entire DOM or require developers to manually update specific DOM nodes.

#### Explanation

React follows a declarative model. Developers describe what the UI should look like for a given state.

When state changes, React needs to determine the difference between the previous UI description and the new UI description.

The diffing algorithm helps React answer:

- What stayed the same?
- What changed?
- What should be updated?
- What should be removed?
- What should be inserted?
- What state can be preserved?

This allows React to keep the UI correct without unnecessary DOM work.

#### Real-World Usage

This is useful in large UIs like dashboards, forms, search pages, data grids, and admin tools where only small parts of the page may change frequently.

### 3. How is React diffing related to reconciliation?

Diffing is a core part of reconciliation.

Reconciliation is the broader process where React compares the old and new UI trees, decides what work is needed, preserves or resets component state, and prepares updates. Diffing is the comparison step inside that process.

#### Explanation

At a high level:

- Components render and return React elements.
- React diffs the new element tree against the previous one.
- Reconciliation decides whether to reuse, update, insert, delete, or move Fibers.
- The commit phase applies the final changes to the real DOM.

So diffing helps React understand what changed, and reconciliation uses that information to decide how React should update the UI.

#### Real-World Usage

Understanding this relationship helps explain why a component can re-render without necessarily causing a DOM update, and why keys and component identity affect state preservation.

### 4. What does React compare during the diffing process?

React compares React elements, not the actual DOM directly.

During diffing, React mainly compares element type, component type, props, keys, and children to decide whether existing UI can be reused or whether parts of the tree need to change.

#### Explanation

React looks at signals such as:

- Is the element type the same, like `div` to `div`?
- Is the component type the same, like `UserCard` to `UserCard`?
- Did props change?
- Did text content change?
- Are children added, removed, moved, or updated?
- Do list items have stable keys?

If type and key match, React usually reuses the existing Fiber and updates props or children. If type or key changes, React may replace that part of the tree.

#### Example

```jsx
function UserCard({ user }) {
  return (
    <section>
      <h2>{user.name}</h2>
      <p>{user.role}</p>
    </section>
  );
}
```

#### Real-World Usage

This matters when building reusable components and dynamic lists. Stable element types and stable keys help React avoid unnecessary remounts and preserve the correct state.

### 5. Why does React try to minimize actual DOM updates?

React tries to minimize actual DOM updates because DOM operations are expensive compared to creating and comparing lightweight React elements in memory.

Updating the DOM can trigger layout recalculation, painting, focus changes, scroll issues, and browser rendering work.

#### Explanation

React’s goal is not just to re-run components. Its goal is to apply the smallest practical set of real UI changes needed to keep the screen correct.

If React replaced large parts of the DOM unnecessarily, users could experience slow interactions, lost input focus, broken animations, scroll jumps, or poor rendering performance.

Diffing helps React avoid unnecessary DOM mutations.

#### Real-World Usage

This is important in forms, tables, dashboards, modals, real-time feeds, and complex layouts where frequent DOM replacement would make the app feel unstable or slow.

### 6. How does React diff two React element trees?

React diffs two React element trees by comparing the previous tree with the new tree node by node.

It mainly looks at element type, component type, props, keys, and children to decide whether to reuse existing work, update it, or replace part of the tree.

#### Explanation

When a component re-renders, React creates a new React element tree. React then compares that new tree with the previous tree.

At a high level:

- Same type usually means React can reuse the existing Fiber.
- Different type usually means React replaces that subtree.
- Same key helps React match children across renders.
- Different key means React treats it as a different identity.
- Changed props cause React to update the existing element or component.

React does not perform a perfect tree-edit comparison. It uses fast heuristics that work well for UI rendering.

#### Real-World Usage

This matters when building dynamic layouts, reusable components, forms, tables, and lists. Stable element types and stable keys help React make efficient and correct update decisions.

### 7. Why does React assume that elements of different types produce different trees?

React assumes elements of different types produce different trees because different types usually represent different UI structure, behavior, state, refs, effects, and DOM requirements.

This assumption lets React avoid expensive comparisons and safely replace the old subtree with a new one.

#### Explanation

For example, a `div` and a `section` may look similar visually, but they are different DOM element types. Similarly, `<UserCard />` and `<AdminCard />` may have completely different internal structure and state.

So React uses a simple rule:

- Same type: try to reuse and update.
- Different type: discard the old subtree and create a new one.

This keeps the diffing algorithm fast and predictable.

#### Real-World Usage

This is important when conditionally rendering different components. If the component type changes, React may reset state because it treats the new component as a different identity.

### 8. What happens when two elements have the same type but different props?

When two elements have the same type but different props, React reuses the existing element or component and updates only the changed props.

For DOM elements, React updates the changed DOM attributes or properties. For custom components, React re-renders the component with the new props.

#### Explanation

If React sees the same type, it assumes the existing UI can usually be reused.

For example, if a `button` changes from `disabled={false}` to `disabled={true}`, React does not replace the button. It updates the `disabled` property.

For a component like `<UserCard user={user1} />` changing to `<UserCard user={user2} />`, React preserves the component identity and passes the new props into the next render.

#### Example

```jsx
function SaveButton({ isSaving }) {
  return <button disabled={isSaving}>{isSaving ? "Saving..." : "Save"}</button>;
}
```

#### Real-World Usage

This behavior helps preserve local state, focus, refs, and DOM identity when only data or visual attributes change.

### 9. How does React diff children inside the same parent?

React diffs children inside the same parent by comparing the previous children with the new children.

If children have keys, React uses the keys to match old and new children by identity. If keys are missing, React mostly compares children by position.

#### Explanation

For simple children in the same order, React can compare them position by position.

For dynamic lists, keys are important because they tell React which child is the same item across renders.

With stable keys, React can detect:

- An existing child was updated.
- A child was inserted.
- A child was removed.
- A child moved to a new position.

Without stable keys, React may reuse the wrong child when items are sorted, filtered, or inserted.

#### Example

```jsx
        <li key={todo.id}>{todo.title}</li>

  );
}
```

#### Real-World Usage

This is critical in sortable tables, drag-and-drop lists, dynamic forms, shopping carts, search results, and notification feeds.

### 10. Scenario: A component changes from `<div>` to `<section>` between renders. What happens during diffing?

React treats `<div>` and `<section>` as different element types.

During diffing, React will not reuse the existing `div` DOM node as a `section`. It will remove the old `div` subtree and create a new `section` subtree.

#### Explanation

React’s diffing rule is that different element types represent different trees.

So if the previous render returned:

```jsx
<div>
  <UserDetails />
</div>
```

and the next render returns:

```jsx
<section>
  <UserDetails />
</section>
```

React treats the root element type as changed. The old subtree under `div` is unmounted, and the new subtree under `section` is mounted.

This can reset state for components inside that subtree if React replaces the entire subtree.

#### Real-World Usage

This matters when changing wrapper elements conditionally, especially around forms, modals, tabs, or stateful child components. If you only need a semantic or styling change, avoid changing the wrapper type unless you are okay with the subtree being recreated.

# React Diffing Algorithm Interview Answers

### 1. What are React’s main heuristics for making diffing efficient?

React’s main heuristics are:

1. Elements of different types produce different trees.
2. Elements with the same type can usually be reused and updated.
3. Keys help React match children across renders.

These rules allow React to avoid an expensive perfect tree comparison and still update UI efficiently in most real-world cases.

#### Explanation

React does not try to calculate every possible way one tree can transform into another. That would be too expensive for frequent UI updates.

Instead, React uses predictable rules:

- If type changes, replace the subtree.
- If type stays the same, update props and reconcile children.
- If children have keys, match them by key.
- If children do not have keys, match them mostly by position.

This makes diffing fast and practical for interactive applications.

#### Real-World Usage

These heuristics matter in complex UI systems like dashboards, forms, tables, drag-and-drop lists, and route-based layouts. Stable types and stable keys help React preserve state and avoid unnecessary work.

#### Common Follow-Up Question

What trade-offs does React make by using heuristics instead of a perfect diff algorithm?

### 2. Why does React avoid using a general-purpose tree-edit distance algorithm?

React avoids a general-purpose tree-edit distance algorithm because it is too expensive for large UI trees and frequent updates.

React applications update often, so React needs a diffing strategy that is fast enough for real-time user interactions.

#### Explanation

A general tree-edit distance algorithm tries to find the minimum number of operations needed to transform one tree into another. That may be theoretically optimal, but it is not practical for large component trees.

React chooses a heuristic approach instead. It uses type and key comparisons to make good enough decisions quickly.

This gives React predictable performance while keeping the UI responsive.

#### Real-World Usage

In production applications, responsiveness is usually more important than mathematically perfect DOM changes. React’s approach works well as long as developers use stable keys and avoid unnecessary type changes.

### 3. How does React’s diffing behavior differ for DOM elements and custom components?

For DOM elements, React compares the element type and props, then updates only the changed DOM attributes, properties, styles, or children.

For custom components, React compares the component type. If the type is the same, React preserves the component identity and re-renders it with new props. If the type changes, React replaces that component subtree.

#### Explanation

For DOM elements:

```jsx
<button disabled={false}>Save</button>
```

changing to:

```jsx
<button disabled={true}>Save</button>
```

does not replace the button. React updates the `disabled` property.

For custom components:

```jsx
<UserCard user={user} />
```

changing props still keeps the same `UserCard` identity, so state can be preserved.

But changing:

```jsx
<UserCard />
```

to:

```jsx
<AdminCard />
```

changes the component type, so React treats it as a different subtree.

#### Real-World Usage

This is important when building reusable components. Passing different props preserves identity, but switching component types can reset state, effects, refs, and internal UI behavior.

### 4. How do keys help React match children correctly during diffing?

Keys help React identify which child is which across renders.

When children are added, removed, sorted, or moved, stable keys allow React to match each new child with the correct previous child instead of relying only on position.

#### Explanation

Without keys, React compares children mostly by index. That can be wrong when list order changes.

With keys, React can understand that an item is the same item even if it moved to a different position.

Good keys should be:

- Unique among siblings.
- Stable across renders.
- Based on real item identity, such as an ID.

Bad keys include array indexes for dynamic lists, random values, timestamps, or values that change often.

#### Example

```jsx
function EmployeeList({ employees }) {
  return (
    <ul>
      {employees.map((employee) => (
        <li key={employee.id}>{employee.name}</li>
      ))}
    </ul>
  );
}
```

#### Real-World Usage

Keys are critical in sortable tables, editable lists, drag-and-drop UIs, carts, search results, dynamic forms, and notification feeds. They prevent state from attaching to the wrong item.

### 5. Scenario: A list item is inserted at the beginning of a list without stable keys. What diffing problems can occur?

If a list item is inserted at the beginning without stable keys, React may match items by position and think many existing items changed.

This can cause incorrect state reuse, unnecessary updates, wrong input values, broken focus behavior, and visual bugs.

#### Explanation

Suppose the list was:

```jsx
[A, B, C];
```

and a new item is inserted at the beginning:

```jsx
[X, A, B, C];
```

Without stable keys, React may compare by position:

- Old `A` is compared with new `X`.
- Old `B` is compared with new `A`.
- Old `C` is compared with new `B`.

React may reuse component instances for the wrong data items. Local state can appear to “move” to the wrong row.

With stable keys, React knows `A`, `B`, and `C` are the same items and only `X` is new.

#### Example

```jsx
function TodoList({ todos }) {
  return (
    <>
      {todos.map((todo) => (
        <TodoItem key={todo.id} todo={todo} />
      ))}
    </>
  );
}
```

#### Real-World Usage

This issue is common in editable lists, form arrays, shopping carts, search results, and tables where items can be inserted at the top. Stable keys prevent incorrect state and unnecessary rendering work.

#### Common Follow-Up Question

How would this behavior differ if the list used stable database IDs as keys?

React’s diffing algorithm affects state preservation by deciding whether a component is the same component as before or a new one.

If React sees the same component type in the same position with the same key, it usually preserves state. If the type, key, or tree position changes, React may replace the component and reset its state.

The main signals are:

- Element position
- Key, especially inside lists

### 2. Why can poor key choices cause unnecessary DOM changes and incorrect UI state?

Poor key choices cause React to match elements incorrectly during diffing.

If keys are unstable, duplicated, or based on array indexes in dynamic lists, React may reuse the wrong component instance, remount items unnecessarily, or attach state to the wrong data item.

Keys represent identity among siblings.

Good keys help React understand that an item is the same item across renders, even if it moves. Bad keys confuse React’s matching process.

### 3. How does React decide whether to update an existing node or replace it completely?

React decides based mainly on element type and key.

If the type and key match, React usually updates the existing node. If the type or key changes, React treats it as a different node and replaces that subtree.

#### Explanation

For DOM elements, if the type is the same, React updates changed props and children.

For example, a `button` with a changed `disabled` prop stays the same DOM node. React only updates the property.

For custom components, if the component type is the same, React preserves the component identity and re-renders it with new props.

But if a `UserCard` becomes an `AdminCard`, or a `div` becomes a `section`, React treats it as a different tree and replaces it.

#### Example

### 4. What performance issues can appear when diffing large dynamic lists?

Large dynamic lists can create performance issues when React has to compare many children, re-render many rows, or perform unnecessary insert, delete, move, and update work.

The biggest problems usually come from missing keys, unstable keys, unnecessary row re-renders, expensive row components, and rendering too many items at once.

#### Explanation

Diffing itself is designed to be efficient, but large dynamic lists can still be expensive.

Common issues include:

- Hundreds or thousands of children being diffed on every update
- Rows re-rendering even when their data did not change
- Index keys causing incorrect matching after sort/filter/insert
- Expensive calculations inside each row render
- New object/function props breaking memoization
- No virtualization for large visible lists
- Heavy DOM updates after reorder operations

React can optimize matching with keys, but it still has to process the tree that is rendered.

#### Real-World Usage

For large tables, feeds, grids, and search results, teams usually combine stable keys, memoized rows, careful state placement, pagination, virtualization, and deferred updates.

- Are rows using array indexes as keys?

Good UI architecture reduces unnecessary diffing work by limiting how much of the component tree changes for each update.

```jsx
const DashboardWidget = React.memo(function DashboardWidget({ widget }) {
  return (
    <section>
      <h2>{widget.title}</h2>
      <WidgetRenderer type={widget.type} config={widget.config} />
    </section>
  );
});

function Dashboard({ widgets }) {
  return (
    <main>
      {widgets.map((widget) => (
        <DashboardWidget key={widget.id} widget={widget} />
      ))}
    </main>
  );
}
```

#### Real-World Usage

This approach is common in analytics dashboards, operational monitoring tools, financial platforms, admin portals, and enterprise reporting systems where many widgets refresh at different frequencies.

# React Diffing Algorithm Interview Answers

### 1. How does React diff keyed children compared to unkeyed children?

React diffs keyed children by using the key as the child’s identity across renders.

For unkeyed children, React mostly compares children by their position in the list. This works for static lists, but it becomes risky when items are inserted, removed, filtered, or reordered.

#### Explanation

With keyed children, React can say: “This item has the same key as before, so it is the same logical child even if its position changed.”

With unkeyed children, React has less information. It compares the first old child with the first new child, the second old child with the second new child, and so on.

That means unkeyed dynamic lists can cause incorrect state preservation or unnecessary updates.

#### Example

```jsx
function MessageList({ messages }) {
  return (
    <>
      {messages.map((message) => (
        <MessageRow key={message.id} message={message} />
      ))}
    </>
  );
}
```

#### Real-World Usage

Keyed diffing is critical in sortable tables, chat messages, notifications, dynamic forms, carts, playlists, and drag-and-drop lists where item identity must survive position changes.

#### Common Follow-Up Question

Why is `key={index}` not a reliable identity for a list that can reorder?

### 2. What happens internally when React detects an insertion, deletion, or move during child diffing?

React records the required work on the Fiber tree instead of immediately changing the DOM.

During child diffing, React marks Fibers with work such as insert, delete, update, or move. Later, during the commit phase, React applies those changes to the actual DOM.

#### Explanation

At a high level:

- **Insertion**: React creates a new Fiber for the new child and marks it for placement.
- **Deletion**: React marks the old Fiber for removal.
- **Move**: React reuses the existing keyed Fiber but marks that its position needs to change.
- **Update**: React reuses the existing Fiber and records prop or child changes.

React does this work during the render phase. The render phase calculates what should happen, while the commit phase performs the real DOM mutations.

#### Real-World Usage

This matters when debugging why a component remounted, why an effect cleanup ran, why focus was lost, or why a list reorder caused unexpected visual behavior.

#### Common Follow-Up Question

What is the difference between a moved Fiber and a remounted Fiber?

---

### 3. How does diffing produce update work without directly mutating the DOM immediately?

Diffing produces update work by comparing the old and new React element trees and recording the required changes on React’s internal Fiber structure.

React does not mutate the DOM immediately during diffing because diffing happens in the render phase, which must be pure, interruptible, and safe to restart.

#### Explanation

React separates work into two major phases:

- **Render phase**: React renders components, diffs elements, and prepares work.
- **Commit phase**: React applies the prepared changes to the DOM.

This separation is important because React may pause, restart, or discard render work in concurrent rendering.

If React mutated the DOM during diffing, the UI could end up partially updated or inconsistent if the render was interrupted.

#### Real-World Usage

This explains why a component can render multiple times without every render causing DOM changes. Only the work that reaches the commit phase becomes visible to the user.

#### Common Follow-Up Question

Why is the commit phase not interruptible like the render phase?

---

### 4. Why can rendering a new object or array each time increase the amount of work React needs to evaluate?

Rendering a new object or array each time can increase work because it changes reference identity, even if the contents are visually the same.

React may need to re-render child components, re-run memo comparisons, recalculate derived output, or evaluate more of the tree because props appear changed by reference.

#### Explanation

React uses shallow comparisons in places like `React.memo`, dependency arrays, and memoized props.

If a parent creates a new object, array, or function on every render, a memoized child may still re-render because the prop reference changed.

For example, this creates a new array every render:

```jsx
const visibleColumns = columns.filter((column) => column.visible);
```

That may be fine for small cases, but in large trees it can cause unnecessary work when passed to expensive children.

#### Example

```jsx
import { useMemo } from "react";

function DataGrid({ columns, rows }) {
  const visibleColumns = useMemo(() => {
    return columns.filter((column) => column.visible);
  }, [columns]);

  return <Grid columns={visibleColumns} rows={rows} />;
}
```

#### Real-World Usage

This matters in large dashboards, tables, charts, forms, and design system components where unstable references can cause expensive children to re-render even when the UI appears unchanged.

#### Common Follow-Up Question

When is `useMemo` useful, and when is it unnecessary?

### 5. Scenario: A component tree appears visually unchanged, but React still performs significant diffing work. What would you analyze?

I would analyze why React is re-rendering that part of the tree and whether props, state, context, or keys are changing even though the visual output looks the same.

A visually unchanged UI can still cause significant diffing if React has to evaluate a large subtree repeatedly.

#### Explanation

I would investigate:

- Parent state changes causing large subtree re-renders.
- Context updates invalidating many consumers.
- New object, array, or function props created every render.
- Missing `React.memo` around expensive stable subtrees.
- Ineffective memoization due to unstable prop references.
- Keys changing unexpectedly and causing remounts.
- Conditional rendering changing component identity.
- Large lists being rendered without virtualization.
- Expensive derived calculations inside render.
- Store subscriptions causing broad updates instead of selective updates.

React may eventually commit few or no DOM changes, but it can still spend time rendering and diffing the tree.

#### Example

```jsx
import { memo, useMemo } from "react";

const Widget = memo(function Widget({ config }) {
  return <section>{config.title}</section>;
});

function Dashboard({ title }) {
  const widgetConfig = useMemo(() => {
    return { title };
  }, [title]);

  return <Widget config={widgetConfig} />;
}
```

#### Real-World Usage

In production, I would use React Profiler and browser Performance tools to separate render/diffing cost from commit, layout, paint, and network cost. This helps identify whether the issue is React work or browser rendering work.
