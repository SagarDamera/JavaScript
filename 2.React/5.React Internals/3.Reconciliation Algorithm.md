# Topic: Reconciliation Algorithm

- Reconciliation is React’s process of comparing old and new Virtual DOM trees.

- React uses reconciliation to decide what needs to change in the real DOM.

- If element type changes, React destroys the old subtree and creates a new one.

- If element type is the same, React updates only changed props and continues checking children.

- Keys help React match list items correctly between renders.

- Stable keys prevent unnecessary remounts and preserve component state.

- Bad keys like array indexes can cause wrong UI state when list order changes.

- Reconciliation is fast because React uses practical assumptions instead of comparing every possibility deeply.

- Common mistake: thinking every re-render updates the real DOM; React updates only actual differences.

- Interview trap: reconciliation decides what changed; rendering creates the new UI description.

## One-Line Final Recall

Reconciliation is how React compares old and new UI trees to update only the necessary parts of the real DOM.

### 1. What is React reconciliation, and why does React need it?

React reconciliation is the process React uses to compare the previous UI tree with the new UI tree and decide what needs to change.

React needs reconciliation because every state or props update can produce a new React element tree. Instead of replacing the entire DOM, React compares the old and new trees and updates only the parts that actually changed.

#### Explanation

When a component re-renders, React does not immediately recreate the whole DOM. It creates a new description of the UI using React elements, compares it with the previous description, and calculates the minimal set of changes needed.

This process is called reconciliation.

The goal is to keep the UI correct while avoiding unnecessary DOM work.

#### Real-World Usage

Reconciliation is important in almost every React app. It allows state-driven UIs to update efficiently when forms change, lists update, filters apply, routes switch, or server data refreshes.

### 2. What problem does the reconciliation algorithm solve during rendering?

The reconciliation algorithm solves the problem of efficiently updating the UI when application state changes.

It helps React determine what changed between the previous render and the next render, so React can update only the necessary parts of the DOM instead of rebuilding everything.

#### Explanation

React components return a tree of React elements. On every update, React gets a new tree.

The reconciliation algorithm compares the old tree and the new tree. Based on that comparison, React decides whether to:

- Reuse an existing component.
- Update props.
- Add a new node.
- Remove an old node.
- Move or reorder list items.
- Reset component state when the component identity changes.

Without reconciliation, React would either do too much DOM work or need developers to manually update the DOM.

#### Real-World Usage

This is useful in dynamic UIs like search results, dashboards, notifications, editable tables, multi-step forms, and real-time data screens where only some parts of the page change frequently.

### 3. How does React decide what parts of the UI need to be updated?

React decides what parts of the UI need to be updated by comparing the previous React element tree with the new React element tree.

It uses component type, element type, props, and keys to decide whether to reuse existing work, update existing nodes, or recreate parts of the tree.

#### Explanation

At a high level:

- If the element type is the same, React usually reuses the existing node and updates changed props.
- If the component type is the same, React preserves the component instance/state and re-renders it with new props/state.
- If the type changes, React usually destroys the old subtree and creates a new one.
- In lists, keys help React match old and new children correctly.

React does not compare every possible tree arrangement perfectly. It uses practical rules that are fast enough for real applications.

#### Example

```jsx
function UserCard({ isAdmin }) {
  return <div>{isAdmin ? <AdminBadge /> : <UserBadge />}</div>;
}
```

#### Real-World Usage

This matters when designing component boundaries and lists. Stable component types and stable keys help React preserve state correctly and avoid unnecessary remounts.

### 4. What is the relationship between JSX, React elements, and reconciliation?

JSX is syntax that describes UI. It gets compiled into React elements. React reconciliation compares those React elements from one render to the next to decide how the actual UI should update.

#### Explanation

JSX itself is not the DOM. It is a convenient way to write React element descriptions.

For example, this JSX:

```jsx
<h1>Hello</h1>
```

represents a React element describing an `h1`.

When state changes, the component returns a new set of React elements. Reconciliation compares the previous elements with the new elements and decides what work is needed.

So the flow is:

JSX → React elements → reconciliation → DOM updates during commit.

#### Real-World Usage

Understanding this helps senior engineers explain why React re-rendering a component does not always mean React is recreating the DOM. React may call the component again, compare the result, and only commit the actual changes.

### 5. Why does React avoid directly re-rendering the entire DOM on every state change?

React avoids directly re-rendering the entire DOM because DOM operations are expensive and can cause poor performance, lost state, layout recalculation, repainting, and visual instability.

Instead, React calculates what changed and updates only the necessary parts of the DOM.

#### Explanation

The DOM is much slower to update than creating and comparing lightweight React elements in memory.

If React replaced the entire DOM on every state change, simple interactions like typing into an input, toggling a checkbox, or updating one row in a table could cause unnecessary work across the whole page.

Reconciliation allows React to keep updates focused and predictable.

#### Real-World Usage

This is especially important in large applications with forms, tables, dashboards, modals, and real-time updates. Efficient DOM updates help keep the UI responsive and preserve user state like focus, input values, and scroll position.

### 6. How does React compare the previous React element tree with the new React element tree?

React compares the previous React element tree with the new React element tree by walking both trees and checking element type, props, and keys.

If React sees the same element type in the same position, it tries to reuse the existing DOM node or component instance and update only what changed. If the type or key changes, React may treat it as a different piece of UI and recreate that part of the tree.

#### Explanation

React reconciliation is based on practical heuristics, not a perfect tree-diff algorithm.

At a high level, React checks:

- Is the element type the same?
- Is the component type the same?
- Are the props different?
- Do children have stable keys?
- Should the existing subtree be reused or replaced?

This lets React update the UI efficiently without doing an expensive full comparison of every possible tree change.

#### Real-World Usage

This is important when designing stable component structures. Keeping element types and keys stable helps React preserve state, avoid unnecessary remounts, and update only the parts of the UI that changed.

### 7. Why does React treat elements of different types as completely different subtrees?

React treats elements of different types as completely different subtrees because different types usually represent different UI structures, behavior, state, and lifecycle requirements.

When the element type changes, React assumes it is safer and faster to discard the old subtree and create a new one instead of trying to reuse it.

#### Explanation

For example, changing from a `div` to a `section`, or from `UserProfile` to `AdminProfile`, means React cannot safely assume the old tree should be reused.

Different component types may have different state, effects, children, refs, and DOM structure.

So React follows a simple rule: different type means different identity. The old subtree is unmounted, and the new subtree is mounted.

#### Example

```jsx
function Profile({ isAdmin }) {
  return isAdmin ? <AdminProfile /> : <UserProfile />;
}
```

#### Real-World Usage

This matters when conditionally rendering components. If the component type changes, React resets that subtree’s state. This can be useful when switching between truly different screens, but harmful if the developer expected state to be preserved.

#### Common Follow-Up Question

### 8. How does React handle reconciliation when a component returns a different element type between renders?

When a component returns a different element type between renders, React usually unmounts the previous subtree and mounts a new subtree.

This means the old DOM nodes are removed, the old component state is discarded, effects are cleaned up, and the new element tree starts fresh.

#### Explanation

For example, if a component previously returned an `input` and later returns a `select`, React treats that as a different element type.

React does not try to convert one element type into another while preserving the same identity. It replaces the subtree because the structure and behavior are different.

This behavior also applies to custom components. Switching from one component type to another causes React to remount that part of the UI.

#### Example

```jsx
function Field({ type }) {
  if (type === "text") {
    return <input placeholder="Enter name" />;
  }

  return (
    <select>
      <option>Select name</option>
    </select>
  );
}
```

#### Real-World Usage

This is important in forms, tabs, modals, and conditional layouts. If state unexpectedly resets, one common cause is that the rendered element or component type changed between renders.

### 9. Why are keys important when reconciling lists?

Keys are important because they help React identify which list item is which across renders.

A stable key lets React match an old item with its new position, preserve the correct component state, and avoid unnecessary remounting when items are added, removed, or reordered.

#### Explanation

Without keys, React mostly compares list children by position.

That can cause problems when items move. React may think the first item became a different item, the second item became another different item, and so on.

With stable keys, React can track item identity independently from position.

Good keys should be:

- Stable.
- Unique among siblings.
- Based on real item identity, such as a database ID.

Indexes should be avoided as keys when the list can reorder, insert, or delete items.

#### Example

```jsx
function UserList({ users }) {
  return (
    <ul>
      {users.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```

#### Real-World Usage

Keys are critical in dynamic lists like search results, drag-and-drop lists, tables, notifications, form arrays, carts, and chat messages. Good keys prevent incorrect state reuse and visual bugs.

### 10. Scenario: A list of items is reordered, but each item has a stable key. How does reconciliation behave?

If a list is reordered and each item has a stable key, React uses the keys to match each old item with its corresponding new item.

React understands that the items moved instead of treating them as completely new items. This allows React to preserve each item’s component state and update the DOM more correctly.

#### Explanation

For example, if item `A` moves from position 1 to position 3, React can still identify it as the same item because its key is unchanged.

React may need to reorder DOM nodes, but it does not need to remount every item.

This is the main reason stable keys are important: they represent identity, not position.

#### Example

```jsx
function TaskList({ tasks }) {
  return (
    <>
      {tasks.map((task) => (
        <TaskRow key={task.id} task={task} />
      ))}
    </>
  );
}
```

#### Real-World Usage

This is common in drag-and-drop lists, sortable tables, priority queues, kanban boards, playlists, and shopping carts. Stable keys ensure each item keeps the correct local state after reordering.

# React Reconciliation Algorithm Interview Answers

### 1. What assumptions does React make to make reconciliation efficient?

React makes two main assumptions to make reconciliation efficient:

1. Elements of different types produce different subtrees.
2. Developers can use keys to tell React which children are stable across renders.

These assumptions allow React to avoid doing an expensive perfect comparison between every old and new tree structure.

#### Explanation

A perfect tree-diff algorithm would be too expensive for real UI updates. React uses practical rules instead.

If the element type changes, React assumes it is a different UI and replaces that subtree. If the type is the same, React tries to reuse it and update only the changed props or children.

For lists, React relies on keys to understand item identity. A stable key helps React know that an item moved, instead of thinking it was removed and recreated.

#### Real-World Usage

This matters when building dynamic UIs with conditional rendering, lists, tables, tabs, and forms. Stable component types and stable keys help React preserve state and avoid unnecessary DOM work.

#### Common Follow-Up Question

### 2. How does React’s reconciliation approach avoid the cost of a full tree-edit distance algorithm?

React avoids the cost of a full tree-edit distance algorithm by using simple heuristics based on element type and keys.

Instead of comparing every possible transformation between two trees, React compares nodes mostly by position, type, and key, which makes reconciliation fast enough for frequent UI updates.

#### Explanation

A full tree-edit distance algorithm can be very expensive for large UI trees. React avoids that by making predictable assumptions.

At a high level:

- Same type usually means reuse and update.
- Different type means replace the subtree.
- Keys help match children across list changes.
- React compares sibling lists using keys instead of trying every possible move.

This gives React a practical balance between correctness and performance.

#### Real-World Usage

This is why React apps can update frequently in response to state changes without developers manually manipulating the DOM. The trade-off is that developers must provide stable keys and avoid unnecessary type changes.

#### Common Follow-Up Question

### 3. What happens during reconciliation when a component’s props change but its element type stays the same?

When a component’s props change but its element type stays the same, React reuses the existing component identity and re-renders it with the new props.

The component’s state is preserved because React sees it as the same component in the same position.

#### Explanation

For example, if React previously rendered `<UserCard user={oldUser} />` and now renders `<UserCard user={newUser} />`, the component type is still `UserCard`.

React does not unmount and remount the component. It calls the component again with the new props, compares the returned element tree, and updates the necessary parts of the UI.

This is different from changing the component type entirely, which would cause React to replace the subtree.

#### Example

```jsx
function UserCard({ user }) {
  return (
    <section>
      <h2>{user.name}</h2>
      <p>{user.role}</p>
    </section>
  );
}
```

#### Real-World Usage

This behavior is important in profile pages, reusable cards, form fields, table rows, and dashboards. It allows React to update displayed data while preserving local component state like input focus, expanded sections, or internal UI state.

### 4. How can unstable keys cause unnecessary unmounting, remounting, or state loss?

Unstable keys can make React think an existing item is a completely different item between renders.

When that happens, React may unmount the old component, mount a new one, lose local state, reset focus, restart effects, or perform unnecessary DOM work.

#### Explanation

Keys represent identity. If a key changes every render, React cannot reliably match the old item to the new item.

Bad examples include:

- `key={Math.random()}`
- `key={Date.now()}`
- Index keys in lists that reorder, insert, or delete items
- Keys based on values that can change frequently

Stable keys should come from persistent item identity, such as a database ID.

#### Example

```jsx
function TodoList({ todos }) {
  return (
    <>
      {todos.map((todo) => (
        <TodoRow key={todo.id} todo={todo} />
      ))}
    </>
  );
}
```

#### Real-World Usage

This is critical in editable tables, form arrays, drag-and-drop lists, carts, playlists, and search results. Unstable keys can cause bugs that look like random state resets or incorrect data appearing in the wrong row.

#### Common Follow-Up Question

Why is `key={Math.random()}` worse than using no key at all?

### 5. Scenario: A developer uses array indexes as keys in a sortable list. What reconciliation problems can occur?

Using array indexes as keys in a sortable list can cause React to match items by position instead of actual identity.

When the list is reordered, React may reuse the wrong component instance for a different item, causing incorrect state, wrong input values, broken animations, focus issues, or unnecessary UI updates.

#### Explanation

Indexes are only stable if the list order never changes.

In a sortable list, the index changes when items move. React sees the item at index `0` as the same identity even if a completely different item moved into that position.

That can cause local state to follow the position instead of the item.

For example, an expanded row, edited input value, selected checkbox, or validation state may appear on the wrong item after sorting.

#### Example

```jsx
function SortableTaskList({ tasks }) {
  return (
    <>
      {tasks.map((task) => (
        <TaskRow key={task.id} task={task} />
      ))}
    </>
  );
}
```

#### Real-World Usage

This issue commonly appears in sortable tables, drag-and-drop lists, kanban boards, shopping carts, playlists, and editable form rows. Stable item IDs should be used instead of array indexes.

#### Common Follow-Up Question

# React Reconciliation Algorithm Interview Answers

### 1. How does reconciliation interact with component identity and state preservation?

Reconciliation determines component identity by looking at where a component appears in the tree, its type, and its key.

If React considers the new component to be the same identity as the previous one, it preserves state. If React considers it a different identity, it unmounts the old component and mounts a new one, which resets local state.

#### Explanation

React state is not stored inside the JSX element itself. React stores state internally and associates it with a specific position in the rendered tree.

For React to preserve state, the component must usually have:

- The same component type.
- The same position in the tree.
- The same key, if a key is used.

If any of these identity signals change, React may treat it as a new component and reset its state.

#### Example

```jsx
function Profile({ user }) {
  return <UserForm key={user.id} user={user} />;
}
```

#### Real-World Usage

This is important in forms, tabs, modals, route layouts, editable tables, and multi-step flows. Understanding identity helps prevent unexpected state resets or intentionally force a reset when switching between different records.

### 2. Why does moving a component to a different position in the tree often reset its state?

Moving a component to a different position in the tree often resets its state because React associates state with the component’s identity at a specific tree position.

If the component appears in a different location, React may treat it as a different component instance, unmount the old one, and mount a new one.

#### Explanation

React does not preserve state just because the JSX looks visually similar.

For example, if `<SearchBox />` moves from one branch of conditional rendering to another, React may see it as removed from the old position and added at a new position.

That means local state, focus, refs, and effects can be reset.

Keys can help in sibling lists, but they do not automatically preserve state across completely different tree locations.

### 3. How does reconciliation determine whether to reuse an existing Fiber or create a new one?

React determines whether to reuse an existing Fiber by comparing the new element with the previous Fiber using type, key, and position among siblings.

If the type and key match, React can usually reuse the existing Fiber and preserve state. If they do not match, React creates a new Fiber and schedules the old one for deletion.

#### Explanation

A Fiber is React’s internal unit of work for a component or DOM element.

During reconciliation, React compares the new React element against the current Fiber tree. React asks: “Does this new element represent the same thing as the previous Fiber?”

At a high level:

- Same type and same key usually means reuse the Fiber.
- Different type means create a new Fiber.
- Different key means create a new Fiber.
- Missing or unstable keys can cause incorrect reuse or unnecessary replacement.
- Position matters when keys are not available.

Reusing a Fiber allows React to keep state, refs, and update history attached to that component.

#### Real-World Usage

This matters in dynamic lists, conditional components, route-level layouts, and reusable form components. Stable type and key decisions help React preserve the correct state and avoid unnecessary mounts.

Common production bugs include unexpected state resets, state being preserved when it should reset, wrong input values appearing in list rows, broken focus behavior, unnecessary remounts, and effects running more often than expected.

Most of these issues come from unstable keys, incorrect keys, changing component types, or moving components between different tree positions.

Typical reconciliation-related bugs include:

These bugs are often difficult because the UI may look correct initially but fail after sorting, filtering, toggling, or switching views.

### 5. Scenario: A form inside a conditional UI unexpectedly loses input state when toggling views. How would you investigate reconciliation-related causes?

I would investigate whether React is treating the form as a new component when the view toggles.

Specifically, I would check whether the form’s type, key, or position in the tree changes between the two conditional branches.

#### Explanation

I would look for these causes:

- The form is rendered in different branches of a conditional.
- The form component type changes between views.
- A `key` changes when toggling views.

- The form is being unmounted instead of hidden.

function ProfilePage({ editMode }) {

Architectural patterns that help preserve component identity include stable component trees, consistent keys, predictable layout boundaries, controlled conditional rendering, and state ownership close to the correct component.

### 4. What are the trade-offs of React’s heuristic reconciliation model?

- Fast enough for frequent UI updates.
- Predictable rules for reuse and replacement.
- Avoids expensive full tree-edit distance computation.

- React may not always produce the theoretical minimum DOM operations.
- Incorrect keys can cause state and rendering bugs.
- Moving components across tree positions can reset state.
- Changing component types can remount subtrees.
- Developers need to understand identity boundaries.

This is a good engineering trade-off because UI updates need to be fast and predictable more than mathematically perfect.

#### Real-World Usage

### 5. Scenario: A design system component is used across many product surfaces and sometimes loses internal state unexpectedly. How would you evaluate whether reconciliation is the root cause?

# React Reconciliation Algorithm Interview Answers

### 1. How does React reconcile children when keys are present versus when keys are missing?

When keys are present, React uses them to match children across renders by identity.

When keys are missing, React mostly matches children by their position in the array.

#### Explanation

Keys help React understand that a child is the same item even if its position changes.

With stable keys:

- React can identify moved items.
- React can preserve the correct component state.
- React can avoid unnecessary remounting.
- React can handle insertions, deletions, and reordering more accurately.

Without keys, React falls back to positional comparison. This works for static lists, but it becomes risky when items are inserted, removed, filtered, or sorted.

If the order changes and keys are missing, React may reuse the wrong component instance for a different item.

#### Example

```jsx
function UserList({ users }) {
  return (
    <>
      {users.map((user) => (
        <UserRow key={user.id} user={user} />
      ))}
    </>
  );
}
```

#### Real-World Usage

Stable keys are essential in sortable tables, search results, form arrays, carts, kanban boards, playlists, and real-time feeds where item order can change.

#### Common Follow-Up Question

### 2. What is the difference between reconciling a single child and reconciling an array of children?

Reconciling a single child is simpler because React only needs to compare one previous child with one new child.

Reconciling an array of children is more complex because React must determine which children were updated, inserted, removed, or moved, and keys become important for tracking identity.

#### Explanation

For a single child, React mainly checks:

- Is the type the same?
- Is the key the same, if present?
- Can the existing Fiber be reused?

For an array of children, React compares multiple siblings. It needs to match old children to new children. If keys are present, React can match children by key. If keys are missing, React compares by position.

This is why lists are where reconciliation issues most commonly appear.

#### Example

```jsx
function Page({ showDetails, users }) {
  return (
    <>
      {showDetails && <DetailsPanel />}

      {users.map((user) => (
        <UserRow key={user.id} user={user} />
      ))}
    </>
  );
}
```

#### Real-World Usage

Single-child reconciliation matters in conditional rendering. Array reconciliation matters in dynamic lists, tables, grids, menus, tab lists, and repeated form sections.

#### Common Follow-Up Question

### 3. How does reconciliation decide between updating, deleting, inserting, and moving elements?

React decides based on whether the new element matches an existing Fiber by type, key, and position among siblings.

If React finds a matching Fiber, it updates and reuses it. If an old Fiber has no matching new element, React deletes it. If a new element has no matching old Fiber, React inserts it. If a keyed item exists but appears in a different order, React may move it.

#### Explanation

At a high level:

- **Update**: Same type and same key. React reuses the Fiber and updates props or children.
- **Delete**: An old child no longer exists in the new tree.
- **Insert**: A new child exists that did not exist before.
- **Move**: A keyed child exists in both old and new lists but appears in a different position.

React records these changes during the render phase as work to be performed later.

The actual DOM mutation happens during the commit phase.

#### Example

```jsx
function TaskList({ tasks }) {
  return (
    <ul>
      {tasks.map((task) => (
        <TaskItem key={task.id} task={task} />
      ))}
    </ul>
  );
}
```

#### Real-World Usage

This behavior matters in sortable tables, drag-and-drop lists, editable rows, virtualized lists, notifications, and any UI where items are frequently added, removed, or reordered.

#### Common Follow-Up Question

### 4. Why does reconciliation produce work for the render phase but not directly mutate the DOM?

Reconciliation produces work during the render phase because React first calculates what the next UI should look like before applying any real DOM changes.

React does not directly mutate the DOM during reconciliation because the render phase must stay interruptible, pure, and safe to pause, restart, or discard.

#### Explanation

React has two important phases:

- **Render phase**: React compares trees and prepares the list of changes.
- **Commit phase**: React applies those changes to the DOM.

During reconciliation, React may determine that a node should be inserted, updated, moved, or deleted. But it only records that work internally.

The actual DOM mutations happen later in the commit phase. This separation is important because render work can be interrupted in concurrent React, while commit work must complete consistently.

#### Real-World Usage

This explains why a component can render multiple times without the DOM changing every time. React may calculate possible work, but only committed work becomes visible to the user.

#### Common Follow-Up Question

### 5. Scenario: A React app shows incorrect item state after filtering, sorting, and inserting list items. What reconciliation behavior would you analyze?

I would analyze whether React is matching list items by the wrong identity during reconciliation.

The first thing I would check is the list key strategy, especially whether the app is using array indexes, duplicate keys, unstable keys, or keys based on mutable values.

#### Explanation

Filtering, sorting, and inserting items all change list positions.

If React uses indexes as keys, item identity becomes tied to position instead of the actual data item. After sorting or filtering, React may reuse the component state from one item for a different item.

That can cause bugs like:

- Input values appearing in the wrong row.
- Expanded state moving to another item.
- Selected checkbox state attaching to the wrong item.
- Validation messages showing on the wrong record.
- Effects or subscriptions staying connected to the wrong item.
- Rows remounting unnecessarily.

I would verify that every rendered item has a stable, unique key based on persistent identity, such as `item.id`.

#### Example

```jsx
function ProductList({ products }) {
  return (
    <>
      {products.map((product) => (
        <ProductRow key={product.id} product={product} />
      ))}
    </>
  );
}
```

#### Real-World Usage

This issue is common in editable tables, filtered search results, dynamic form arrays, shopping carts, kanban boards, and admin grids. Correct keys are required for both correctness and performance.
