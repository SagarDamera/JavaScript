# Topic: Render Phase

- Render phase is when React calls components to calculate the next UI.

- React creates a new Virtual DOM / Fiber tree during this phase.

- It compares new output with previous output to figure out what changed.

- Render phase should be pure; do not cause side effects here.

- Avoid API calls, DOM updates, timers, or subscriptions during render.

- React may pause, restart, or discard render work in concurrent rendering.

- Because render can run more than once, component logic must be predictable and side-effect free.

- State and props are read during render to decide what UI should look like.

- Common mistake: doing side effects directly inside the component body.

- Interview trap: render phase calculates changes, but it does not update the real DOM yet.

## One-Line Final Recall

Render phase is React’s pure calculation step where it prepares the next UI before committing changes to the DOM.

### 1. What is the render phase in React?

The render phase is the step where React calls component functions and calculates what the UI should look like for the current state and props.

#### Explanation

During this phase, React does not directly update the browser DOM. It builds a new React element tree, compares it with the previous tree, and prepares a list of changes that may need to be committed later.

In modern React, the render phase can be paused, restarted, or even discarded, especially with concurrent rendering. That is why render code must be predictable and side-effect free.

#### Example

```jsx
function UserProfile({ user }) {
  return <h1>Hello, {user.name}</h1>;
}
```

Here, React calls `UserProfile` during the render phase to calculate the JSX output.

#### Real-World Usage

This is important when debugging why a component re-renders. A render means React recalculated the UI, but it does not always mean the DOM was changed.

### 2. What is React trying to calculate during the render phase?

React is trying to calculate the next version of the UI based on the latest props, state, and context.

#### Explanation

React takes the current application data and produces a new React element tree. Then it compares that tree with the previous one to determine what has changed.

The output of the render phase is not the actual DOM update. It is React’s internal understanding of what the UI should become.

#### Example

```jsx
function Counter({ count }) {
  return <button>Count: {count}</button>;
}
```

If `count` changes from `1` to `2`, React calculates that the button text should change.

#### Real-World Usage

In real applications, this helps React efficiently update only the parts of the UI affected by state or prop changes, instead of rebuilding the whole page.

### 3. Why does rendering in React not always mean updating the DOM immediately?

Rendering does not always update the DOM immediately because React separates calculating the UI from committing changes to the screen.

#### Explanation

The render phase is where React prepares the next UI. The commit phase is where React applies changes to the DOM.

React may render a component and later decide not to commit that work. This can happen if a newer update arrives, rendering is interrupted, or React determines that no actual DOM change is needed.

This separation allows React to optimize performance and support concurrent rendering.

#### Real-World Usage

This matters when analyzing performance. Seeing a component render does not always mean the browser DOM changed or the user saw a visual update.

### 4. What is the difference between calling a component function and committing changes to the screen?

Calling a component function means React is calculating what the component should return. Committing changes means React is applying the final changes to the DOM and updating what the user sees.

#### Explanation

A functional component is just a function that returns React elements. React may call this function during rendering to understand the desired UI.

Committing is a separate step. During commit, React updates DOM nodes, attaches refs, runs layout effects, and makes the result visible in the browser.

So, component execution is preparation. DOM commit is the actual screen update.

#### Example

```jsx
function Message({ text }) {
  console.log("Component function called");

  return <p>{text}</p>;
}
```

The log may run because React called the component function, but that does not guarantee the DOM was immediately changed.

#### Real-World Usage

This distinction helps avoid incorrect assumptions during debugging. For example, logs inside a component body show render activity, not necessarily committed UI changes.

### 5. Why should render logic be pure and free of side effects?

Render logic should be pure because React may call it multiple times, pause it, restart it, or discard it before committing changes.

#### Explanation

A pure render means the component returns the same UI for the same props, state, and context, without changing anything outside the render process.

Side effects like API calls, subscriptions, timers, DOM mutations, or local storage writes should not happen during render. They belong in event handlers or effects.

If side effects happen during render, they may run unexpectedly multiple times and cause bugs, duplicate API calls, inconsistent UI, or performance issues.

#### Example

```jsx
function Product({ product }) {
  return <h2>{product.name}</h2>;
}
```

This is pure because it only calculates UI from props and does not modify external state.

#### Real-World Usage

In production React apps, keeping render pure makes components easier to test, debug, optimize, and safely run under Strict Mode and concurrent rendering.

## 6. What happens when React starts rendering a component after a state or props change?

When state or props change, React schedules an update and starts rendering the affected component tree to calculate the next UI.

### Explanation

React calls the component function again with the latest props, state, and context. The component returns new React elements, and React compares them with the previous output.

At this point, React is only calculating what should change. The actual DOM update happens later during the commit phase.

### Example

```jsx
function Counter() {
  const [count, setCount] = React.useState(0);

  return <button onClick={() => setCount(count + 1)}>Count: {count}</button>;
}
```

When `setCount` runs, React schedules a render, calls `Counter` again, and calculates that the button text should change.

### Real-World Usage

This is useful when debugging re-renders. A state change does not directly mutate the DOM; it asks React to recalculate the UI first.

## 7. How does React use the render phase to build or update the Fiber tree?

During the render phase, React creates or updates Fiber nodes that represent the component tree and the work needed for each part of the UI.

### Explanation

A Fiber is React’s internal unit of work for a component or DOM element. When rendering starts, React walks through the component tree and builds a work-in-progress Fiber tree.

This work-in-progress tree represents the next possible UI. React compares it with the current Fiber tree and records changes such as insertions, updates, and deletions.

Those changes are not applied immediately. They are prepared during render and later committed during the commit phase.

### Real-World Usage

Fiber is what allows React to split rendering work into smaller units. This is important for large applications where React needs to keep the UI responsive while processing updates.

## 8. Why can the render phase be paused, restarted, or abandoned in Concurrent Rendering?

The render phase can be paused, restarted, or abandoned because it is only calculation work, not visible DOM mutation.

### Explanation

In Concurrent Rendering, React can work on updates with different priorities. If a high-priority update arrives, like user typing or clicking, React can pause lower-priority rendering work and handle the urgent update first.

React may also restart rendering if newer state or props make the previous render result outdated. It can abandon work that is no longer needed.

This is safe only because render logic is expected to be pure and free of side effects.

### Real-World Usage

This helps React keep applications responsive during expensive UI updates, such as filtering large lists, rendering dashboards, or switching routes with heavy components.

## 9. What kinds of operations should not happen during the render phase?

Side effects should not happen during the render phase.

### Explanation

Render should only calculate UI. It should not perform operations that change external systems or depend on timing.

Operations that should not happen during render include:

- API calls
- DOM mutations
- subscriptions
- timers
- logging analytics events
- updating local storage
- modifying global variables
- calling `setState` unconditionally
- starting WebSocket connections

These operations should be placed in event handlers or effects depending on when they need to run.

### Example

```jsx
function UserProfile({ userId }) {
  React.useEffect(() => {
    fetch(`/api/users/${userId}`);
  }, [userId]);

  return <div>User Profile</div>;
}
```

The API call is placed in `useEffect`, not directly inside the component body.

### Real-World Usage

In production apps, keeping render free of side effects prevents duplicate requests, memory leaks, inconsistent UI, and bugs under Strict Mode or Concurrent Rendering.

## 10. Scenario: A component makes an API call directly inside the component body. Why is this a render phase problem?

Making an API call directly inside the component body is a render phase problem because the component body runs during rendering, and React may render multiple times before committing.

### Explanation

A component function should only calculate JSX. If an API call is placed directly inside the body, it may run every time React renders the component.

This can cause duplicate network requests, race conditions, infinite loops, performance problems, and inconsistent data. In development, React Strict Mode may also intentionally call render logic more than once to detect unsafe patterns.

API calls should usually be placed inside `useEffect` or triggered from an event handler.

### Example

```jsx
function UserProfile({ userId }) {
  const [user, setUser] = React.useState(null);

  React.useEffect(() => {
    let cancelled = false;

    async function loadUser() {
      const response = await fetch(`/api/users/${userId}`);
      const data = await response.json();

      if (!cancelled) {
        setUser(data);
      }
    }

    loadUser();

    return () => {
      cancelled = true;
    };
  }, [userId]);

  return <div>{user ? user.name : "Loading..."}</div>;
}
```

Here, the API call runs as an effect after render, and it is tied to `userId`.

### Real-World Usage

This pattern is common in real applications when loading user details, dashboard data, or page-level content. For larger apps, this logic is often moved into React Query, SWR, route loaders, or a custom data-fetching hook.

### Common Follow-Up Question

## 11. How does React decide which components need to be evaluated during the render phase?

React starts from the component where an update was scheduled and evaluates the affected part of the component tree.

### Explanation

A render can be triggered by state changes, prop changes, context updates, or parent re-renders. React walks the relevant subtree and calls component functions to calculate the next UI.

By default, when a parent renders, React may also evaluate its children. However, React can skip some work when it determines nothing relevant changed, for example through `React.memo`, stable props, memoized values, or internal bailout checks.

React is not blindly re-rendering the entire application every time. It schedules work based on where the update happened and then decides how far that work needs to propagate.

### Real-World Usage

## 12. What is the relationship between the render phase and reconciliation?

Reconciliation happens during the render phase. It is the process React uses to compare the new UI description with the previous one and decide what changed.

### Explanation

During render, React calls components and produces a new React element tree. React then reconciles that new output against the previous Fiber tree.

The goal is to figure out which parts of the UI can be reused, which need updates, which should be inserted, and which should be removed.

The render phase prepares the changes. The commit phase applies those changes to the DOM.

### Real-World Usage

Reconciliation is important when working with lists, conditional rendering, and keys. Good keys help React correctly match previous and next elements during render.

## 13. How does React compare the new output of a component with the previous output during render?

React compares the new React elements returned by the component with the previous Fiber tree using element type and key.

### Explanation

If the element type and key match, React can usually reuse the existing Fiber and update its props. If the type or key changes, React treats it as a different element and may unmount the old subtree and create a new one.

For lists, keys help React match items across renders. Without stable keys, React may reuse the wrong elements, causing incorrect UI behavior or unnecessary work.

React does not deeply compare every object or every prop by default. It uses its reconciliation rules to decide what work is needed.

### Example

```jsx
function TodoList({ todos }) {
  return (
    <ul>
      {todos.map((todo) => (
        <li key={todo.id}>{todo.title}</li>
      ))}
    </ul>
  );
}
```

Here, `todo.id` helps React match each list item between the previous render and the next render.

### Real-World Usage

This is critical in dynamic lists such as search results, tables, notifications, and form rows. Stable keys prevent unnecessary remounts and preserve component state correctly.

## 14. Why can expensive calculations inside render cause UI responsiveness issues?

Expensive calculations inside render can block React from finishing rendering quickly, which can make the UI feel slow or unresponsive.

### Explanation

Render logic runs on the main thread. If a component performs heavy filtering, sorting, formatting, deep cloning, or expensive computations during render, it can delay user interactions, painting, and other urgent updates.

In Concurrent Rendering, React can split and prioritize some rendering work, but expensive synchronous JavaScript still has a cost. The better approach is to keep render lightweight and memoize or move expensive work when appropriate.

### Example

```jsx
function ProductList({ products, searchText }) {
  const filteredProducts = React.useMemo(() => {
    return products.filter((product) =>
      product.name.toLowerCase().includes(searchText.toLowerCase()),
    );
  }, [products, searchText]);

  return (
    <ul>
      {filteredProducts.map((product) => (
        <li key={product.id}>{product.name}</li>
      ))}
    </ul>
  );
}
```

`useMemo` avoids recalculating the filtered list unless `products` or `searchText` changes.

### Real-World Usage

This is common in dashboards, large data tables, search pages, and complex forms. Keeping render lightweight improves typing responsiveness, scrolling smoothness, and perceived performance.

## 15. Scenario: A parent component re-renders frequently, causing many child components to re-render. What render phase issues would you investigate?

I would investigate whether the parent is causing unnecessary render work by passing unstable props, recreating functions or objects, updating state too high in the tree, or missing memoization boundaries.

### Explanation

When a parent renders, its child components may also be evaluated. That does not always mean the DOM changes, but it can still create render phase cost.

I would check:

- whether state is colocated properly or lifted too high
- whether child props change on every render due to inline objects, arrays, or functions
- whether expensive calculations are running inside child renders
- whether context updates are causing broad re-renders
- whether large lists need virtualization
- whether `React.memo`, `useMemo`, or `useCallback` would actually reduce work
- whether keys are stable and not causing remounts

The goal is not to memoize everything. The goal is to identify the real source of repeated render work and reduce unnecessary evaluation.

### Example

```jsx
const Child = React.memo(function Child({ user, onSelect }) {
  return <button onClick={onSelect}>{user.name}</button>;
});

function Parent({ user }) {
  const handleSelect = React.useCallback(() => {
    console.log("Selected", user.id);
  }, [user.id]);

  return <Child user={user} onSelect={handleSelect} />;
}
```

`React.memo` helps the child skip re-rendering when props are the same, and `useCallback` keeps the function reference stable when `user.id` has not changed.

### Real-World Usage

This situation often appears in layout components, dashboards, data grids, sidebars, and forms where one parent owns too much state. Profiling with React DevTools is usually the first step before applying optimizations.

## 16. Why is the render phase considered interruptible while the commit phase is not?

The render phase is interruptible because React is only calculating the next UI and has not changed the DOM yet. The commit phase is not interruptible because React is applying real changes to the screen and must keep the UI consistent.

### Explanation

During render, React can pause work, switch to a higher-priority update, restart rendering, or discard unfinished work. This is safe because no visible DOM mutation has happened yet.

During commit, React updates DOM nodes, attaches refs, runs layout effects, and makes changes visible to the user. If React interrupted the commit phase halfway, the screen could end up in an inconsistent state.

So render is preparation work. Commit is final application work.

### Real-World Usage

This distinction matters in large applications where React may delay or interrupt expensive rendering work to keep typing, clicking, and animations responsive.

## 23. How should teams decide where to place expensive computations so they do not block rendering?

Teams should first measure the cost, then decide whether the computation belongs in render, memoized derived state, a selector, the server, a Web Worker, or a precomputed data layer.

### Explanation

- If it is derived from props or state and moderately expensive, use `useMemo`
- If it is shared across components, move it to a memoized selector
- If it processes large datasets, consider server-side computation or a Web Worker
- If it is needed across pages, compute it in the data layer or API response

The important point is that render should stay lightweight enough to keep the UI responsive.

This avoids recalculating filtered and sorted rows unless the source data or filters change.

## 26. What internal work does React perform during the render phase on each Fiber node?

## 27. How does React collect effects during the render phase without executing DOM mutations immediately?

React marks Fibers with effect flags during the render phase and defers executing those effects until the commit phase.

During render, React determines what needs to happen later. For example, it may mark that a DOM node needs to be inserted, updated, removed, or that an effect hook needs to run.

This separation allows React to safely prepare UI updates without causing partial DOM changes. It is one reason Concurrent Rendering can interrupt render work without leaving the screen in a broken state.

## 28. How do lanes or priorities affect which render phase work is processed first?

Lanes represent update priority. React uses them to decide which updates should be rendered first and which work can wait.

### Explanation

React does not treat all updates equally. User-blocking updates, like typing into an input, are higher priority than non-urgent updates, like rendering a large filtered list.

When an update is scheduled, React assigns it to a lane. During render, React works on the lanes that are currently selected. If a higher-priority lane appears, React can pause lower-priority work and process the more urgent update first.

This is how React balances responsiveness with rendering complex UI.

### Real-World Usage

This is useful in real applications when using features like transitions. For example, React can keep input typing responsive while delaying a heavy search result update.

## 29. Why can React call a component multiple times before committing a single visible update?

React can call a component multiple times because rendering is only calculation work, and React may need to retry, restart, or validate that work before committing.

### Explanation

A component function can be called multiple times before one visible update for several reasons:

- Strict Mode intentionally re-renders components in development
- Concurrent Rendering may restart work when newer updates arrive
- A lower-priority render may be interrupted by higher-priority work
- Suspense may cause React to retry rendering
- State updates may be batched before one final commit

This is why component bodies must be pure. Calling a component does not guarantee the result will be committed to the screen.

### Example

```jsx
function Profile({ user }) {
  console.log("Rendering Profile");

  return <h2>{user.name}</h2>;
}
```

This log can run more than once even if React commits only one final visible update.

### Real-World Usage

This explains why logs inside component bodies can be misleading during debugging. React DevTools Profiler is better for understanding what actually rendered and committed.

## 30. Scenario: A low-priority render is interrupted by a high-priority update. What happens to the partially completed render work?

React pauses or abandons the low-priority render work and processes the high-priority update first.

### Explanation

Because the low-priority work is still in the render phase, it has not changed the DOM yet. React can safely stop working on it.

After the high-priority update is handled, React may resume the previous work, restart it with newer state, or discard it completely if it is no longer relevant.

For example, a large chart update may be interrupted by a text input update. React prioritizes the input so the UI stays responsive.

### Real-World Usage

This behavior is important in performance-sensitive apps with live dashboards, search pages, and large data views. It allows urgent user interactions to stay responsive even while heavier UI updates are pending.
