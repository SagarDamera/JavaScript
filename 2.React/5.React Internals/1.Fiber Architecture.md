# Topic: Fiber Architecture

- Fiber is React’s internal rendering architecture for scheduling and updating UI work.

- It allows React to break rendering work into small units instead of blocking the main thread.

- Fiber helps React pause, resume, prioritize, or discard rendering work when needed.

- Each React element is represented internally as a Fiber node.

- Fiber enables features like concurrent rendering, Suspense, transitions, and better responsiveness.

- React separates work into render phase and commit phase.

- Render phase can be interrupted, but commit phase must finish once started.

- High-priority updates like typing can be handled before low-priority updates like large list rendering.

- Interview trap: Fiber is not something developers directly use; it explains how React manages rendering internally.

- Real-world relevance: Fiber helps React apps stay responsive during complex UI updates.

### 1. What is React Fiber, and why was it introduced in React?

React Fiber is React’s internal reconciliation architecture. It was introduced to make rendering more incremental, interruptible, and prioritizable instead of doing all rendering work synchronously in one blocking pass.

#### Explanation

Before Fiber, React reconciliation was mostly synchronous. Once React started rendering a large update, it generally had to finish before the browser could handle higher-priority work like user input or animations.

Fiber solves this by breaking rendering work into small units. React can pause, resume, restart, or abandon work based on priority. This made features like concurrent rendering, transitions, Suspense, and better scheduling possible.

#### Real-World Usage

In large React applications, Fiber helps keep the UI responsive during expensive updates. For example, typing in an input should stay smooth even if a large list or complex component tree is updating elsewhere.

#### Common Follow-Up Question

How does React decide which updates are higher priority?

### 2. How is Fiber different from the older stack-based reconciliation approach?

The older stack-based reconciliation approach worked like a normal recursive call stack and rendered updates synchronously. Fiber replaced that with a custom data structure that lets React control the rendering work manually.

#### Explanation

In the stack-based model, React depended on the JavaScript call stack. Once rendering started, React could not easily pause in the middle and continue later.

Fiber nodes act like virtual stack frames. Because React owns this structure, it can split rendering work into units, pause between units, yield control back to the browser, and continue later.

The key difference is control. Stack reconciliation was synchronous and blocking. Fiber reconciliation is interruptible, schedulable, and priority-aware.

#### Real-World Usage

This matters when a page has expensive UI updates. With Fiber, React can prioritize urgent updates like clicks, typing, and hover interactions over less urgent updates like rendering a large filtered list.

#### Common Follow-Up Question

Does Fiber make rendering faster, or does it mainly make rendering more controllable?

### 3. What is a Fiber node in React?

A Fiber node is an internal JavaScript object that represents a unit of work for a React element, component, or host DOM node.

#### Explanation

Each Fiber node corresponds roughly to something in the React component tree. React uses Fiber nodes to track what needs to be rendered, updated, inserted, deleted, or committed to the DOM.

A Fiber node is not the same as a React element. A React element is a lightweight description of UI. A Fiber node is the internal working structure React uses to process that UI over time.

#### Example

```jsx
function Profile({ user }) {
  return <h1>{user.name}</h1>;
}
```

For this component tree, React internally creates Fiber nodes for `Profile`, the `h1`, and the text content. These nodes help React compare previous and next render output and decide what work is needed.

#### Real-World Usage

As an application developer, you do not directly create or modify Fiber nodes. But understanding Fiber helps explain React behavior around rendering, scheduling, effects, Suspense, and concurrent features.

### 4. What kind of information does a Fiber node store about a component?

A Fiber node stores information React needs to render, update, and manage a component across renders.

#### Explanation

A Fiber node can store details such as:

- The component type
- Props and pending props
- State and update queue
- Parent, child, and sibling relationships
- Effects that need to run
- Priority information for updates
- The associated DOM node for host components
- A link to the previous version of the Fiber node

This allows React to compare the current tree with the work-in-progress tree and decide what changes are needed.

#### Real-World Usage

This is why React can preserve component state across renders when the component stays in the same position in the tree. It is also why changing keys can force React to treat a component as a new instance.

### 5. Why does React represent the UI as a tree of Fiber nodes?

React represents the UI as a tree of Fiber nodes because the UI itself is hierarchical, and React needs an efficient way to traverse, compare, update, and commit changes.

#### Explanation

Components naturally form a tree: parents render children, children render more children, and eventually React reaches DOM nodes.

By representing this as a Fiber tree, React can walk through the UI one unit at a time. It can process part of the tree, pause work, resume later, skip unchanged branches, or prioritize more important updates.

React typically works with two trees: the current tree that represents what is already on the screen, and a work-in-progress tree that represents the next version of the UI.

#### Real-World Usage

This tree structure is important for features like conditional rendering, list updates, preserving state, Suspense boundaries, and concurrent rendering. It helps React update only the parts of the UI that actually need to change.

### 6. How does the Fiber tree relate to the React component tree?

The Fiber tree is React’s internal representation of the React component tree. The component tree is what developers write using components and JSX, while the Fiber tree is what React uses internally to render, update, and manage that UI.

#### Explanation

When a React component returns JSX, React creates React elements. During reconciliation, React converts those elements into Fiber nodes.

Each Fiber node represents a unit of work for React. It may represent a function component, class component, host DOM element like `div`, text node, fragment, Suspense boundary, or other React internal type.

So, the component tree is the developer-facing structure, and the Fiber tree is the runtime structure React uses to process that tree efficiently.

#### Example

      <Header />
      <Content />
    </main>

React may internally create Fiber nodes for `App`, `main`, `Header`, and `Content`. These Fiber nodes allow React to track updates, props, state, effects, and relationships between nodes.

#### Real-World Usage

This helps explain why React can preserve state when a component stays in the same position in the tree and why changing component structure or keys can cause remounting.

### 7. What is the difference between the current Fiber tree and the work-in-progress Fiber tree?

The current Fiber tree represents the UI that is already committed and visible on the screen. The work-in-progress Fiber tree represents the next version of the UI that React is preparing during rendering.

#### Explanation

React does not immediately mutate the currently visible tree while rendering. Instead, it builds or updates a separate work-in-progress tree.

Once React finishes rendering the new tree and decides what changes are needed, it enters the commit phase. During commit, React applies the final changes to the DOM and makes the work-in-progress tree the new current tree.

This separation helps React render updates safely before showing them to the user.

#### Real-World Usage

In real applications, this allows React to prepare a new UI version without corrupting the currently displayed UI if rendering is interrupted, restarted, or abandoned.

#### Common Follow-Up Question

### 8. Why does React maintain two Fiber trees instead of directly modifying the current tree?

React maintains two Fiber trees so it can prepare updates without immediately changing the UI that users currently see.

#### Explanation

If React directly modified the current tree during rendering, an interrupted or failed render could leave the UI in an inconsistent state.

By using a current tree and a work-in-progress tree, React can safely compute the next UI in memory. If the work is completed successfully, React commits the result. If the work is interrupted or replaced by a higher-priority update, React can discard or restart the work-in-progress tree without affecting the visible UI.

This approach is often called **"double buffering"** usually used in gaming applications for better rendering.

#### Real-World Usage

This is important in concurrent rendering. For example, React can start rendering a large page update, pause it for a user input update, and continue later without showing a partially updated UI.

### 9. What is the purpose of the alternate pointer in a Fiber node?

The `alternate` pointer links a Fiber node in the current tree to its matching Fiber node in the work-in-progress tree.

#### Explanation

React uses the `alternate` pointer to move between the two versions of the same logical component.

For example, one Fiber represents the currently committed version of a component, and its alternate represents the version being prepared for the next render.

This allows React to reuse Fiber nodes across renders instead of creating everything from scratch. It also helps React compare previous props, state, effects, and update queues with the new version.

#### Real-World Usage

The `alternate` pointer is part of how React efficiently updates complex UIs. It allows React to preserve component identity, reuse work, and switch between the current and next tree during rendering and commit.

### 10. In a real application, how does Fiber help React pause and resume rendering work?

Fiber helps React pause and resume rendering by breaking rendering into small units of work instead of processing the entire component tree in one blocking call stack.

#### Explanation

Each Fiber node represents a unit of work. React can process one Fiber node, check if the browser needs control back, and then decide whether to continue, pause, or switch to higher-priority work.

For example, if React is rendering a large list and the user starts typing, React can prioritize the input update so the UI feels responsive.

This is possible because Fiber is not tied to the native JavaScript call stack. React manages its own work loop using Fiber nodes.

#### Example

```jsx
import { useState, useTransition } from "react";

function SearchPage({ items }) {
  const [query, setQuery] = useState("");
  const [filteredQuery, setFilteredQuery] = useState("");
  const [isPending, startTransition] = useTransition();

  function handleChange(event) {
    const value = event.target.value;
    setQuery(value);

    startTransition(() => {
      setFilteredQuery(value);
    });
  }

  const filteredItems = items.filter((item) =>
    item.name.toLowerCase().includes(filteredQuery.toLowerCase()),
  );

  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending && <p>Updating results...</p>}

      {filteredItems.map((item) => (
        <div key={item.id}>{item.name}</div>
      ))}
    </>
  );
}
```

#### Real-World Usage

This is useful in search pages, dashboards, large tables, autocomplete experiences, and data-heavy screens where urgent interactions should stay responsive while expensive rendering happens in the background.

### 11. How does Fiber enable React to split rendering work into smaller units?

Fiber enables React to split rendering work by representing each part of the UI as a Fiber node, where each node is treated as a small unit of work.

#### Explanation

Instead of rendering the entire component tree in one synchronous recursive call, React can process one Fiber node at a time.

For each Fiber node, React can:

- Begin work on that component
- Compare props and state
- Reconcile its children
- Prepare any required updates
- Move to the next Fiber node

Because the work is broken into Fiber nodes, React can pause between units, check if higher-priority work exists, and continue later.

#### Real-World Usage

In a large dashboard, React may need to update many widgets, tables, and charts. Fiber allows React to process that work in smaller pieces instead of blocking the browser until the entire screen is finished.

### 12. What does it mean when people say Fiber makes rendering interruptible?

It means React can pause rendering work before it finishes the full tree, handle something more urgent, and then come back to the paused work later.

#### Explanation

Before Fiber, rendering was mostly synchronous. Once React started rendering an update, it generally had to complete the whole render before the browser could respond to user input or paint.

With Fiber, React controls its own work loop. After processing a unit of work, React can decide whether to continue or yield control back to the browser.

This does not mean the DOM is partially updated. The render phase can be interrupted, but the commit phase is still applied in a consistent way.

#### Real-World Usage

If a user is typing while a large list is being filtered, React can prioritize the typing update so the input feels responsive instead of freezing while the list render completes.

### 13. What are the parent, child, and sibling links used for inside the Fiber tree?

The parent, child, and sibling links are used to let React traverse the Fiber tree efficiently without relying on the JavaScript call stack.

#### Explanation

Each Fiber node has links that describe its position in the tree:

- `return` points to the parent Fiber
- `child` points to the first child Fiber
- `sibling` points to the next sibling Fiber

These links let React move through the tree manually. React can go down to a child, across to a sibling, or back up to the parent when a subtree is complete.

This structure is one reason React can pause and resume rendering work.

#### Example

```jsx
function App() {
  return (
    <section>
      <Header />
      <Content />
      <Footer />
    </section>
  );
}
```

Internally, the Fiber for `section` may have its first child as `Header`. `Header` can point to `Content` as its sibling, and `Content` can point to `Footer` as its sibling.

#### Real-World Usage

This tree navigation helps React process complex component structures incrementally, including nested layouts, lists, fragments, Suspense boundaries, and portals.

#### Common Follow-Up Question

### 14. How does React use Fiber nodes to continue work after being interrupted?

React uses Fiber nodes as saved work units. Because each Fiber node stores its relationships, pending work, props, state, and alternate link, React can remember where it stopped and continue from that point later.

#### Explanation

When React is rendering, it walks the Fiber tree one unit at a time. If React needs to pause, it keeps track of the next unit of work.

Later, when the browser has time or the higher-priority task is complete, React can resume from that saved Fiber node.

This is possible because Fiber is React’s own data structure. React is not depending only on the native JavaScript call stack, which would be lost once execution pauses.

#### Real-World Usage

In real applications, this helps with expensive screens like search results, reporting pages, large forms, and admin dashboards. React can continue rendering less urgent UI after handling urgent interactions like typing, clicking, or navigation.

### 15. What problems can happen if React rendering work is not interruptible in a large UI?

If rendering is not interruptible, large UI updates can block the main thread, causing input lag, delayed clicks, dropped frames, and a frozen user experience.

#### Explanation

JavaScript runs on the browser’s main thread. If React spends too much time rendering a large update synchronously, the browser cannot quickly handle user input, animations, layout, or painting.

This can cause problems such as:

- Typing feels delayed
- Buttons do not respond immediately
- Animations become choppy
- The page appears frozen
- Lower-priority updates block urgent interactions

Fiber helps reduce these problems by allowing React to split and prioritize rendering work.

#### Real-World Usage

This is especially important in enterprise applications with large tables, dashboards, filters, charts, dynamic forms, and deeply nested component trees.

Fiber helps React handle large list updates by breaking rendering into smaller units of work, assigning priority to updates, and allowing less urgent rendering work to be interrupted or delayed.

Fiber is the internal architecture that makes concurrent rendering possible, but concurrent rendering is a higher-level React behavior built on top of Fiber.

Fiber gives React the ability to split rendering into units, track update priority, pause work, resume work, and discard unfinished work.

Concurrent rendering uses those capabilities to prepare UI updates in the background without immediately blocking urgent user interactions.

So Fiber is the foundation. Concurrent rendering is a feature enabled by that foundation.

A good way to explain it is: Fiber is the engine architecture, while concurrent rendering is one driving mode React can use with that engine.

Features like `startTransition`, Suspense, and deferred rendering rely on React being able to prepare work without immediately committing it to the screen.

## 3. What trade-offs does React make by using a Fiber tree instead of a simple recursive call stack?

React gains more control over rendering, scheduling, priority, and interruption, but the trade-off is more internal complexity and memory overhead.

#### Explanation

A simple recursive call stack is easier to understand and implement. React can walk the component tree recursively and finish rendering in one pass.

Fiber replaces that with an explicit tree of Fiber nodes. This gives React control over how work is performed, but it also means React has to manage more internal bookkeeping.

The trade-offs include:

- More memory usage because React keeps Fiber nodes
- More internal complexity
- More scheduling logic
- More complex debugging behavior in concurrent scenarios
- More work to preserve consistency between current and work-in-progress trees

The benefit is that React can support modern rendering behavior like interruptible rendering, priorities, transitions, and Suspense.

#### Real-World Usage

As an application engineer, this trade-off is usually worth it because the UI can remain responsive in large applications. The complexity stays mostly inside React, while developers get better scheduling behavior through APIs like `useTransition`.

## 4. How would your component design change if you know React can pause, restart, or discard render work?

I would make sure render logic is pure, repeatable, and free from side effects because React may run rendering work more than once before committing the final result.

#### Explanation

Since Fiber allows React to pause, restart, or abandon render work, components should not depend on render being executed exactly once.

That means I would avoid:

- Mutating external objects during render
- Starting API calls directly inside render
- Reading or writing DOM during render
- Generating unstable values that affect behavior
- Performing business side effects inside component bodies

Instead, I would keep render logic focused on describing UI. Side effects should go into `useEffect`, layout reads into `useLayoutEffect` when required, and expensive calculations into `useMemo` when justified.

#### Example

```jsx
function OrderSummary({ order }) {
  const total = order.items.reduce((sum, item) => {
    return sum + item.price * item.quantity;
  }, 0);

  return <strong>Total: ${total}</strong>;
}
```

This component is safe because rendering only calculates output from props. It does not mutate external state or trigger side effects.

#### Real-World Usage

This matters in concurrent rendering and Strict Mode. Components with side effects during render can behave unpredictably when React re-runs or discards render work.

# React Fiber Interview Answers

## 1. Walk me through what happens at a high level when a component update creates work on a Fiber node.

When a component update happens, React marks the related Fiber node as having pending work, schedules that work with a priority, builds a work-in-progress tree, reconciles the affected part of the UI, and finally commits the completed changes to the DOM.

#### Explanation

At a high level, the flow is:

- A state, props, context, or external store update happens.
- React finds the Fiber node related to that component.
- React adds the update to that Fiber’s update queue.
- React marks the Fiber and its ancestors as having pending work.
- React schedules the update based on priority.
- During the render phase, React creates or updates the work-in-progress Fiber tree.
- React compares the new output with the previous tree.
- If changes are needed, React records the required effects.
- During the commit phase, React applies the final changes to the DOM.

The important interview point is that Fiber turns a component update into scheduled work that React can prioritize and process incrementally.

#### Real-World Usage

This matters when debugging why a state update causes a parent or child area to re-render. React does not immediately mutate the DOM when `setState` runs; it schedules work on the Fiber tree first.

#### Common Follow-Up Question

What is the difference between scheduling an update and committing an update?

---

## 2. How does React move through the Fiber tree when performing work?

React moves through the Fiber tree manually using each Fiber node’s child, sibling, and parent links.

#### Explanation

React does not depend only on recursive JavaScript function calls to walk the tree. Instead, it uses Fiber nodes as a linked tree structure.

React typically follows this pattern:

- Start from the root or updated subtree.
- Begin work on a Fiber node.
- Move down to its child if it has one.
- If there is no child, complete that Fiber.
- Move to its sibling if one exists.
- If there is no sibling, move back up to the parent.
- Continue until the required work is complete.

This traversal allows React to pause after processing a unit of work and resume later from the next Fiber node.

#### Example

```jsx
function App() {
  return (
    <main>
      <Header />
      <Content />
      <Footer />
    </main>
  );
}
```

React can process `App`, then `main`, then move through `Header`, `Content`, and `Footer` using Fiber links instead of relying only on the native call stack.

#### Real-World Usage

This is useful in large applications because React can walk large UI trees in controlled pieces. That control is what allows interruptible rendering and prioritization.

## 3. Why is Fiber described as a virtual stack frame?

Fiber is described as a virtual stack frame because each Fiber node stores enough information for React to pause work on a component and continue later, similar to how a normal call stack frame stores execution state.

#### Explanation

In the old stack-based approach, React relied heavily on the JavaScript call stack. Once rendering started, the stack had to keep running until the render finished.

Fiber replaces that with React-managed stack frames. Each Fiber node represents a unit of work and stores information like:

- The component type
- Props and state
- Child, sibling, and parent links
- Pending updates
- Effects
- Alternate Fiber reference

Because this information is stored in Fiber objects, React can stop work, leave the browser free to handle urgent tasks, and later continue from where it left off.

#### Real-World Usage

This is the architectural reason React can support modern features like concurrent rendering, transitions, Suspense, and interruption of low-priority rendering work.

## 4. What role does Fiber architecture play in separating “rendering work” from “DOM mutation work”?

Fiber helps React separate the render phase, where React calculates what should change, from the commit phase, where React actually mutates the DOM.

#### Explanation

In the render phase, React walks the Fiber tree and builds the work-in-progress tree. This phase is mostly about calculation:

- Calling components
- Comparing previous and next output
- Reconciling children
- Preparing changes
- Collecting effects

This work can be paused, restarted, or abandoned.

In the commit phase, React applies the final result to the host environment, usually the browser DOM. This includes:

- Inserting DOM nodes
- Updating DOM attributes
- Removing nodes
- Running layout effects
- Updating refs

The commit phase must be consistent and is not interrupted the same way render work can be.

#### Real-World Usage

This distinction helps explain why render should stay pure. DOM reads/writes, subscriptions, and side effects should not happen during render because React may render without committing.

## 5. If React starts rendering an update but later receives a more important update, how does Fiber architecture make that possible?

Fiber makes this possible by allowing React to pause or abandon the current work-in-progress render and switch to a higher-priority update.

#### Explanation

Because rendering work is stored in Fiber nodes instead of only the JavaScript call stack, React can control the work loop.

For example, React may start rendering a large low-priority update, such as filtering a big list. While that work is in progress, the user types into an input. The input update is more urgent.

Fiber allows React to:

- Stop the current render work
- Keep track of what was being processed
- Prioritize the urgent update
- Render and commit the urgent update first
- Resume, restart, or discard the previous lower-priority work

The current UI remains safe because React is preparing changes on the work-in-progress tree, not directly mutating the committed current tree.

#### Real-World Usage

This is important in search pages, dashboards, live feeds, forms, and large data-heavy screens. User interactions should stay responsive even when expensive UI updates are happening in the background.
